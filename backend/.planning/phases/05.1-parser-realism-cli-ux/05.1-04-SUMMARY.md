# Phase 5.1 Plan 04: Viewer Affordances for Unresolved Parser States Summary

Ship the viewer side of D-01 (unresolved module placeholders) and D-02 (non-literal count/for_each) BEFORE the parser work in Plan 02 so both halves can land independently; the viewer is additive and activates when Plan 02 later emits the marker fields.

## Objective Recap

Add two visual affordances to `ResourceNode.tsx`:
1. An orange ring tint + `⚠` floating marker when `data.type` starts with `_infracanvas_unresolved_` (matches Plan 02's placeholder-node synthesis).
2. A slate `×?` floating pill when `data.attributes._unresolved_count === true` (matches Plan 02's collapsed-count fallback).

No new color tokens, no `DriftStatus` union changes, no schema churn. Reuse the shadow visual vocabulary verbatim.

## Implementation Highlights

### ResourceNode.tsx extension points (c45b1df)

Two new derived booleans sit alongside the existing drift booleans:

```typescript
const isUnresolvedModule =
  typeof data.type === 'string' && data.type.startsWith('_infracanvas_unresolved');
const hasUnresolvedCount = Boolean(
  (data.attributes as Record<string, unknown> | undefined)?.['_unresolved_count'],
);
```

`driftTint` gained one OR branch (`isShadow || isUnresolvedModule`) so the icon wrapper's `boxShadow` ring picks up the existing `#D97706` codepath with zero new CSS.

Two new absolutely-positioned `<div>` badges were appended after the severity-badge block and before the source `<Handle>`:

- `⚠` marker at `top: -4, left: 22` with `data-testid="resource-node-unresolved-marker"` and `title="Unresolved module — check stderr for parse errors"`.
- `×?` pill at `bottom: 18, right: -4` with `data-testid="resource-node-unresolved-count-badge"` and `title="Non-literal count/for_each — expanded instance count unknown"`.

Both render only static text children (⚠, ×?), mitigating XSS risk from untrusted `data.type` (T-05.1-11) and `data.attributes` (T-05.1-12). Confirmed `NODE_W=120`, `NODE_H=90`, `ICON_SIZE=40` unchanged.
