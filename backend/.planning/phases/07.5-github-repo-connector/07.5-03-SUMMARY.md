## Decisions Made

- **RSAPrivateKey isinstance guard.** `cryptography.hazmat.primitives.serialization.load_pem_private_key` returns a union type covering DH / Ed25519 / Ed448 / EC / RSA / ML-DSA keys. `jwt.encode(..., algorithm="RS256")` only accepts RSA + EC. Without the isinstance guard, mypy `--strict` correctly fails compilation, and at runtime an EC key would silently sign with ES256 instead of RS256. The defensive raise turns the failure into a single-line config error rather than an opaque algorithm mismatch downstream.
- **Redis singleton + injection kwarg dual-path.** Production uses `_get_redis()` lru_cache (matches `storage/r2.py::get_r2_client()` shape). Tests pass `redis_client=fake_redis` directly so the lru_cache never traps a test-time URL. Cleaner than monkeypatching the module-level singleton between tests, and zero overhead in production (the kwarg is `None` and the lru_cache fires on first call).
- **`decode_responses=True` on Redis.from_url.** Without it, `.get()` returns bytes and `json.loads(bytes)` works on CPython but emits a `DeprecationWarning` on some Python versions and complicates the test fixture (fakeredis is already configured with `decode_responses=True`). Single-line change keeps the str-vs-bytes contract crisp.
- **RED commits bundle package `__init__.py` with the test file.** Otherwise pytest discovery fails on the missing package before reaching the import-error in the test module — masking the RED gate. The two `__init__.py` files contain only their docstring; they're discovery infrastructure, not implementation, and ship with the test.
- **`urllib.parse.quote(branch, safe="")` for `get_head_sha`.** The `git/ref/heads/{branch}` endpoint treats slashes in `branch` as path segments — `feature/foo` would be sent as `git/ref/heads/feature/foo`, which GitHub interprets as the `feature` ref under the `foo` directory and returns the wrong sha (or 404). With `safe=""`, the slash becomes `%2F` and stays a single segment. Test 8 (`test_branch_with_slash`) regression-locks this.
- **`get_installation_metadata` uses App JWT, NOT installation token.** Per GitHub Apps API, `/app/installations/{id}` is App-level metadata; an installation token isn't sufficient. Test 7 explicitly registers the `/access_tokens` POST and asserts `call_count == 0` to lock the discrimination — easy to regress without it.
- **Substring filter happens server-side after the bulk fetch.** GitHub's `/search/repositories?q=...+user:{owner}` requires a user OAuth token, not an installation token, so we can't use it. The 100-repo `per_page` cap on `/installation/repositories` plus a `[r for r in repos if ql in r['full_name'].lower()]` server-side filter is sufficient for the typeahead UX (D-10b).
- **`ScanFromGitHubReq.repo` regex `^[\w.\-]+/[\w.\-]+$`.** Matches GitHub's allowed owner/repo character set (alphanumerics, `_`, `.`, `-`), with a single `/` separator and no leading/trailing whitespace. Stricter regexes (e.g. lowercase-only) would reject valid GitHub names like `MyOrg/MyRepo`.
