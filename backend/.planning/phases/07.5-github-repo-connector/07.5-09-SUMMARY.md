# Phase 07.5 Plan 09: Live Integrations Page + ScanTriggerForm Summary

Composes the Plan 08 reusable components into the user-facing scan-trigger flow on `/settings/integrations` — a three-state machine (loading / preinstall / postinstall) driven off `GET /api/github/installations`, with a post-install hydration poll capped at 5 attempts (~15s) when the user lands back with `?install=success`.

## What Shipped

### `dashboard/components/integrations/ScanTriggerForm.tsx` (new — ~117 LoC)

Composing client component with the signature `ScanTriggerForm({ installationId: number })`. Internal state: `repo` (RepoResp | null), `branch` (string, init `""`), `path` (string, init `"."`), `submitting` (boolean), `error` (string | null). `canSubmit` derived from `!!repo && !!branch && !submitting`.

Render structure (vertical stack inside `space-y-3 p-4 border border-slate-200 rounded-md`):
- Label "Repository" + `<RepoCombobox installationId value={repo} onSelect={r => setRepo(r); setBranch('')}/>`
- Label "Branch" + `<BranchPicker installationId selectedRepo={repo} value={branch} onChange={setBranch}/>` (BranchPicker itself returns null when selectedRepo is null — no extra branch logic in this parent)
- Label "Path (subdirectory)" + `<Input>` with `aria-label="Subdirectory path"` and placeholder `"."`, value defaults to `"."`
- Inline `role="alert"` block when error is set
- `<Button onClick={handleSubmit} disabled={!canSubmit}>` showing "Starting…" mid-flight else "Scan"

`handleSubmit` POSTs `{ installation_id, repo: repo.full_name, branch, path: path || '.' }` to `/api/scans/from-github`, throws `Error("http_<status>")` on `!res.ok`, then on success `router.push("/scans/" + data.scan_id)` so Plan 10's polling page picks up the pending state. On error, maps `http_503` → "GitHub is rate-limiting us right now. Please retry in a minute." and everything else → "Failed to start scan. Please retry." (T-07.5-09-03 — never renders raw backend bodies).

### `dashboard/app/(dashboard)/settings/integrations/page.tsx` (rewrite)

Three-state machine keyed off `installations: InstallationResp[] | null`:
- `null`  → loading skeleton (`data-testid="github-loading"`, "Loading installations…")
- `[]`    → InstallButton + helper copy ("Connect a repo to scan it on demand."); when `?install=success` is in the URL, also shows "Waiting for GitHub to confirm the install…"
- `[...]` → ordered list (`<ul>`) of installation rows: account login + installed-at date + "Manage on GitHub" link (`https://github.com/settings/installations/{installation_id}`, `target=_blank rel=noopener noreferrer`) + inline `<ScanTriggerForm installationId={row.installation_id}/>`

Two `useEffect` hooks:
1. **Initial mount fetch** — GET `/api/github/installations`, cancelled-flag cleanup (CC-14). On error sets `error` and falls through to `[]` so the InstallButton is still reachable.
2. **Post-install hydration poll** — when `installSuccess && installations !== null && installations.length === 0`, `setInterval` every `POLL_INTERVAL_MS=3000ms` for at most `POLL_MAX_ATTEMPTS=5` attempts (~15s ceiling). Stops as soon as the list becomes non-empty OR the cap is reached (T-07.5-09-01 — bounded so a hung backend can't pin the page).

Slack stub block preserved verbatim (Phase 8 will wire it up).
