## Polling Contract

The polling state machine is deliberately small and predictable:

```
mount with status='pending'
  └─ setInterval(tick, 2000) registered
       └─ tick: fetch /api/scan-status?id={scan.id}
            ├─ res.ok && data.status === 'pending'  → setScan(data); next tick fires in 2s
            ├─ res.ok && data.status === 'ready'    → setScan(data); router.refresh(); cleanup clears interval
            ├─ res.ok && data.status === 'failed'   → setScan(data); cleanup clears interval; render error_message + Retry
            ├─ !res.ok                              → swallow; next tick retries
            └─ throw                                → swallow; next tick retries

unmount mid-poll                                   → cancelled = true; clearInterval

mount with status='ready'                          → router.refresh() once; no setInterval started
mount with status='failed'                         → render error_message + Retry; no setInterval started
```

Cancelled-flag teardown drops in-flight `await fetch().json()` resolutions on unmount so a
late tick can't `setScan()` on a dead tree (the React-strict-mode and tab-switch failure
modes documented in CC-14).

## Retry POST Shape

When the user clicks Retry on a failed scan, the component re-POSTs to the existing
Plan 07 `/api/scans/from-github` proxy. **Payload is sourced exclusively from the scan
row's server-fetched columns** (T-07.5-10-03 mitigation):

```json
POST /api/scans/from-github
{
  "installation_id": <scan.github_installation_id>,
  "repo":            <scan.github_repo>,
  "branch":          <scan.github_branch>,
  "path":            <scan.source_path || ".">
}
```

When any github_* column is null (e.g. scan was triggered via CLI rather than the
dashboard), the Retry button is replaced with a hint: "Cannot retry from here — re-scan
from /settings/integrations". 503 maps to friendly rate-limit copy; everything else to a
generic "Failed to re-trigger scan" message — same status-only error mapping pattern as
ScanTriggerForm (T-07.5-09-03).

## End-to-End Wire (Plan 11 smoke test prerequisite)

The full Phase 7.5 happy path is now wired end-to-end:

1. User clicks **Install GitHub App** on `/settings/integrations` (Plan 08 InstallButton).
2. GitHub redirects back with `?install=success` → page poll-hydrates installations
   list (Plan 09 — bounded 5-attempt × 3s).
3. User picks repo + branch + path and clicks **Scan** (Plan 09 ScanTriggerForm) →
   POST `/api/scans/from-github` (Plan 07 proxy → Plan 05 backend) → 202 with
   `{scan_id}`.
4. `router.push('/scans/{scan_id}')` lands on the detail page in `pending` status
   (Plan 06 worker is still mid-clone).
5. **This plan**: page status-gate routes the pending row to ScanPendingClient →
   `setInterval` polls `/api/scan-status` every 2s.
6. Plan 06 worker finalizes → `status` flips to `ready` (or `failed`) → polling
   detects, calls `router.refresh()` → parent RSC re-renders with the now-non-null
   `presigned_get_url` → existing `ScanViewerClient` mounts (UNCHANGED Phase 7 D-08
   path).

This unblocks Plan 11's manual smoke + verification checkpoint.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 – Bug] Next.js 15 page-module export restriction tripped TS2344**
- **Found during:** Task 3 GREEN (after the initial co-located `renderScanByStatus`
  export passed all 4 vitest cases, `npx tsc --noEmit` flagged
  `.next/types/app/(dashboard)/scans/[id]/page.ts(12,13): error TS2344: Type
  'OmitWithTag<typeof import("..."), "default" | ...>' does not satisfy the constraint
  '{ [x: string]: never; }'. Property 'renderScanByStatus' is incompatible with index
  signature.`).
- **Issue:** Next.js 15 enforces a fixed allow-list of page-module exports (default
  + metadata + config + generateStaticParams + revalidate + a few more). Adding any
  other named export to a `page.tsx` is a compile-time error.
- **Fix:** Extracted the helper into a sibling module
  `dashboard/app/(dashboard)/scans/[id]/renderScanByStatus.tsx`. Both the page and the
  vitest test now import from the new module. Test mock chain also simplified: no
  longer needs to mock MetadataHeader / ScanDetailActions because the helper does not
  render them.
- **Files modified:** `dashboard/app/(dashboard)/scans/[id]/{page.tsx, renderScanByStatus.tsx}`
  + `dashboard/__tests__/scan-detail-polling.test.tsx`
- **Folded into commit:** 75a6875

**2. [Rule 1 – Bug] vi.mock 3-arg `{ virtual: true }` form rejected by vitest 4**
- **Found during:** Task 3 GREEN (after the export-fix above, `npx tsc --noEmit`
  flagged `__tests__/scan-detail-polling.test.tsx(50,3): error TS2554: Expected 1-2
  arguments, but got 3.`).
- **Issue:** The vitest 4 `vi.mock` API only accepts 2 arguments (path + factory).
  The 3-arg form (`{ virtual: true }`) belongs to Jest, not Vitest.
- **Fix:** Dropped the 3rd arg entirely — once the helper was extracted into its own
  real module (Rule-1 fix #1), `virtual: true` was no longer needed for any of the
  mocks anyway.
- **Files modified:** `dashboard/__tests__/scan-detail-polling.test.tsx`
- **Folded into commit:** 75a6875
