## What Shipped

### `dashboard/components/integrations/InstallButton.tsx`

Props: `()` (no props — derives state from Clerk).

Renders a shadcn `<Button variant="default">Install InfraCanvas GitHub App</Button>` when `useOrganization().organization` is truthy; returns `null` otherwise. Click opens
`https://github.com/apps/{NEXT_PUBLIC_GITHUB_APP_SLUG}/installations/new?state={organization.id}` in a new tab via `window.open(url, '_blank', 'noopener,noreferrer')`. Default slug is `infracanvas-dev` when the env var is unset.

### `dashboard/components/integrations/RepoCombobox.tsx`

Props: `{ installationId: number; value?: RepoResp | null; onSelect: (repo: RepoResp) => void }`.

shadcn Popover + Command (cmdk) recipe. CommandInput is debounced 250ms via the CC-15 `useRef<setTimeout>` pattern; on each `installationId` or `query` change a new debounce timer fires `fetch('/api/github/repos?installation_id=X&q=Y')`. `shouldFilter={false}` is critical — backend already filters server-side. Renders `<CommandItem value={repo.full_name}>` for each repo with a lucide `<Lock aria-label="private">` icon when `repo.private`. On 503 (proxy preserves `Retry-After: 60`) shows inline `role="alert"` "Rate limited — try again in 60s". Cancelled-flag cleanup (CC-14) on unmount.

### `dashboard/components/integrations/BranchPicker.tsx`

Props: `{ installationId: number; selectedRepo: RepoResp | null; value: string; onChange: (branch: string) => void }`.

shadcn Select primitive; renders `null` when no repo. On `installationId` or `selectedRepo.full_name` change, fetches `/api/github/branches?installation_id=X&repo=org%2Fname` (URL-encoded). When the fetch resolves AND `value` is empty, calls `onChange(selectedRepo.default_branch)` so the dropdown shows the repo's default branch as the initial selection. Cancelled-flag cleanup. 503 handled with the same inline alert pattern.

## Decisions Made

1. **`shouldFilter={false}` on Command** — Backend filters by `q` (D-05); cmdk's default substring re-filter would hide repos that the server-side heuristic considers a match but cmdk's substring doesn't (e.g. fuzzy or relevance-ordered matches). Locking this off keeps server-side authority over the result set.
2. **BranchPicker default-branch fallback only when `value` is empty** — Guards against clobbering a user's prior selection when the parent rerenders with the same `selectedRepo` object identity but different `value`. The current effect deps `[installationId, selectedRepo?.full_name]` already short-circuits unrelated rerenders, but the in-effect `if (!value)` adds belt-and-braces for fast clicks.
3. **eslint-disable react-hooks/exhaustive-deps on BranchPicker** — Effect intentionally keys on `[installationId, selectedRepo?.full_name]` only. Including `value` and `onChange` in deps would cause an infinite refetch loop because the default-branch fallback inside `.then` calls `onChange`, which the parent uses to set `value`. Documented inline.
4. **InstallButton `state == organization.id` (Clerk org id)** — NOT `team_id`. Confirmed against Plan 04 PATTERNS divergence note: the dashboard never knows `team_id` client-side. The backend install-callback resolves the Team via `resolve_team_from_clerk_org(state)` and validates `state == team.clerk_org_id`. CSRF mitigation is preserved by the equivalence; the divergence keeps the secret/internal id off the wire.
5. **Composition deferred to Plan 09** — RepoCombobox does NOT import BranchPicker. The 3 components are independently testable and reusable; ScanTriggerForm in Plan 09 wires them into a single form, and Plan 11's polling page may render RepoCombobox alone without the trigger form. The plan frontmatter `requires` graph documents this composition shape for downstream consumers.
6. **Inline 503 alert (role='alert'), not toast** — The 503+Retry-After:60 contract from Plan 04→07 is honored at the UI: surfaces the rate-limit gap directly inline next to the input that triggered it. Toast plumbing is a dashboard-global concern; introducing it here would couple the picker to a global UI notification system.
7. **Lock icon with `aria-label='private'`** — accessibility-first: tests use `getAllByLabelText('private')` which is the same query path screen readers use to identify the visual marker. No `data-testid` needed.
8. **Test 6 of branch-picker fixed during GREEN** — Originally asserted on `onChange` being called with the default branch when `value="dev"` was preset. Wrong: the default-branch fallback only runs when `value` is empty (decision 2). Updated assertion to wait for the Select trigger to drop its `disabled` attribute (the loading-finished signal). One Rule 1 fix in TDD GREEN, properly bundled with the implementation commit.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `branch-picker.test.tsx` test 6 had incompatible assertion with default-branch fallback design**

- **Found during:** Task 3 GREEN — first GREEN run reported 5/6 pass.
- **Issue:** The original test 6 assertion was:
  ```ts
  await waitFor(() => expect(onChange).toHaveBeenCalledWith('dev'))
  ```
  while rendering with `value="dev"`. But the BranchPicker default-branch fallback intentionally only fires when `value` is empty (to avoid clobbering a user's prior selection — see Decision 2). With `value="dev"` preset, `onChange('dev')` never fires, so the assertion times out.
- **Fix:** Replaced the wait-for-default-branch-fire assertion with a wait-for-the-Select-to-finish-loading assertion (`expect(screen.getByRole('combobox')).not.toHaveAttribute('disabled')`). Then the rest of test 6 (open dropdown → click feature/x → expect onChange('feature/x')) runs as intended.
- **Files modified:** `dashboard/__tests__/branch-picker.test.tsx`
- **Commit:** `83e5309` (rolled into Task 3 GREEN per TDD cadence)
