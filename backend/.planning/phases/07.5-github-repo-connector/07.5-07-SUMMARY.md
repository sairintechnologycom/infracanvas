# Phase 07.5 Plan 07: Dashboard Proxy Routes Summary

Four Next.js Route Handlers under `dashboard/app/api/` proxy authenticated requests
to the backend's `/v1/github/*` and `/v1/scans/from-github` endpoints, attaching
the Clerk JWT via the existing server-only `backendFetch` helper. lib/types.ts
gains TypeScript mirrors of `backend/app/schemas/github.py` plus the six optional
fields Plan 05 added to `ScanGetResp` (error_message, source_path, github_*).

## What Shipped

- **GET /api/github/installations** → backend `/v1/github/installations`. No query
  string. Returns the list of GitHub App installations scoped to the caller's
  team via RLS. Backs Plan 09's `<InstallationsList>`.
- **GET /api/github/repos** → backend `/v1/github/repos?installation_id=&q=`. Pass-
  through query string. Special-cases 503 (GitHub rate-limited) → returns 503
  with `Retry-After: 60` header so the combobox can show a precise back-off
  countdown. Backs Plan 08's `<RepoCombobox>`.
- **GET /api/github/branches** → backend `/v1/github/branches?installation_id=&repo=`.
  Pass-through query string. Same 503 handling as repos. Backs Plan 08's
  `<BranchPicker>`.
- **POST /api/scans/from-github** → backend `/v1/scans/from-github`. Forwards JSON
  body 1:1. Preserves 422 (form validation) + 503 (queue overload) + 404
  (installation/branch not found); everything else → 500 + `{error: 'request_failed'}`.
  Backs Plan 09's `<ScanTriggerForm>`.
- **lib/types.ts**: adds `InstallationResp`, `RepoResp`, `BranchResp` exports.
  Extends `ScanGetResp` with 6 new optional fields (`error_message`,
  `source_path`, `github_installation_id`, `github_repo`, `github_branch`,
  `github_sha`) and relaxes `presigned_get_url` to `string | null`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Gated `<ScanViewerClient>` render on non-null `presigned_get_url`**
- **Found during:** Task 1 verification (`tsc --noEmit` after lib/types.ts edit)
- **Issue:** Plan 05 relaxed `ScanGetResp.presigned_get_url` to `string | null`,
  but `dashboard/app/(dashboard)/scans/[id]/page.tsx:33` was passing the value
  straight into `<ScanViewerClient initialPresignedUrl={...}>` which expects
  `string`. tsc errored:
  `Type 'string | null' is not assignable to type 'string'`.
- **Fix:** Wrapped the `<ScanViewerClient>` render in a null-check. When
  `presigned_get_url` is null (i.e. status is `pending` or `failed`), render a
  placeholder ("Scan in progress…" or "Scan failed: <error_message>") instead.
  No change to the ready-scan path. Plan 11 will replace this with a proper
  polling state machine driven by the new `error_message` + `status` fields
  also added in this plan.
- **Files modified:** `dashboard/app/(dashboard)/scans/[id]/page.tsx`
- **Commit:** f520100

## Threat Flags

None. The proxies are pure pass-throughs to existing backend endpoints (Plans 04
+ 05) — no new trust boundary introduced. T-07.5-07-01 (info disclosure via
backend error bodies) is mitigated by always returning generic
`{error: 'Internal error' | 'request_failed'}` bodies; only the *status code* is
propagated for 404/422/503. T-07.5-07-02 (query string tampering) is mitigated
upstream at the backend Pydantic schemas (Plan 03) — the proxy is intentionally
a thin pass-through.
