## Decisions Made

1. **Sentinel values `'all'`/`'any'` for empty-state Select items** — Radix Select rejects `value=""` at runtime (`A <Select.Item /> must have a value prop that is not an empty string`). Used `'all'` (date-range, source) and `'any'` (score-threshold) as sentinels; mapped back to `''` in the change handler so the URL semantics are unchanged (`pushFilter({ source: value === 'all' ? '' : value })`).
2. **`TabsList variant="line"` (not default `default`)** — UI-SPEC §/settings/* spec'd the visual as "underline strip", not a pill bar. The shadcn Tabs primitive ships both styles; `variant="line"` gives transparent background + bottom-border indicator on active, which is the closer behavioural match.
3. **`next/link` mock upgraded to `React.forwardRef`** — Radix Tabs `asChild` uses the Slot pattern (`React.cloneElement` + ref forwarding) to project role/data-state onto the child element. The previous mock destructured `{href, children, className}` and dropped all other props/refs, so the asChild merge silently produced a plain `<a>` without `role="tab"`. Forward all props + ref → role/data-state propagate correctly.
4. **Runtime combobox check uses `container.querySelectorAll`, not `getAllByRole`** — Testing Library's `getAllByRole` walks the accessibility tree, which under jsdom + 3× radix-ui Selects exceeds the default 5000ms timeout deterministically. The DOM query is equivalent for the assertion (`role="combobox"` is set as a literal attribute on `Select.Trigger`) and runs in <1ms.
5. **Plan acceptance criterion #4 of Task 2 (`grep -c "<TabsTrigger" ≥3`) intentionally returns 1** — The plan's own action snippet uses `TAB_ROUTES.map(t => <TabsTrigger ...>)` (one literal trigger inside a map). 1 match is faithful to the plan's example; the runtime test (`role="tab"` count === 3) is the substantive check. Documented here so the verifier doesn't flag a false positive.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `ScanFilters` props mismatch in plan test snippet**

- **Found during:** Task 1 RED (writing the new test from plan's snippet)
- **Issue:** Plan's example test code uses `<ScanFilters branches={['main']} sources={['cli']} />` but the actual component signature is `export function ScanFilters()` (no props — branches/sources are derived from URL searchParams server-side and never passed as props in either Phase 7 or any callsite — `app/(dashboard)/scans/page.tsx:42` uses `<ScanFilters />`). Calling with the documented props would only produce a React warning (unknown JSX attribute) — not a runtime failure — but the test would document a non-existent contract.
- **Fix:** Test renders `<ScanFilters />` with no props; mocks `useSearchParams` to return an empty params object. Functionally equivalent to plan intent (verifies shadcn primitives mount).
- **Files modified:** `dashboard/__tests__/scan-filters.test.tsx`
- **Commit:** `90852b6`

**2. [Rule 1 — Bug] `scans-table.test.tsx` debounce test broke from Radix Select render time**

- **Found during:** Task 1 GREEN — running the full vitest suite after migration.
- **Issue:** The pre-existing `ScanFilters debounce` test in `__tests__/scans-table.test.tsx` (which uses `vi.useFakeTimers()` set in `beforeEach` BEFORE `render(<ScanFilters />)`) timed out at the default 5000ms. Radix Select's internal layout effects depend on the real microtask queue / `requestAnimationFrame`; faking timers before render froze them. Additionally, rendering 3× Radix Select primitives under jsdom genuinely takes 3–5s before any assertion can run.
- **Fix:**
  1. Move `vi.useFakeTimers({ shouldAdvanceTime: true })` to AFTER `render(...)` so Radix's effects flush against real timers first.
  2. Bump the per-test timeout to 15000ms (3rd arg to `it(...)`) to absorb jsdom render cost.
- **Files modified:** `dashboard/__tests__/scans-table.test.tsx`
- **Commit:** `4f5b1fa` (rolled into Task 1 GREEN)

**3. [Rule 1 — Bug] `next/link` mock dropped Radix Slot props (Tabs migration)**

- **Found during:** Task 2 GREEN — runtime tests `data-state="active"` and `role="tab"` failed despite source-level Tabs migration succeeding.
- **Issue:** The pre-existing `next/link` mock in `settings-routes.test.tsx` only destructured `{href, children, className}` and dropped all other props/refs. Radix `Tabs.Trigger` with `asChild` uses `React.cloneElement` (via Slot) to inject `role="tab"`, `data-state`, `aria-selected`, and a forwarded ref onto the child element — those props were silently lost in the mock, so the rendered anchor had no Tabs semantics.
- **Fix:** Replace the mock with `React.forwardRef<HTMLAnchorElement, AnchorHTMLAttributes>(({children, ...rest}, ref) => <a ref={ref} {...rest}>{children}</a>)`. Now Slot's prop merge propagates correctly.
- **Files modified:** `dashboard/__tests__/settings-routes.test.tsx`
- **Commit:** `4ccb8e0`

**4. [Rule 1 — Bug] `getAllByRole('combobox')` deterministic timeout**

- **Found during:** Task 1 GREEN — runtime SelectTrigger assertion timed out at 5000ms in both isolated and full-suite runs.
- **Issue:** Testing Library's `getAllByRole` performs a full accessibility-tree walk (computes implicit roles, names, and ARIA-mapped attributes for every node). Under jsdom with 3× radix-ui Select primitives, this walk + radix's own focus/pointer effect chain exceeds 5s.
- **Fix:** Use `container.querySelectorAll('[role="combobox"]')` — `Select.Trigger` sets `role="combobox"` as a literal attribute, so the DOM query is semantically equivalent and completes in <1ms. Also bumped the per-test timeout to 15000ms as belt-and-braces against jsdom worker contention.
- **Files modified:** `dashboard/__tests__/scan-filters.test.tsx`
- **Commit:** `4f5b1fa`

**5. [Rule 1 — Bug] Existing settings-layout link assertions broke under Tabs (`role="link"` → `role="tab"`)**

- **Found during:** Task 2 GREEN — `screen.getByRole('link', { name: /members/i })` returned null.
- **Issue:** When Radix Tabs' `Trigger asChild` projects onto a `next/link`-backed `<a>`, the `role="tab"` attribute overrides the anchor's implicit `link` role. `getByRole('link')` then can't find the element. Same for the `border-amber-400` className assertion — that class belongs to the old hand-rolled nav, not the Tabs migration.
- **Fix:**
  - Replace `getByRole('link', ...)` with `document.querySelectorAll('a[href^="/settings/"]')` and assert href list contains all 3 expected paths.
  - Replace `border-amber-400` className check with `[data-state="active"]` query (the Tabs idiomatic active-state attribute).
- **Files modified:** `dashboard/__tests__/settings-routes.test.tsx`
- **Commit:** `4ccb8e0`
