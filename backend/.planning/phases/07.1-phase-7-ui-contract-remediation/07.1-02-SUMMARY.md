## Decisions Made

- **Toaster props chosen via UI-SPEC:** `richColors` (so the success-green / error-red theming uses the OKLCH severity tokens shadcn loaded in plan 01) and `position="bottom-right"` (UI-SPEC §Interaction). The plan's contract-test asserts the literal `position="bottom-right"` string, which acts as a poor-man's grep for spec compliance.
- **Bridge `isOpen`/`onClose` to shadcn `open`/`onOpenChange`:** Both modals already had call sites passing `isOpen`/`onClose`. Renaming the props would force a sweep across `MetadataHeader`, `CompareButton`, `share/page.tsx`, etc. — strictly out of scope for a primitive migration. The bridge `(o) => !o && onClose()` is a one-liner per modal and preserves the existing public API.
- **Surface generate-failure with duration:Infinity:** UI-SPEC §Copywriting explicitly calls for a sticky toast on share-link generation failure (the user has no other recovery cue once the modal closes). The existing inline `error` state inside the modal is preserved as a defence-in-depth surface so the failure is visible even if the toast is dismissed.
- **lucide-react `importOriginal` mock:** Plan 01's shadcn `DialogContent` imports `XIcon` for its built-in close button. The previous test-suite mock used a closed enumeration `{ Copy, Share2, X, Check }` and would have thrown `No "XIcon" export is defined on the "lucide-react" mock` once tests rendered DialogContent. Switching to `importOriginal()` + selective overrides fixes this generally — any future shadcn primitive that pulls additional icons will Just Work without test churn.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] lucide-react test mock missing XIcon caused all 5 new ShareModal tests to throw at DialogContent mount**

- **Found during:** Task 3, after rewriting ShareModal to use `<DialogContent>`, the first test run produced `Error: [vitest] No "XIcon" export is defined on the "lucide-react" mock` from inside `dashboard/components/ui/dialog.tsx:75:14` (DialogContent's `<XIcon />` close-button glyph).
- **Issue:** `__tests__/share-modal.test.tsx` already mocked `lucide-react` with a literal `{ Copy, Share2, X, Check }` map (preserved verbatim from the pre-shadcn test). shadcn's DialogContent imports `XIcon` (a different export), which the literal mock does not provide. Vitest mocks are exclusive — `XIcon` resolves to `undefined` and React throws.
- **Fix:** Switched the mock to `vi.mock('lucide-react', async (importOriginal) => { const actual = await importOriginal(); return { ...actual, Copy: ..., Share2: ..., X: ..., Check: ... } })`. The four explicit overrides keep the existing render-as-`<span data-testid="...">` behaviour (so existing assertions on `data-testid="icon-copy"` etc. still pass), and every other lucide icon (XIcon, Search, Eye, etc.) forwards to the real module.
- **Files modified:** `dashboard/__tests__/share-modal.test.tsx`
- **Verification:** `cd dashboard && npm test -- --run __tests__/share-modal.test.tsx` — 11/11 pass; full suite 111/111 pass.
- **Committed in:** `fa481b8` (Task 3, alongside the ShareModal source change — the test fix is part of the test-extension work).
