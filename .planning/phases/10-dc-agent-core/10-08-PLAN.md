<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → Go module proxy | Public proxy.golang.org; trust transitive deps via go.sum hashes |
| Released binary → end-user host | Binary downloaded over TLS from github.com; signature verification deferred to enterprise tier |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-08-01 | Tampering | dependency confusion / supply-chain attack | mitigate | `go mod verify` step in both ci.yml and release.yml validates go.sum hashes; pinned versions (Plan 10-01) prevent silent upgrades; new versions only enter via explicit `go get version@vX.Y.Z`. T-10-01-01 sibling. |
| T-10-08-02 | Spoofing | adversary commits and pushes a tag | accept | Repo permissions + branch protection are the mitigation; release.yml runs only on tag push. Tag signing is enterprise-tier (v1.2). |
| T-10-08-03 | Tampering | binary modified between build and download | mitigate | TLS to github.com prevents in-transit modification; SHA-256 SBOM in CAB packet (Plan 10-09) gives operators a hash they can verify post-download. |
| T-10-08-04 | Information Disclosure | tag-name leak via -ldflags | accept | `main.version` is intentionally set to the tag name and printed by the `version` subcommand — this IS the desired behavior. T-10-03-04 sibling. |
| T-10-08-05 | Tampering | CGO_ENABLED=1 sneaks into a future PR and breaks reproducibility | mitigate | Acceptance criteria locks `CGO_ENABLED: 0` literal in release.yml via grep; future executors cannot silently flip it. RESEARCH Pitfall 4. |
</threat_model>
