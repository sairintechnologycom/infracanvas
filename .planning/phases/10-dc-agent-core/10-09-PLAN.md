    ```markdown
    # InfraCanvas DC Agent — Change Advisory Board (CAB) Security Review Packet

    **Audience:** Enterprise procurement, IT security review boards, change advisory boards.
    **Phase:** 10 (DC Agent Core)
    **Version:** Generated against the InfraCanvas DC Agent at the tag listed in `sbom.cyclonedx.json`.
    **Last updated:** 2026-05-07.

    This packet contains the documents an enterprise reviewer needs to approve
    deployment of the InfraCanvas DC Agent (`infracanvas-agent`) in a regulated
    environment. It is intentionally self-contained — no external references
    required to complete a review.

    ## Contents

    | Document | Purpose |
    |----------|---------|
    | [architecture.md](./architecture.md) | System architecture, component map, dependency footprint |
    | [dataflow.md](./dataflow.md) | Data classification (operational/secret/PII), trust boundaries crossed |
    | [threat-model.md](./threat-model.md) | STRIDE threat register, mitigations, accepted risks |
    | [sbom.cyclonedx.json](./sbom.cyclonedx.json) | CycloneDX SBOM — every dep with version + hash |

    | [operator-runbook.md](./operator-runbook.md) | Step-by-step deployment guide for site operators |

    ## What the agent does (one-paragraph)

    The InfraCanvas DC Agent is a single-binary Go process that runs inside a
    customer's data centre, collects routing-table state from network devices
    (via NETCONF, SSH, or static config-file import) and NetFlow v9/IPFIX
    records from network exporters, and pushes the collected data over HTTPS
    to the InfraCanvas SaaS backend. It does NOT receive or execute any
    inbound commands; it does NOT modify device state; it does NOT transmit
    device credentials.

    ## What the agent does NOT do (Phase 10 scope)

    - Modify any device configuration (read-only collection only).
    - Transmit device username/password (read-only NETCONF service accounts
      live entirely on the agent host).
    - Maintain persistent state on disk (NetFlow buffer is in-memory only).
    - Accept any inbound network traffic except UDP/2055 NetFlow records.
    - Provide a remote-control plane (no inbound HTTP, no SSH server, no API).
    ```

    ## Component map

    | Component | Path | Purpose |
    |-----------|------|---------|
    | CLI entry | `cmd/infracanvas-agent/main.go` | cobra root + `run` daemon + `version` |
    | Config loader | `internal/config/` | `agent.yaml` + static route file parser |
    | NETCONF collector | `internal/netconf/` | nemith.io/netconf SSH transport, subtree filter |
    | SSH collector | `internal/ssh/` | golang.org/x/crypto/ssh, PTY + `terminal length 0` |
    | NetFlow listener | `internal/netflow/` | UDP :2055, goflow2/v2 decoder, ring buffer |
    | Push client | `internal/push/` | net/http POST, Bearer auth, retry-twice-then-drop |

    ## Process model

    - Single OS process. No fork/exec.
    - One goroutine per ticker (5 min routes / 1 min BGP / 30 s NetFlow flush).
    - One goroutine for the NetFlow UDP read loop.
    - Graceful shutdown on SIGINT/SIGTERM via signal.NotifyContext.

    ## Network footprint

    | Direction | Port/Proto | Peer | Purpose |
    |-----------|-----------|------|---------|
    | Outbound  | 830/tcp   | NETCONF-enabled devices | Route collection |
    | Outbound  | 22/tcp    | SSH-only devices        | Route collection |
    | Inbound   | 2055/udp  | NetFlow exporters       | Flow ingestion |
    | Outbound  | 443/tcp   | api.infracanvas.dev     | Push (HTTPS) |

    No other listeners, no other dialers. The agent does NOT talk to DNS
    servers other than via the OS resolver for `api.infracanvas.dev`.

    ## Dependency footprint

    Go module: `github.com/infracanvas/infracanvas/agent`, Go 1.25.

    Direct dependencies (locked versions in agent/go.mod):

    ```markdown
    # Data Flow & Classification

    ## Data Inventory

    | Datum | Source | Storage Lifetime | Classification | Crosses Trust Boundary? |
    |-------|--------|------------------|----------------|------------------------|
    | Device username | `agent.yaml` | Process lifetime + host filesystem | **Secret** (operational) | NO — never transmitted |
    | Device password | `agent.yaml` | Process lifetime + host filesystem | **Secret** (operational) | NO — never transmitted |
    | Site token | `agent.yaml` | Process lifetime + host filesystem | **Secret** (auth credential) | YES — sent in `Authorization: Bearer` header to api.infracanvas.dev (HTTPS) |
    | Routes (prefix, next-hop, protocol, metric) | NETCONF/SSH/file | In-memory until next push | Operational | YES — POST `/v1/agent/routes` (HTTPS) |
    | NetFlow records (5-tuple, bytes, packets) | UDP/2055 | In-memory ring buffer (~5 min) | Operational (network metadata) | YES — POST `/v1/agent/flows` (HTTPS) |
    | Decoded sampler IPs | NetFlow packet src | In-memory template cache | Operational | NO — not pushed beyond aggregate FlowRecord src_ip field |
    | Daemon logs | stdout/stderr | Operator-managed (systemd journal) | Operational + diagnostic | NO — local-only unless operator forwards |

    Classification scheme:
    - **Secret**: must never appear in logs, telemetry, or push payloads.
    - **Operational**: routine telemetry; encrypted in transit but not considered sensitive at rest.
    - **PII**: not collected by Phase 10 (no end-user identifiers in the data path).

    ## Trust Boundaries

    | # | Boundary | Crossed by |
    |---|----------|-----------|
    | TB-1 | Network device → Agent | NETCONF reply XML, SSH command stdout, NetFlow UDP packets |
    | TB-2 | Agent → Cloud backend | HTTPS POST with site_token Bearer + JSON payload |
    | TB-3 | Filesystem → Agent process | agent.yaml read at startup; static route files read on tick |

    ## Data NOT Transmitted (Phase 10 hard guarantees)

    - Device usernames / passwords are **never** included in any push payload.
      The `internal/push/client.go` only marshals `RoutesPayload` and `FlowsPayload`,
      neither of which contains a credential field. (Lock evidence:
      `grep -c "Username\|Password" agent/internal/push/types.go` returns 0.)
    - The agent does not collect device configuration text (no `show running-config`).
      Only routing-table state and BGP neighbour summaries are in scope.
    - The agent does not modify device state. NETCONF `<edit-config>` and SSH
      `configure terminal` are not exercised. The collector issues only
      `<get>` (NETCONF) and `show ip route` (SSH).

    ## Encryption in Transit

    - Backend push: TLS 1.2+ via Go stdlib `net/http` default. Certificate
      chain validated against the host trust store. No custom CA pinning in
      Phase 10 (deferred to enterprise tier).
    - NETCONF: SSH transport encrypts the protocol channel.
    - SSH CLI: SSH transport encrypts the protocol channel.
    - NetFlow UDP: **Not encrypted** — NetFlow is a UDP-only legacy protocol.
      Operators are advised to deploy the agent on the management VLAN where
      NetFlow exporters live.

    ## Encryption at Rest

    - `agent.yaml` is plaintext on the agent host filesystem; protected only
      by `chmod 600` (D-05). See [known-limitations.md](./known-limitations.md).
    - The backend stores the **SHA-256 hash** of the site token (not the
      plaintext) — the plaintext is returned exactly once at issuance and
      never persists in the cloud.
    ```

    ```markdown
    # Known Limitations (Phase 10)

    The InfraCanvas DC Agent is shipped in Phase 10 (Year 1) with the
    following acknowledged limitations. Each has a documented remediation
    path; enterprise tier (v1.2+) addresses the most security-significant ones.

    ## L-1: SSH host key verification disabled

    **What:** The agent uses `ssh.InsecureIgnoreHostKey()` for both NETCONF
    (port 830) and SSH-CLI fallback (port 22) connections to network devices.

    **Risk:** A network adversary capable of TCP MITM on the management VLAN
    could impersonate a device and capture device credentials sent during
    SSH authentication, OR feed crafted route data to the agent.

    **Why this posture in Phase 10:** Operators commonly run agents on the
    same management VLAN as devices, where the trust assumption is "no
    untrusted hosts on this VLAN". A `known_hosts` workflow requires a
    one-time human-trusted bootstrap, increasing deployment friction during
    initial pilot phase.

    **Operator mitigations available today:**
    - Deploy the agent on the same management VLAN as the devices.
    - Use ACLs to ensure only the management VLAN can reach 22/tcp + 830/tcp.
    - Use dedicated read-only NETCONF service accounts so MITM yields no
      escalation beyond read-only telemetry.

    **Future remediation:** Enterprise tier will add `known_hosts` /
    `FixedHostKey` verification with a one-shot bootstrap (`infracanvas-agent
    learn-hosts`) for operators who want certificate-equivalent identity
    on the SSH/NETCONF channel.

    ## L-2: Device credentials stored plaintext on agent host

    **What:** `agent.yaml` contains plaintext SSH/NETCONF passwords for each
    device the agent collects from. The file is protected only by Unix
    file permissions (`chmod 600`).

    **Risk:** Anyone with root access on the agent host can read device
    credentials. A compromised agent host yields all configured device
    credentials.

    **Why this posture in Phase 10:** OS-keyring integration is platform-
    specific (gnome-keyring on Linux, Keychain on macOS, no equivalent on
    bare-metal Linux servers without a desktop session). Phase 10 ships
    with the operator-managed-secrets posture for compatibility breadth.

    **Operator mitigations available today:**
    - chmod 600 the agent.yaml (operator-runbook step).
    - Use read-only NETCONF service accounts (T-10-04-01 sibling).
    - Run the agent process as a non-root user via systemd `User=`.
    - Use full-disk encryption on the agent host.

    **Future remediation:** Enterprise tier will support SSH-key-based
    auth for SSH/NETCONF (no passwords) and HashiCorp Vault / AWS SSM
    Parameter Store integration for credential retrieval at runtime.

    ## L-3: Site token has no automatic rotation

    **What:** The site token issued by `POST /v1/sites` is long-lived and has
    no expiry. The operator must manually rotate by:
    1. Issuing a new token via `POST /v1/sites` (creates a new site).
    2. Updating agent.yaml with the new token.
    3. Restarting the agent.
    4. Deleting the old `dc_sites` row to revoke the old token.

    **Risk:** A token leak (e.g. via `agent.yaml` exfiltration) is exploitable
    indefinitely until manual operator action.

    **Operator mitigations available today:**
    - Audit `dc_sites` table monthly; delete unused tokens.
    - Use a token-naming convention (e.g. `dc-east-2026-q2`) so operators
      can identify which site is at risk.
    - Monitor structlog `agent_routes_received` events for unexpected site_id
      activity.

    **Future remediation:** Enterprise tier will add token-rotation API
    (`POST /v1/sites/{id}/rotate-token`), automatic expiry (90-day default),
    and a refresh-token model.

    ## L-4: NetFlow data not encrypted in transit

    **What:** NetFlow v9 / IPFIX is a UDP-only legacy protocol that does
    NOT encrypt the wire payload.

    **Risk:** Anyone on the network path between exporters and the agent
    can passively observe flow records.

    **Why this posture in Phase 10:** NetFlow is the de-facto standard for
    network telemetry and exporters do not support encrypted variants
    (IPFIX-over-TLS exists but field deployment is rare).

    **Operator mitigations available today:**
    - Deploy the agent on the same management VLAN as the exporters
      (defense in depth — same advice as L-1).
    - Use IPSec or VLAN-level isolation if NetFlow traverses untrusted
      segments.

    ## L-5: No persistent NetFlow storage

    **What:** The NetFlow ring buffer is in-memory only. If the agent
    crashes or restarts, the un-flushed buffer (up to 30 seconds worth)
    is lost.

    **Risk:** Operational data loss only — no security impact.

    **Why this posture in Phase 10:** Disk-backed queue adds dependency
    surface (SQLite + corruption recovery + lifecycle management) for
    a 30-second loss window.

    **Future remediation:** Disk-backed queue is on the Phase 10+
    deferred-items list (CONTEXT.md).

    ## L-6: No mTLS to backend

    **What:** Backend authentication is bearer-token only, not mTLS.

    **Risk:** A bearer-token leak yields full push capability for the
    affected site until manual revocation. mTLS would require possession
    of the per-site client certificate AND its private key.

    **Why this posture in Phase 10:** mTLS adds PKI infrastructure that
    smaller teams cannot operationally maintain.

    **Future remediation:** mTLS deferred to enterprise tier (v1.2+) per
    CONTEXT.md.

    ## L-7: No release-binary signing

    **What:** Agent binaries attached to GitHub releases are not signed
    (no Sigstore cosign, no GPG).

    **Risk:** TLS to github.com prevents in-transit modification but
    does not provide post-download attestation.

    **Operator mitigations available today:**
    - Verify SHA-256 hash from this CAB packet's [sbom.cyclonedx.json](./sbom.cyclonedx.json)
      after download.
    - Pin a known-good binary on internal artifact repositories.

    **Future remediation:** Sigstore cosign signing is on the enterprise-tier
    roadmap.
    ```

    # Operator Runbook — DC Agent Deployment

    Audience: A site operator deploying the InfraCanvas DC Agent at a
    customer's data centre. This is the minimum-viable end-to-end deployment
    for Phase 10.

    Estimated time: ~20 minutes.

    ## Prerequisites

    - A Linux amd64 or macOS arm64 host on the management VLAN.
    - Outbound HTTPS reachability to `api.infracanvas.dev` (port 443/tcp).
    - Inbound UDP reachability on port 2055 from NetFlow exporters (only
      required if collecting flow data).
    - SSH/NETCONF reachability from the agent host to each target device.
    - A team owner account on the InfraCanvas SaaS dashboard.

    ## Step 1 — Provision a site token

    On a workstation with `curl`:

    ```bash
    # Replace TEAM_OWNER_JWT with a Clerk session JWT for a team owner.
    curl -X POST https://api.infracanvas.dev/v1/sites \
      -H "Authorization: Bearer TEAM_OWNER_JWT" \
      -H "Content-Type: application/json" \
      -d '{"name": "dc-east-1"}'
    ```

    Response (the `site_token` is shown ONCE — store it now):
    ```json
    {
      "site_id": "uuid",
      "name": "dc-east-1",
      "site_token": "ic_site_abcdef..."
    }
    ```

    Treat `site_token` like a long-lived bearer secret. If it leaks, delete
    the corresponding `dc_sites` row to revoke and reissue (see [known-
    limitations.md](./known-limitations.md) §L-3).

    ## Step 2 — Place agent.yaml

    On the agent host:
    ```bash
    sudo mkdir -p /etc/infracanvas
    sudo $EDITOR /etc/infracanvas/agent.yaml
    ```

    Paste a config based on [agent.yaml.example](../../agent.yaml.example),
    substituting the site_token from Step 1 and your device credentials.

    Then lock down permissions:
    ```bash
    sudo chmod 600 /etc/infracanvas/agent.yaml
    sudo chown root:root /etc/infracanvas/agent.yaml
    ```

    ## Step 3 — Install the agent binary

    Download the appropriate release binary:
    ```bash
    # Linux amd64:
    sudo curl -fsSL https://github.com/infracanvas/infracanvas/releases/latest/download/infracanvas-agent-linux-amd64 \
      -o /usr/local/bin/infracanvas-agent
    sudo chmod 755 /usr/local/bin/infracanvas-agent

    # macOS arm64:

      -o /usr/local/bin/infracanvas-agent
    chmod 755 /usr/local/bin/infracanvas-agent
    ```


    Expected: a semver string matching the release tag (e.g. `v0.1.0`).

    ## Step 4 — Verify connectivity

    Run a one-shot test (Ctrl-C to exit after first push cycle):
    ```bash
    sudo /usr/local/bin/infracanvas-agent run --config /etc/infracanvas/agent.yaml
    ```

    Within 5 minutes you should see structured JSON logs with:
    - `agent_starting` (with version + device count)
    - `netflow_listener_started` (if NetFlow is configured)
    - per-tick collection logs

    On the SaaS side, the dashboard's `dc_sites` row for this site should
    show recent activity (Phase 11+ adds the UI; until then check Axiom
    structlog for `agent_routes_received` events).

    ## Step 5 — Install as a systemd service

    Create `/etc/systemd/system/infracanvas-agent.service`:
    ```ini
    [Unit]
    Description=InfraCanvas DC Agent
    After=network-online.target
    Wants=network-online.target

    [Service]
    Type=simple
    User=infracanvas
    Group=infracanvas
    ExecStart=/usr/local/bin/infracanvas-agent run --config /etc/infracanvas/agent.yaml
    Restart=on-failure
    RestartSec=10
    StandardOutput=journal
    StandardError=journal
    # Hardening
    NoNewPrivileges=true
    ProtectSystem=strict
    ProtectHome=true
    PrivateTmp=true
    ReadOnlyPaths=/etc/infracanvas

    [Install]
    WantedBy=multi-user.target
    ```

    ```bash
    sudo chmod 600 /etc/infracanvas/agent.yaml                       # still chmod 600
    sudo systemctl daemon-reload
    sudo systemctl enable --now infracanvas-agent
    ```

    Verify:
    ```bash
    sudo systemctl status infracanvas-agent
    sudo journalctl -u infracanvas-agent -f
    ```
