## Summary

Phase 5 restructures `viewer/` from a single-output Vite app into a dual-output package: one build produces the existing `dist/index.html` single-file HTML (CLI path, behavior byte-for-byte identical), and a second build produces `dist/lib/` containing an ESM React component library with pre-compiled CSS and TypeScript declarations. The dual-build is implemented via two separate Vite configs invoked sequentially in `npm run build`, which is the cleanest approach for Vite 6 because `build.lib` and `vite-plugin-singlefile` conflict at the plugin level.

The three technically demanding areas are: (1) getting `'use client'` preserved in the library bundle output so Next.js 15 App Router does not attempt to SSR `@xyflow/react`; (2) generating a pre-compiled `styles.css` from Tailwind v4 that contains exactly the classes used by the library's source files (not the whole project); and (3) refactoring the Zustand module-level singleton to a `createStore`-based factory with React Context while keeping the 136 existing Vitest tests passing with minimal changes.

**Primary recommendation:** Two separate Vite config files (`vite.config.app.ts` for the single-file HTML, `vite.config.lib.ts` for the ESM library) plus a CSS pre-compilation step via `@tailwindcss/cli`. The library config uses `preserveModules: true` + rollup `output.banner` to emit `'use client'` at every ESM entry file. Declarations are generated by `vite-plugin-dts` v4. Store tests need a one-time `beforeEach` wrapper swap; component tests are unchanged.

### Pattern 1: Two Separate Vite Configs (Dual Build)

**What:** One config for the app (single-file HTML), one for the library (ESM). Invoked sequentially via npm scripts.
**When to use:** Whenever `vite-plugin-singlefile` and `build.lib` must coexist — they conflict because `vite-plugin-singlefile` forcibly sets `assetsInlineLimit` to `Infinity`, which breaks library output. Separate configs is the canonical Vite 6 solution [CITED: vite.dev/config/, github.com/vitejs/vite/discussions/8222].

**Example — `viewer/vite.config.app.ts`:**

```typescript
// Source: v6.vite.dev/guide/build
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import tailwindcss from '@tailwindcss/vite'
import { viteSingleFile } from 'vite-plugin-singlefile'

export default defineConfig({
  plugins: [react(), tailwindcss(), viteSingleFile()],
  build: {
    target: 'esnext',
    assetsInlineLimit: 100_000_000,
    cssCodeSplit: false,
    outDir: 'dist',
  },
  test: {
    environment: 'jsdom',
    globals: true,
    setupFiles: ['./src/__tests__/setup.ts'],
  },
})
```

**Example — `viewer/vite.config.lib.ts`:**

```typescript
// Source: v6.vite.dev/config/build-options (build.lib)
import { resolve } from 'node:path'
import { fileURLToPath } from 'node:url'
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import dts from 'vite-plugin-dts'

const __dirname = fileURLToPath(new URL('.', import.meta.url))

export default defineConfig({
  plugins: [
    react(),
    dts({
      include: ['src'],
      exclude: ['src/__tests__', 'src/main.tsx', 'src/sample-data.ts'],
      outDir: 'dist/lib',
      tsconfigPath: './tsconfig.lib.json',   // separate tsconfig, see §4
    }),
  ],
  build: {
    lib: {
      entry: resolve(__dirname, 'src/index.ts'),
      formats: ['es'],
      fileName: 'index',
      cssFileName: 'styles',
    },
    outDir: 'dist/lib',
    copyPublicDir: false,
    rollupOptions: {
      external: [
        'react',
        'react-dom',
        'react/jsx-runtime',
        '@xyflow/react',
        'zustand',
        'zustand/vanilla',
        'dagre',
        'elkjs',
        'lucide-react',
        'aws-react-icons',
      ],
      output: {
        // 'use client' banner at every entry chunk (see §3)
        banner: (chunk) => {
          if (chunk.isEntry) return `'use client'\n`
          return ''
        },
        preserveModules: false,   // single-chunk ESM is fine for this package size
        assetFileNames: '[name][extname]',  // places CSS as dist/lib/styles.css
      },
    },
  },
})
```

**`viewer/package.json` scripts:**

```json
{
  "scripts": {
    "build": "tsc -b && npm run build:app && npm run build:lib && npm run build:css",
    "build:app": "vite build --config vite.config.app.ts",
    "build:lib": "vite build --config vite.config.lib.ts",
    "build:css": "npx @tailwindcss/cli -m -i src/lib-styles.css -o dist/lib/styles.css",
    "postbuild": "cp dist/index.html ../cli/infracanvas/export/viewer_template.html",
    "dev": "vite --config vite.config.app.ts",
    "preview": "vite preview --config vite.config.app.ts",
    "test": "vitest run",
    "test:watch": "vitest"
  }
}
```

**Note on `preserveModules`:** Setting `preserveModules: true` produces one JS file per source file (better tree-shaking) but requires downstream consumers to handle CSS themselves (each component imports its own CSS in the lib-inject-css pattern). Since D-07 mandates a single `styles.css` import, a single-chunk ESM bundle (`preserveModules: false`) is correct and simpler.

### Pattern 2: `'use client'` Directive Emission

**Recommendation: rollup `output.banner` on entry chunks.**

**Why:** Next.js 15 App Router documentation explicitly calls this out as the pattern library authors should use [CITED: nextjs.org/docs/app/getting-started/server-and-client-components, "Advice for Library Authors" section]. The alternative (`rollup-preserve-directives` plugin) requires `preserveModules: true` which conflicts with D-07's single-CSS-file requirement. The banner approach is also what Vercel Analytics and React Wrap Balancer (Vercel-maintained libraries) use in production.

**How Next.js 15 handles it:** When Next.js encounters an import from `node_modules` where the resolved file begins with `'use client'`, it treats that entire module (and its imports) as part of the client bundle — no SSR attempt. The directive does NOT need to be present in the consuming page; it only needs to be at the library's entry point. [CITED: nextjs.org/docs/app/api-reference/directives/use-client]

**Config (already shown in lib config above):**

```typescript
output: {
  banner: (chunk) => {
    if (chunk.isEntry) return `'use client'\n`
    return ''
  },
}
```

**Result in `dist/lib/index.js`:**

```javascript
'use client'

// ... bundled component code
```

**Alternative considered and rejected:** `vite-plugin-preserve-directives` — requires `preserveModules: true`, and community reports [CITED: github.com/vitejs/vite/discussions/13200] show it still has edge cases with Vite 6 bundling. The banner approach is lower-complexity and battle-tested.

### Pattern 3: TypeScript Declaration Generation

**Recommendation: `vite-plugin-dts` v4.5.4.**

**Why over `tsc --emitDeclarationOnly`:**
- `vite-plugin-dts` runs in-process during `build:lib`, no second toolchain invocation
- Works with Vite 6 (peerDeps: `*`) [VERIFIED: npm registry]
- Handles the `allowImportingTsExtensions: true` problem automatically (the existing `tsconfig.json` sets `noEmit: true` which is required when `allowImportingTsExtensions` is on; `vite-plugin-dts` manages its own emission)
- The separate-tsconfig approach is needed because `isolatedModules: true` in the current `tsconfig.json` plus `noEmit: true` means `tsc --emitDeclarationOnly` would fail

**Required `viewer/tsconfig.lib.json` (new file):**

```json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "noEmit": false,
    "allowImportingTsExtensions": false,
    "declaration": true,
    "declarationDir": "dist/lib",
    "emitDeclarationOnly": true
  },
  "include": ["src"],
  "exclude": ["src/__tests__", "src/main.tsx", "src/sample-data.ts"]
}
```

**Known issue:** `rollupTypes: true` in `vite-plugin-dts` v4 has reported issues with `"type": "module"` packages [CITED: github.com/qmhc/unplugin-dts/issues/345]. Use `rollupTypes: false` (default) to avoid. Declarations ship as per-file `.d.ts` alongside the JS files, which is the standard pattern and works well with the `"exports"` map.

When Phase 7 imports `<DiagramCanvas>` from `@infracanvas/viewer`, it can do so from a Server Component file — Next.js will correctly trace the `'use client'` boundary at the library's `index.js` and assign all viewer components to the client bundle. No wrapper file is needed.

### Pattern 5: Tailwind v4 Pre-compiled CSS

**Problem:** `@tailwindcss/vite` inside `build.lib` mode either includes the entire project's CSS or produces empty output. Both behaviors are confirmed community-reported bugs [CITED: github.com/tailwindlabs/tailwindcss/discussions/17025, github.com/tailwindlabs/tailwindcss/discussions/17736].

**Solution:** Use `@tailwindcss/cli` as a separate build step with `@source` scoping.

**`viewer/src/lib-styles.css` (new file):**

```css
/* Tailwind v4 library CSS entry */
/* source(none) disables automatic project scanning */
/* @source directives pin exactly which files are scanned */
@import "tailwindcss" source(none);
@source "./components";
@source "./components/flowmap";
@source "./components/flowmap/edges";
@source "./components/flowmap/lib";
@source "./components/flowmap/nodes";
@source "./App.tsx";
```

**Build command (in `build:css` script):**

```bash
npx @tailwindcss/cli -m -i src/lib-styles.css -o dist/lib/styles.css
```

The `-m` flag minifies. The `@source` directives tell Tailwind v4 to scan only the component files — not `node_modules`, not test files, not sample data. [CITED: tailwindcss.com/docs/detecting-classes-in-source-files]

**Consumer side (Next.js dashboard, Phase 7):**

```typescript
// app/layout.tsx
import '@infracanvas/viewer/styles.css'  // resolves via package.json exports map
```

The dashboard imports the static, pre-compiled CSS file. It can also run its own Tailwind for its own UI classes — the two CSS sets are independent and do not conflict. No `@source` directive pointing to `node_modules/@infracanvas/viewer` is needed because the library's CSS is already compiled.

### Pattern 6: Workspace Wiring

**Recommendation: npm workspaces at the repo root.**

**Why over `file:` reference:** npm workspaces creates a symlink in `dashboard/node_modules/@infracanvas/viewer` → `../../viewer`, making `import '@infracanvas/viewer'` resolve correctly. `file:` also works but npm workspaces handles hoisting automatically and is the standard monorepo pattern [CITED: docs.npmjs.com/cli/v11/using-npm/workspaces]. The `cli/` Python package is NOT added to workspaces — workspaces are JS-only.

**Root `package.json` (new file):**

```json
{
  "name": "infracanvas-monorepo",
  "private": true,
  "workspaces": ["viewer"]
}
```

When Phase 7 adds `dashboard/` (a Next.js app), update to `"workspaces": ["viewer", "dashboard"]`. For now, only `viewer/` is a workspace member.

**`viewer/package.json` exports map:**

```json
{
  "name": "@infracanvas/viewer",
  "version": "0.1.0",
  "private": true,
  "type": "module",
  "files": ["dist/lib"],
  "main": "./dist/lib/index.js",
  "module": "./dist/lib/index.js",
  "types": "./dist/lib/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/lib/index.d.ts",
      "import": "./dist/lib/index.js"
    },
    "./styles.css": "./dist/lib/styles.css"
  },
  "peerDependencies": {
    "react": "^18.0.0 || ^19.0.0",
    "react-dom": "^18.0.0 || ^19.0.0"
  },
  "sideEffects": ["**/*.css"]
}
```

**Why this shape works with Next.js 15:** Next.js (Turbopack + Webpack 5) resolves `"exports"` maps. The `"./styles.css"` subpath export is the canonical way to expose a CSS file. Next.js allows CSS imports from `node_modules` since v9.5.4. TypeScript resolves types from the `"types"` condition. The `"sideEffects": ["**/*.css"]` prevents bundlers from tree-shaking the CSS import. [CITED: nextjs.org/docs/app/getting-started/css]

### Anti-Patterns to Avoid

- **Putting `@tailwindcss/vite` in the lib config:** It scans the wrong files and produces either bloated or empty CSS in `build.lib` mode.
- **`rollupTypes: true` in `vite-plugin-dts`:** Known breakage with `"type": "module"` packages on vite-plugin-dts v4 [CITED: github.com/qmhc/unplugin-dts/issues/345].
- **`preserveModules: true` with banner:** The banner applies only to entry chunks, not to preserved module files — each preserved module would need `'use client'` individually, which requires `rollup-preserve-directives`. Not needed here since we use `preserveModules: false`.
- **Externalizing `@fontsource/*`:** Font packages are safe to bundle into the single-file HTML (they are already inlined). In the lib build, they should also be externalized or excluded — fonts are imported only in `main.tsx`, which is excluded from the lib build entirely.
- **Forgetting `sideEffects: ["**/*.css"]`:** Without this, Webpack/Turbopack may eliminate the CSS import via tree-shaking in the consuming Next.js app.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| TypeScript declaration files | Custom `tsc` orchestration script | `vite-plugin-dts` | Handles `isolatedModules`, `.tsx` files, Vite lifecycle integration |
| CSS class scanning | Manual grep / regex of component source | Tailwind v4 `@source` directives | Tailwind's scanner handles all class variants, conditional classes, template literals |
| Workspace symlinks | `npm link` + manual path setup | npm workspaces | npm workspaces auto-hoists, handles reinstall correctly |
| Bundle size gate | Custom Node script reading file stats | `du -k dist/index.html` in CI (simple and sufficient) | No additional dependency; a single shell command |
| `'use client'` injection | Custom rollup plugin | `output.banner` callback | Rollup 4 (bundled in Vite 6) supports per-chunk banner callbacks natively |

**Key insight:** The major footgun in React component library packaging is managing CSS. The Tailwind v4 + `build.lib` combination has known issues that the community has already worked around — use the CLI approach rather than fighting the Vite plugin in library mode.

## Zustand Store Factory + Context Pattern

### Current Pattern (Singleton)

```typescript
// store.ts — current
export const useStore = create<StoreState>((set) => ({ ... }))
```

All components call `useStore(s => s.foo)` — module-level singleton shared across all mounts.

### Target Pattern (Factory + Context)

```typescript
// store.ts — after refactor
import { createStore } from 'zustand/vanilla'
import { useStore } from 'zustand'
```

## Test Migration Strategy

### Current State

- 136 tests total, 130 passing (6 pre-existing failures in `PathEdge.test.tsx` — not introduced by Phase 5)
- Tests use `useStore` module import directly:
  - `store.test.ts`: calls `useStore.setState(...)` and `useStore.getState()` directly
  - Component tests: `useStore.setState({...})` in `beforeEach`
  - Tab tests: `useStore.setState({ activeTab, hasFlowMap })` in `beforeEach`

### Minimal Migration Path

The key insight: Zustand's `create` (from `zustand`) returns a hook that has `.getState()` and `.setState()` attached. After migrating to `createStore` (from `zustand/vanilla`), the store API (`getState`, `setState`) is the same, but the hook (`useStore`) is created by `useStore(store, selector)` from `zustand`.

**Store test migration:** Export the default store instance for test use:

```typescript
// store.ts — add export
export const _defaultStore = createViewerStore()  // for test access only
```

Then in `store.test.ts`, replace `useStore.setState(...)` with `_defaultStore.setState(...)` and `useStore.getState()` with `_defaultStore.getState()`. The mock data structures are identical — only the access path changes.

**Alternatively** (zero store.test.ts changes): Keep the module-level singleton alongside the factory:

```typescript
// store.ts
export const useStore = create<StoreState>(stateCreator)  // legacy singleton
export function createViewerStore() {
  return createStore<StoreState>(stateCreator)
}
```

Components that switch to `useViewerStore` get Context-based isolation; `store.test.ts` continues using `useStore` directly. This is the lowest-risk migration — the singleton stays for tests and CLI HTML; the factory is the NEW path for dashboard. The two coexist without conflict because `useStore` is a different store instance from what `ViewerProvider` creates.

**Recommendation:** Use the coexistence approach. Zero test rewrites; factory is available for dashboard. After Phase 7 validation, `useStore` singleton can be deprecated and removed.

### Component Tests (TabBar, FlowMapCanvas, etc.)

Component tests call `useStore.setState({...})` in `beforeEach`. These work because `useStore` is the module singleton — and the module singleton continues to exist. The only components that MUST use `useViewerStore` are those exported in the library entry (since the dashboard will provide a `ViewerProvider`, not a singleton). For Phase 5, components can internally use either — tests continue passing without changes.

**Expected test changes: 0 in any component test file.**

## Common Pitfalls

### Pitfall 1: `vite-plugin-singlefile` + `build.lib` Conflict

**What goes wrong:** If both are in the same Vite config, `vite-plugin-singlefile` sets `assetsInlineLimit` to `Infinity` globally, which causes the library build to inline all its assets (including external CSS) rather than emitting separate files.
**Why it happens:** `vite-plugin-singlefile` modifies global Vite config options, not just its own plugin scope.
**How to avoid:** Always use separate Vite configs (Pattern 1 above).
**Warning signs:** Library build produces a single giant `.js` file with inlined CSS instead of separate `index.js` + `styles.css`.

### Pitfall 2: Tailwind v4 CSS Empty in Library Build

**What goes wrong:** `@tailwindcss/vite` in the lib config produces an empty `styles.css` or includes every class in the entire project.
**Why it happens:** The Tailwind Vite plugin uses the Vite root to scan files. In `build.lib` mode, the plugin's scanning context doesn't narrow to just the component source files.
**How to avoid:** Use `@tailwindcss/cli` as a separate script (Pattern 5 above).
**Warning signs:** `dist/lib/styles.css` is 0 bytes, or it is much larger than expected (> 100 KB).

### Pitfall 3: `'use client'` Stripped by Rollup

**What goes wrong:** Rollup's optimization pass treats `'use client'` as an unused string literal and removes it from the bundle.
**Why it happens:** `'use client'` looks like an expression statement to Rollup's AST. Without the banner approach, it gets stripped.
**How to avoid:** Use `output.banner` (not a source-level directive). The banner string is injected after bundling, bypassing Rollup's optimizer.
**Warning signs:** Running `head -1 dist/lib/index.js` does not show `'use client'`. Next.js reports "Client Components cannot be async functions" or server-side `window is not defined` errors.

### Pitfall 4: `@types/react` Version Mismatch in Library Build

**What goes wrong:** `tsconfig.lib.json` picks up `@types/react@18` but the dashboard (Phase 7) uses React 19 types, causing `.d.ts` incompatibilities.
**Why it happens:** The library's declared types reference React 18 internal types.
**How to avoid:** Use `react: "^18.0.0 || ^19.0.0"` in peerDependencies and update `@types/react` devDependency to `^18.0.0 || ^19.0.0`. The React types for 18 and 19 are broadly compatible for component props.
**Warning signs:** TypeScript errors in dashboard import of `<DiagramCanvas>` about incompatible JSX types.

### Pitfall 5: Store Singleton Shared Across Dashboard Pages

**What goes wrong:** If the dashboard uses the module-level `useStore` singleton instead of the `ViewerProvider` factory, switching between two scan pages does not clear the previous scan's graph state.
**Why it happens:** Module singletons are shared across all render trees in the same JS runtime.
**How to avoid:** Dashboard MUST use `<ViewerProvider>` which creates a fresh store per mount. The singleton `useStore` is intentionally NOT exported from the library's public API for this reason.
**Warning signs:** Navigating from `/scans/1` to `/scans/2` shows scan 1's data until `setGraph` is called.

### Pitfall 6: `noEmit: true` + `allowImportingTsExtensions: true` Blocks Declaration Emit

**What goes wrong:** Running `tsc --emitDeclarationOnly` fails because the existing `tsconfig.json` requires `noEmit: true` when `allowImportingTsExtensions: true`.
**Why it happens:** TypeScript enforces that `allowImportingTsExtensions` is only valid when the compiler is not emitting, because `.ts` extensions in imports would need runtime resolution.
**How to avoid:** Use a separate `tsconfig.lib.json` that overrides `allowImportingTsExtensions: false` and `noEmit: false`. `vite-plugin-dts` accepts a `tsconfigPath` option pointing to this file.
**Warning signs:** `tsc` fails with "Option 'allowImportingTsExtensions' can only be used when 'noEmit' is set to true."

## React 18 || 19 CI Smoke Test

**Approach: GitHub Actions matrix with `--overrideInstall` (npm install override).**

Since the viewer package itself stays on React 18 (CLAUDE.md constraint), the CI smoke test validates that the compiled library output is importable and renderable under both React 18 and React 19 without peer dep conflicts.

```yaml
name: Viewer Peer Compatibility
on:
  push:
    paths: ['viewer/**']

jobs:
  peer-compat:
    strategy:
      matrix:
        react-version: ['18', '19']
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'

      - name: Install dependencies
        working-directory: viewer
        run: npm ci

      - name: Build package
        working-directory: viewer
        run: npm run build

      - name: Override React version
        working-directory: viewer
        run: |
          npm install \
            react@${{ matrix.react-version }} \
            react-dom@${{ matrix.react-version }} \
            @types/react@${{ matrix.react-version }} \
            @types/react-dom@${{ matrix.react-version }} \
            --legacy-peer-deps

      - name: Run test suite
        working-directory: viewer
        run: npm test
```

**Why `--legacy-peer-deps`:** `@testing-library/react@16.3.2` declares `react: "^18.0.0 || ^19.0.0"` [VERIFIED: npm registry]. npm strict peer resolution may still conflict on some minor version combos. `--legacy-peer-deps` is the standard workaround for CI override scenarios.

**React 19 latest available:** `19.2.5` [VERIFIED: npm registry].

**Important:** This CI job tests the built library under both React versions, not the source. The source always builds with React 18. The test suite (136 tests) runs under React 19 after the override install — this catches API breakage (e.g., if `@testing-library/react` or `@xyflow/react` have undiscovered React 19 incompatibilities).

|--------------|------------------|--------------|--------|

- Manual `content: ['./src/**/*.{ts,tsx}']` glob in `tailwind.config.js`: Replaced by automatic file detection + `@source` in v4. The v4 CLI does not use `tailwind.config.js` at all.
- `vite build --mode lib` for dual-build: Not idiomatic; `--config` flag with separate files is the documented approach.

## Open Questions

1. **`ResourceNode` naming collision**
   - What we know: `ResourceNode` is both a component (`./components/ResourceNode.tsx`) and a type (`./types.ts`).
   - What's unclear: Whether the dashboard will prefer the component name or the type name in import statements.
   - Recommendation: Export the component as `ResourceNodeComponent` and the type as `ResourceNode` (types are erased at runtime, so the names don't conflict at the JS level — only in TypeScript's namespace).

2. **`elkjs` bundling in library mode**
   - What we know: `elkjs` is a WASM-backed layout library used by `FlowMapCanvas`. It's a peer dep in the current `package.json`.
   - What's unclear: Whether `elkjs` should be externalized in the lib build or bundled. If externalized, the Next.js dashboard must also install it.
   - Recommendation: Externalize it (add to `rollupOptions.external`). It's already a direct dep in `viewer/package.json`; the dashboard will get it via workspace hoisting.

3. **Font imports in `main.tsx`**
   - What we know: `main.tsx` imports `@fontsource/inter` and `@fontsource/jetbrains-mono`.
   - What's unclear: These are NOT in `src/index.ts`, so the lib build correctly excludes them. But the CLI HTML build needs them inlined.
   - Recommendation: Fonts stay in `main.tsx` only. Library consumers (dashboard) load their own fonts via Next.js `next/font` or their own CSS.
