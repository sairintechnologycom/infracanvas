# Phase 5.1 Plan 03: CLI UX (--quiet / --json / --open) Summary

Land the CLI UX deliverables for Success Criterion 3 — add `--quiet` (one-line summary), `--json` (JSON to stdout), and `--open` (browser) flags across **scan**, **plan**, and **export** — while resolving the `--quiet` semantic collision flagged by PATTERNS.md note 1 and CONTEXT.md D-03.

## Objective Recap

Three orthogonal output-shape flags for three output shapes:

| Flag | stdout | exit code | applies to |
|------|--------|-----------|------------|
| `--quiet` / `-q` | one-line human summary | 0 on success | scan / plan / export |
| `--json` | full JSON (parseable) | 0 regardless of findings | scan / plan |
| `--ci` | full JSON (parseable) | 1 if findings ≥ threshold, else 0 | scan only (unchanged) |
| `--open` | (no change to stdout) | 2 if `--format != html`, else unchanged | scan / plan / export |

Plus: move the parse-error warning loop AFTER `resolve_modules()` and reroute to stderr so submodule errors surface and `--quiet` stdout stays clean (D-01 tail).

## Final Flag Matrix

| Command | `--quiet` | `--json` | `--open` | `--ci` |
|---------|-----------|----------|----------|--------|
| `scan`   | ✅ one-liner | ✅ stdout JSON, exit 0 | ✅ HTML+browser | ✅ JSON+findings-gated |
| `plan`   | ✅ drift one-liner `+A ~C -D · delta ±$X.XX/mo` | ✅ plan-annotated graph JSON, exit 0 | ✅ HTML+browser | ❌ (not applicable) |
| `export` | ✅ resource-count one-liner `· exported to <path>` | ❌ (format=json IS the stdout JSON output) | ✅ HTML+browser | ❌ |

### Mutual exclusions (exit code 2)

| Combination | scan | plan | export |
|-------------|------|------|--------|
| `--open` + `--format json` | exit 2 | exit 2 | exit 2 |
| `--quiet` + `--json` | exit 2 | exit 2 | — (no --json on export) |
| `--quiet` + `--ci` | exit 2 | — | — |

### TTY-gated tick character

Success summary prefix:
- `✓` when `sys.stdout.isatty()` is true (terminal)
- `OK` when piped or in tests (CliRunner is non-TTY)

## Migration Notes (breaking changes)

### `scan --quiet` semantic flip

**Before (Phase 5):** `infracanvas scan <dir> --quiet` emitted the full scan JSON to stdout and exited 0 regardless of findings (overlapping with `--ci` minus the exit-code coupling).

**After (Phase 5.1):** `infracanvas scan <dir> --quiet` emits a single human-readable line:
```
✓ 6 resources · 10 findings · score 17
```

**Script migration path for CI pipelines previously calling `scan ... --quiet | jq ...`:**
```diff
- infracanvas scan ./tf --quiet | jq '.nodes | length'
+ infracanvas scan ./tf --json  | jq '.nodes | length'
```

### `export <report>` default-browser-open removed

**Before:** `infracanvas export report.json --format html` unconditionally opened the browser.

**After:** browser opens only when `--open` is passed. Scripted users who relied on the auto-open behavior must add `--open`:
```diff
- infracanvas export report.json
+ infracanvas export report.json --open
```

This intentionally aligns `export` with `scan` and `plan`, which never auto-opened the browser.

## D-01 Tail Confirmation

The parse-error warning loop previously lived BEFORE `resolve_modules()` in `_run_scan`, which meant submodule parse errors (appended into `parsed.parse_errors` DURING `resolve_modules`) were never printed. After this plan:

1. The `if parsed.parse_errors: for path, err in ...` loop is moved to AFTER `resolve_modules(directory, parsed)` in `cli/infracanvas/main.py` line ~96.
2. It prints through `_err_console` (stderr) instead of the stdout console.
3. Display format is `{parent_dir.name}/{path.name}: {err}` for real file paths (so "broken/main.tf" is distinguishable from "vpc/main.tf") and `<sentinel:name>: {err}` passthrough for synthetic sentinels (`<non-local-module:...>`, `<count-cap:...>`, `<missing-module-dir:...>` from Plan 02).

**Test 5.1-H** exercises the full path: `scan envs_layout/envs/prod --quiet` runs with a deliberately broken `modules/broken/main.tf`; asserts stdout has exactly one line (the summary), stderr contains `Could not parse` AND `broken`, and exit code is 0.

## Deviations from Plan

### Rule 1 — parent-dir basename in parse warning

**Found during:** Task 2 (running 5.1-H for the first time)

**Issue:** The plan's Task 1 Edit 6 specified emitting only `path.name` (basename) in the stderr warning — per the T-05.1-04 basename-only mitigation against CWD leaks. But for envs_layout, the broken submodule's parse error has `path.name == "main.tf"`, which does not contain the string "broken". Test 5.1-H asserts `"broken" in result.stderr.lower()`, and the assertion fails because the warning reads "Could not parse main.tf: ..." with no submodule identifier.

**Fix:** Extended the warning emission in `cli/infracanvas/main.py` to use `"{parent_dir_name}/{filename}"` when the path is a real file, passing synthetic sentinels (`<non-local-module:...>`, `<count-cap:...>`, `<missing-module-dir:...>`) through as-is. `path.parent.name` is a single basename component — the user-authored module directory name, not a leak of the absolute project filesystem path. T-05.1-04 mitigation still holds: "no working-directory leak", both formats satisfy.

**Files modified:** `cli/infracanvas/main.py` (warning emission block in `_run_scan`)

**Commit:** `ddc883b`

## Threat Model Compliance

- **T-05.1-07 (EoP / Command Injection via --open):** Mitigated. All three commands use `webbrowser.open(out_path.resolve().as_uri())` — Python stdlib, no subprocess, no shell metacharacter expansion. `Path.resolve().as_uri()` produces RFC-8089 URIs with deterministic escaping. No `subprocess.run(["open", ...])` or platform-specific shell calls.
- **T-05.1-08 (Information Disclosure — stderr parse messages):** Mitigated. Emitted strings use `path.name` + (optionally) `path.parent.name` — two basename components, never absolute paths. `path.parent.name` is the user-authored module directory name (e.g., "broken") from their own Terraform `source = "../../modules/broken"` declaration. No `cwd`, no `/home/alice/...` leakage.
- **T-05.1-09 (DoS — --open during CI):** Mitigated. Test 5.1-G monkeypatches `webbrowser.open`; real CI usage of `--open` is deliberate (user passed an explicit flag). Existing `_should_open_browser()` still gates `config.open_browser` auto-open in headless environments.
- **T-05.1-10 (Tampering — flag combo confusion):** Mitigated. Three orthogonal flags with explicit `typer.Exit(code=2)` guards: `--open` + `--format != html`, `--quiet` + `--json`, `--quiet` + `--ci`. Each guard emits a human-readable stderr error before exiting. Tests 5.1-F and 5.1-I exercise the mutex paths.
- **T-05.1-14 (Tampering — export auto-open behavior change):** Mitigated by documentation. The previous unconditional browser-open on `export … --format html` is replaced with explicit `--open` opt-in. Fails safe (no surprise browser windows). Documented as a breaking change in the Migration Notes section above and must be surfaced in release notes.
