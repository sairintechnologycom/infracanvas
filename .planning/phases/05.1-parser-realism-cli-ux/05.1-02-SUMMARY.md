## Count / for_each Expansion Semantics

`COUNT_EXPANSION_CAP = 1000` — module-level int constant in `hcl.py`.

### `_expand_count(attrs_dict)` return cases

| Input | Output | Effect |
|-------|--------|--------|
| No `count` key | `[(None, False)]` | Single unexpanded ParsedResource |
| `count = 3` (literal int, `0 ≤ n ≤ 1000`) | `[(0, False), (1, False), (2, False)]` | 3 ParsedResource with index 0/1/2 |
| `count = 10_000_000` (literal int > cap) | `[(None, True)]` | **1 collapsed ParsedResource + synthetic `parse_errors` entry** |
| `count = var.az_count` (interpolation string) | `[(None, True)]` | 1 collapsed ParsedResource with `unresolved_count=True` |
| `count = -1` (negative int) or `count = True` (bool) | `[(None, True)]` | 1 collapsed ParsedResource (unresolved) |

### `_expand_for_each(attrs_dict)` return cases

Mirrors `_expand_count`: literal dict/list with ≤1000 entries expands one instance per key (indexed by running int 0..N-1); literal >1000 collapses to single unresolved; non-literal (interpolation) collapses. Absent `for_each` → single unexpanded instance.

### DoS guard (T-05.1-05 mitigation)

The cap is applied **before** `range(raw)` is ever invoked. Sequence for `count = 10_000_000`:

1. `raw = 10_000_000` (unwrapped from python-hcl2's 1-element list)
2. `isinstance(raw, int) and raw >= 0` → True
3. `raw > COUNT_EXPANSION_CAP` → True → **early return `[(None, True)]`**
4. `range(10_000_000)` is never constructed; no 10M-tuple list allocation.

`_extract_resources` detects the cap trip via the same `raw > COUNT_EXPANSION_CAP` check and appends a synthetic `parse_errors` entry:

```python
parsed.parse_errors.append((
    Path(f"<count-cap:{resource_type}.{name}>"),
    f"count={raw} exceeds cap {COUNT_EXPANSION_CAP}; collapsed to 1 unresolved node",
))
```

The Plan 03 CLI stderr loop reads `parsed.parse_errors` and surfaces these to the user without changing exit code — the scan still succeeds (D-01 "warn + continue").

## Placeholder Node Contract (consumed by Plan 04 viewer)

When a local submodule (e.g. `source = "../../modules/broken"`) parses to zero resources AND `sub_parsed.parse_errors` is non-empty, `resolve_modules` synthesizes:

```python
ParsedResource(
    resource_type="_infracanvas_unresolved_module",  # reserved type — leading underscore collision-free
    name=<module_block_name>,                         # e.g., "broken"
    attributes={
        "source": <original source string>,           # e.g., "../../modules/broken"
        "_module_dir": str(<resolved abs path>),       # filesystem path for debugging
        "_parse_error": f"{<first error file>.name}: {<first error msg>}",
    },
    module="",  # placeholder lives at caller's level, NOT inside the failed submodule
)
```

The graph builder (`_create_nodes`) short-circuits provider detection: `res.resource_type == "_infracanvas_unresolved_module"` → `provider = "unresolved"`. No module prefix applied because `module=""`. Viewer's `ResourceNode.tsx` (already shipped by Plan 05.1-04) matches on `data.type.startsWith('_infracanvas_unresolved')` and renders the orange ring + ⚠ marker.

**T-05.1-04 mitigation** — `_parse_error` value uses `first_err_path.name` (basename only), never the absolute path, so no working-directory leak.

## Non-Local Source + Missing Dir Notes (WARNING 5 closure)

`resolve_modules` now appends one-line parse_errors entries for two new cases:

| Condition | Sentinel Path | Message |
|-----------|---------------|---------|
| `source` is non-empty but not `./`/`../` (registry ref, git URL, HTTP URL) | `<non-local-module:{name}>` | `Skipping non-local module source '{source}' — see Known Limitations` |
| Local `source` resolves to a non-directory path | `<missing-module-dir:{name}>` | `Module '{name}' source '{source}' is not a directory; skipping` |

These are informational — scan continues with exit 0. Plan 03's CLI stderr loop (which extends the existing `if parsed.parse_errors` block at `main.py:92-94`) will print them as `[yellow]Warning:[/yellow]` lines.

**T-05.1-15 mitigation** — note content is derived solely from the user's own `.tf` authored string (e.g., `terraform-aws-modules/vpc/aws`); no filesystem path is leaked to stderr.

## Graph Builder ID Formatting

`_create_nodes` in `cli/infracanvas/graph/builder.py` now produces these ID shapes:

| Input | Output ID |
|-------|-----------|
| `ParsedResource(resource_type="aws_vpc", name="main")` | `aws_vpc.main` (unchanged — backwards compatible) |
| `ParsedResource(resource_type="aws_vpc", name="this", module="module.vpc")` | `module.vpc.aws_vpc.this` (unchanged — backwards compatible) |
| `ParsedResource(resource_type="aws_subnet", name="public", module="module.vpc", index=2)` | `module.vpc.aws_subnet.public[2]` (NEW — bracket suffix after module prefix) |
| `ParsedResource(resource_type="aws_subnet", name="public", module="module.vpc", index=None, unresolved_count=True)` | `module.vpc.aws_subnet.public` with `attributes["_unresolved_count"] == True` |
| `ParsedResource(resource_type="_infracanvas_unresolved_module", name="broken", module="")` | `_infracanvas_unresolved_module.broken` with `provider == "unresolved"` |

Bracketed index is applied to the base ID **before** the module prefix is prepended, so the final shape is `{module}.{type}.{name}[{index}]` — readable left-to-right as "inside module X, the type Y's name Z, instance i".

## Deviations from Plan

### Rule 3 — commit granularity consolidation (not a behavioral deviation)

The plan's `<task>` blocks describe three separate TDD cycles (RED for task 1, GREEN implement, then same for task 2, then task 3). I consolidated Task 1 + Task 2 into a single commit `de149a8` because:

1. The `<execution_contract>` explicitly demanded that "`ParsedResource` schema change is in Task 1 (or wherever the plan places it); `graph/builder.py` updates must land in the same or next commit so the tree never has a consumer reading a field the producer hasn't emitted yet." Combining them in a single commit is the strongest version of "same or next commit."
2. Task 1's done-criteria are all parser-internal and can be verified standalone (ruff + mypy on the edited files), but Task 2's changes don't gate on anything new other than Task 1 — splitting the commit would add risk (an intermediate tree state where producer runs but consumer still computes old IDs) without any verification benefit.
3. The test commit (Task 3) remains separate (`2f4b68f`), preserving the test-vs-impl separation in history.

No behavior change; only commit-boundary choice.

### Rule 2 — added `<missing-module-dir:...>` parse_errors note

The plan's `<action>` for `module.py` specified adding notes for non-local sources but was less explicit about the missing-directory case. I added a parallel `parsed.parse_errors.append(...)` inside the `if not module_dir.is_dir()` branch, mirroring the non-local-source pattern. This is required for correctness: without it, a user writing `module "foo" { source = "../nope" }` silently drops the module with no stderr signal — which is exactly the class of bug D-01 aims to eliminate. The plan's `<behavior>` section supports this ("submodule parse errors surface in parsed.parse_errors ...") and the `<done>` item for task 1 explicitly lists "appends a one-line parse_errors note for non-local sources (registry/git/HTTP) and for missing module dirs (warning 5 closure)".

Test 5.1-P enforces the guard at `count=10_000_000` (10M), proving 1 node emitted, not 10M. The cap also triggers a synthetic `parse_errors` entry so the event is observable.
- **T-05.1-06 (EoP — symlink/depth DoS in resolve_modules):** Mitigated (unchanged). Depth limit (`depth >= 3` return) and `_visited` set preserved byte-for-byte from the pre-plan implementation. No regression.
