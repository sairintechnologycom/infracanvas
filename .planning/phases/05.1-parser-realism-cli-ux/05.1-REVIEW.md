# Phase 5.1: Code Review Report

**Reviewed:** 2026-04-21T10:26:18Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

Phase 5.1 introduces three user-facing capabilities: (1) local module resolution with placeholder nodes for broken submodules, (2) literal `count` / `for_each` expansion with a DoS cap, and (3) CLI UX polish (`--quiet` / `--json` / `--open` flag semantics with proper stderr routing). The implementation is generally well-structured — the DoS cap is enforced **before** `range(raw)`, the `webbrowser.open` call correctly uses the stdlib (no shell), the CliRunner uses `mix_stderr=False` to assert the stdout/stderr contract, and placeholder nodes flow through the pipeline as the plan intended.

However, there is a significant latent bug in `graph/builder.py::_build_edges` that pre-dates Phase 5.1 but is now **exposed** by the new module-prefixing and `[index]` suffix logic: edges are sourced by the unprefixed / unbracketed `{type}.{name}` key while the node ID set uses the full prefixed/bracketed form. Every expanded or module-scoped resource therefore loses its outgoing edges in the graph. This is a correctness regression that Phase 5.1 tests do not currently cover (tests assert ParsedResource shapes, not ResourceGraph edges). Additionally, count-expanded instances will emit duplicate edges because the expansion loop re-visits `parsed.implicit_deps` once per expansion.

A handful of smaller issues — display-path handling for temporary Path names, accidental sharing of `attrs_dict` across count expansions, a silent `resource_ids`-driven filter, and minor test/docs inconsistencies — are covered below.

No critical security issues found. `webbrowser.open` uses `file://` URIs on already-validated paths. No shell escapes, no hardcoded secrets, no unsafe deserialization beyond the JSON+Pydantic validation already in place.

## Warnings

### WR-01: Edge source IDs miss module prefix and [index] suffix, silently dropping edges

**File:** `cli/infracanvas/graph/builder.py:85`

**Issue:** `_build_edges` computes edges with `source_id = f"{res.resource_type}.{res.name}"` — no module prefix and no `[index]` suffix. But node IDs constructed at line 31–35 DO include both (`module.vpc.aws_subnet.public[0]` in the new world). The set `resource_ids = {n.id for n in nodes}` (line 82) therefore contains fully-qualified IDs, while the `if dep in resource_ids` and `if target_id in resource_ids` checks compare against un-prefixed strings. The net effect: **every resource with a non-empty `module` or a non-None `index` will emit zero edges**. The Phase 5.1 envs_layout fixture (module.vpc.aws_vpc.this ← module.vpc.aws_subnet.public) is the canonical case that will break. This is latent but newly-exposed by Phase 5.1's prefixing/suffixing logic.

Additionally, because `_build_edges` iterates `parsed.resources` (not `nodes`), count-expanded resources contribute N identical loop iterations, and the `implicit_deps[source_id]` lookup returns the same ref set every time — producing **N copies of every implicit edge** per expanded resource.

**Fix:** Iterate over the already-constructed `nodes` (whose `.id` is fully qualified) instead of `parsed.resources`, and keep a parallel map from `ParsedResource` to `ResourceNode.id` for explicit/implicit dep lookup. Something like:

```python
def _build_edges(
    g: nx.DiGraph,
    parsed: ParsedTerraform,
    nodes: list[ResourceNode],
) -> list[dict[str, str]]:
    edges: list[dict[str, str]] = []
    resource_ids = {n.id for n in nodes}
    # Map node.id -> the ParsedResource it came from so we can look up depends_on
    # and implicit_deps per *node* (not per parsed resource).
    for node, res in zip(nodes, parsed.resources, strict=True):
        source_id = node.id
        base_key = f"{res.resource_type}.{res.name}"
        for dep in res.depends_on:
            if dep in resource_ids:
                edges.append({"source": source_id, "target": dep, "type": "depends_on"})
                g.add_edge(source_id, dep, type="depends_on")
        for target in parsed.implicit_deps.get(base_key, set()):
            # TODO: targets also need module/index resolution for fully-qualified IDs.
            if target in resource_ids:
                edges.append({"source": source_id, "target": target, "type": "implicit"})
                g.add_edge(source_id, target, type="implicit")
    return edges
```

Note: the `target` side also needs fully-qualified resolution — `implicit_deps` is keyed and valued by un-prefixed `{type}.{name}`, but targets in a module will only exist in `resource_ids` under their module-prefixed form. A follow-up task should rewrite `implicit_deps` to carry module-aware targets, or resolve target IDs at edge-build time by scanning `nodes` for `name`/`type` matches within the same module.

Add a test that asserts `len(graph.edges) > 0` for `envs_layout/envs/prod` and for the literal `count = 3` fixture.

### WR-02: Duplicate ParsedResource objects share the same `attrs_dict` reference

**File:** `cli/infracanvas/parser/hcl.py:155-165`

**Issue:** When `count = 3` expands, three `ParsedResource` objects are created at lines 155–165 all pointing to the **same** `attrs_dict` instance. The builder in `_create_nodes` takes a `dict(res.attributes)` shallow copy (line 50) for the non-Azure path, which is fine for the direct branch. But:

1. The Azure branch (`normalize_azure_attrs`, line 44–46) may mutate the dict in-place — if it does, all three expansions observe the mutation.
2. Any downstream consumer that reads `parsed.resources[i].attributes` and mutates will corrupt siblings.
3. The `_unresolved_count` marker is added only to the builder's *copy*, so it is not visible on the shared ParsedResource (good, but fragile — if someone refactors to drop the shallow copy, all siblings will suddenly share a marker).

**Fix:** Shallow-copy `attrs_dict` when creating each ParsedResource so the expansions are independent:

```python
for idx, unresolved in expansions:
    result.resources.append(
        ParsedResource(
            resource_type=resource_type,
            name=name,
            attributes=dict(attrs_dict),  # per-expansion copy
            depends_on=list(depends_on),
            index=idx,
            unresolved_count=unresolved,
        )
    )
```

### WR-03: `display` path construction surfaces project-relative basenames for real files on any OS but silently degrades for sentinel paths placed in `/` tempdirs

**File:** `cli/infracanvas/main.py:112-122`

**Issue:** The logic at lines 117–120 uses `path.parent.name` to disambiguate `broken/main.tf` vs `vpc/main.tf`. When a caller runs `infracanvas scan /tmp/x` or under pytest `tmp_path`, the parent directory name will be the full test-tmpdir basename (e.g. `pytest-of-user`), not a user-recognisable module name — correct behavior, because the real parse error came from a top-level file, not a submodule. That's fine.

However, the check `path.parent.name not in {"", "."}` misses the Windows drive-letter case (`WindowsPath("C:\\").parent.name == ""`) — that is handled — but fails to cover the `Path("foo.tf").parent.name == ""` case where a **relative** Path is used (rare for real files but possible if HCL loader raises). In that case the else-branch prints a bare `name_str` which is also fine. So this is a minor inconsistency, not a bug.

The real fragility: if ANY submodule (not just `broken`) happens to sit at the root of two sibling module directories with identical basenames (`modules/a/main.tf` vs `modules/b/main.tf`), the `display` string is unique because `parent.name` differs — good. But if the parse error comes from a nested `modules/a/sub/main.tf`, `display` becomes `sub/main.tf`, which doesn't identify module `a`. Acceptable for a warning, but the comment claim "users can identify WHICH submodule failed" is slightly stronger than reality.

**Fix:** Either accept the current basenames-only behavior (and soften the comment), or extend the display to walk up to the nearest user-authored module source. Low-priority — informational warning only, and the existing behavior is already better than the pre-5.1 state.

```python
# Update the comment to match reality:
# "Real file — include the parent dir name so the user can usually tell
# which submodule failed (nested submodules show only their immediate parent)."
```

### WR-04: `_run_watch` initial scan can emit duplicate warnings on stderr

**File:** `cli/infracanvas/main.py:533-554`

**Issue:** In `_run_watch`, the initial scan at line 535 calls `_run_scan`, which prints parse warnings to `_err_console`. Then the `on_modified` handler at line 551 calls `_run_scan` again on every change, each of which re-prints the same parse warnings. Over a long watch session with a stable (broken) submodule, the user sees the same warning repeatedly. In `--ci`+`--watch` mode (not a documented combo but not blocked either) this noise will flood logs.

Also, `_run_watch` does not dedupe warnings across re-scans — each scan treats `parse_errors` as fresh.

**Fix:** Optional. If warnings are expected to be persistent for the scan's lifetime (which they are, for broken submodules), consider caching the `parse_errors` signature and only printing new warnings on subsequent scans. At minimum, add a one-line `[dim]` hint on the second scan: "(parse warnings suppressed; run a single scan to re-surface)". For Phase 5.1 this is nice-to-have only.

### WR-05: `_build_edges` misses the `[index]` suffix for expanded resource *targets* in `depends_on`

**File:** `cli/infracanvas/graph/builder.py:87-98`

**Issue:** When a user writes `depends_on = [aws_vpc.main]` (a resource with no count), the HCL parser strips it to `"aws_vpc.main"` in `_normalize_depends_on`. That un-indexed string is then matched against `resource_ids` which, under the new Phase 5.1 ID scheme, contains only bracketed forms like `aws_vpc.main[0]` if `aws_vpc.main` was count-expanded. So `depends_on` pointing at an expanded resource drops silently. This is related to WR-01 but distinct — here the *target* side is the problem, not the source. Same root cause (IDs not normalized at edge time).

**Fix:** At edge-build time, resolve a bare `{type}.{name}` target by scanning `nodes` for any node whose `(resource_type, name)` pair matches and whose module matches the source's module. If multiple match (expanded), emit one edge per expansion (fan-out semantics matching Terraform's behavior). A test fixture exercising `resource "x" "a" { count = 2 }` + `resource "y" "b" { depends_on = [x.a] }` would catch this.

### IN-01: `test_51g_scan_open_invokes_webbrowser` asserts `open` is called but does not exercise the `ok = False` branch

**File:** `cli/tests/test_cli.py:269-297`

**Issue:** The test `test_51g` monkeypatches `webbrowser.open` to return `True`. There is a matching CLI code path at `main.py:477-480` that prints a stderr warning when `webbrowser.open` returns `False`, but no test covers it. Given that `webbrowser.open` returning `False` is an environmental failure (no browser), the miss is acceptable, but adding the negative case would prevent silent regression of the warning text.

**Fix:** Add a sibling test that returns `False` from the monkeypatched `open` and asserts `"webbrowser.open returned False"` appears on stderr and exit code stays 0.

### IN-02: `_build_edges` does nothing with the `g` parameter beyond mirroring edges

**File:** `cli/infracanvas/graph/builder.py:75-100`

**Issue:** `_build_edges` mutates both the edges list and the NetworkX `g` DiGraph. Downstream only `return ResourceGraph(nodes=nodes, edges=edges)` is used — the NetworkX graph `g` is constructed, nodes are added (line 71), edges are added (lines 91/98), and then `g` is discarded. Dead side-effect. Either the `g.add_node` / `g.add_edge` calls should be removed, or the graph should be returned for callers that want NetworkX (several algorithms in other phases might want it).

**Fix:** Remove `g: nx.DiGraph` parameter construction entirely, or expose it as `ResourceGraph.nx_graph` for consumers. Low priority — no correctness impact.

### IN-03: `known_resources` in `_parse_file` / `parse_directory` misses module-prefixed IDs

**File:** `cli/infracanvas/parser/hcl.py:81-91`

**Issue:** The `known_resources` set used for implicit-ref detection at line 81–84 uses `{type}.{name}` — no module prefix. This is consistent with how `find_references` scans string values (it looks for `RESOURCE_REF_PATTERN` which is also un-prefixed). But because `resolve_modules` appends submodule resources *after* `parse_directory` returns, the `implicit_deps` map is built *only* from the top-level module's resources. Submodule cross-references never populate `implicit_deps`. Combined with WR-01, this means edges within or across submodules are doubly-missed: (a) targets aren't in the map, (b) the map is keyed by un-prefixed IDs anyway.

**Fix:** Either rebuild `implicit_deps` after `resolve_modules` (in `main.py`), or move the `implicit_deps` pass out of `parse_directory` and into a dedicated phase that runs once all resources are collected. A follow-up phase should own this.

### IN-04: `_extract_data_sources` uses compound `data_type.name` as the `ParsedBlock.name`

**File:** `cli/infracanvas/parser/hcl.py:213-229`

**Issue:** Line 226 stores the data source as `name=f"{data_type}.{name}"`. This is inconsistent with resource blocks where `name` is just the local identifier. Downstream code that expects `ParsedBlock.name` to be a bare identifier (e.g., for grouping or variable resolution) will see `aws_ami.ubuntu` instead of `ubuntu`. No new code in Phase 5.1 uses this, but it's a latent trap. Not a Phase 5.1 regression, but worth flagging while the file is being reviewed.

**Fix:** Store `block_type` as `"data"`, a new `resource_type=data_type` field, and `name=name` bare. Requires a model change — defer unless it blocks a later phase.

### IN-06: `test_51d_count_nonliteral_collapsed` relies on the submodule flow and would be clearer as a pure `tmp_path` test

**File:** `cli/tests/test_parser.py:229-245`

**Issue:** `test_51d` asserts that `count = var.az_count` collapses to a single unresolved instance, but it does so by going through `resolve_modules` on the `envs_layout` fixture. The parse path is: `envs/prod/main.tf` calls `module "vpc"` → `modules/vpc/main.tf` has `aws_subnet.public { count = var.az_count }`. This couples the test to three moving parts (module resolution, submodule parse, count expansion). If any of those break, the test fails for the wrong reason. A companion `tmp_path`-style test (like `test_51c` uses) would isolate the non-literal-count behavior.

**Fix:** Add a direct sibling test:

```python
def test_51d_count_nonliteral_collapsed_direct(self, tmp_path):
    """Non-literal count (e.g., var ref) collapses regardless of module context."""
    (tmp_path / "main.tf").write_text(
        'resource "aws_subnet" "x" {\n'
        '  count      = var.n\n'
        '  cidr_block = "10.0.0.0/24"\n'
        '}\n'
    )
    parsed = parse_directory(tmp_path)
    xs = [r for r in parsed.resources if r.name == "x"]
    assert len(xs) == 1
    assert xs[0].index is None
    assert xs[0].unresolved_count is True
```

Keep `test_51d` for the module integration guarantee.
