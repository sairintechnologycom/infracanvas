# Phase 07 Plan 11: Home Summary Dashboard + Settings Sub-Routes Summary

Delivered the home `/` summary dashboard (D-04) — score card, score sparkline, top-findings
card, and recent-scans table, all driven by the existing `/v1/scans?limit=10` listing
endpoint — and the `/settings/{members,billing,integrations}` sub-routes (D-03, closing
DSH-05) with a URL-routed amber-400 active-tab strip. Empty state on `/` ships the CLI
install hint card per 07-UI-SPEC.

## What was built

**Task 1 — Home dashboard components (commit `c5e9cc1`)**

- `dashboard/components/home/ScoreCard.tsx` — three-column flex card. Col 1: grade pill
  (28×28 with palette per `gradeInfo()`) + display-sized score number (`text-[28px]
  font-semibold tabular-nums`). Col 2: critical/high/drift counts as 3 vertical pairs,
  severity-token coloured (`text-sev-critical`, `text-sev-high`, `text-sev-medium`).
  Col 3 (`ml-auto`): UTC timestamp, `branch @ a1b2c3d`, and `Open scan →` Next/Link.
  Exports `gradeInfo(score)` utility used by RecentScansTable + tests.
- `dashboard/components/home/ScoreSparkline.tsx` — wraps the handrolled SVG `Sparkline`
  from Plan 07-06 with a title strip ("Score over last 10 scans"). Extracts scores from
  `summary_json.score` (skipping null), takes first 10, then reverses so the newest
  score sits on the right. Falls back to an empty-state copy ("Need at least 2 scans
  to render trend.") when fewer than 2 valid scores exist.
- `dashboard/components/home/TopFindings.tsx` — renders the green well-done pill when
  `summary_json.findings.critical === 0`; otherwise renders a single critical-count
  summary card with rule-id-style chrome (border-l-4 border-sev-critical, bg-red-50/40)
  + an `Open scan →` link to the scan detail. See deviation 1 for why this is a
  degraded version of UI-SPEC §"Top critical findings".
- `dashboard/components/home/RecentScansTable.tsx` — `'use client'` so rows can call
  `router.push(/scans/{id})`. Caps at 5 rows. Columns: Date (relative-or-absolute),
  Branch (sans), Score (grade pill + number), Crit (severity colour or slate-400),
  High (same). Title bar has `View all →` Link to `/scans`.
- `dashboard/__tests__/home-dashboard.test.tsx` — 23 vitest assertions covering grade
  derivation (A+/A/B+/B/C/D/F), score-card rendering, link hrefs, null summary
  resilience, sparkline title + empty fallback, top-findings green pill at zero,
  recent-scans 5-row cap.
- `dashboard/tsconfig.json` — Rule 3 fix: `next/tsconfig` extends doesn't resolve in
  Next 15.5.x, so the equivalent compilerOptions are inlined.

## Deviations from Plan

**1. [Rule 1 — Bug] TopFindings cannot render full Finding objects from /v1/scans**

- **Found during:** Task 1 (planning, before writing the component)
- **Issue:** Plan §`<context><interfaces>` declares `findings: Finding[]` (with rule_id,
  title, resource_id), but `dashboard/lib/types.ts` defines
  `summary_json.findings` as a count map (`{ critical: number; high: number; medium:
  number; info: number }`). The existing `MetadataHeader` and `ScansTable` already use
  the count-map shape — it is the actual API contract. Fetching full Finding objects
  for the home dashboard would require downloading all 10 scans' JSON from R2 (10 ×
  presigned URL fetches per page load), which violates D-17 caching/perf intent.
- **Fix:** TopFindings renders a degraded summary — critical count from the map plus
  an `Open scan →` CTA to the scan detail page where the full Finding list is loaded.
  When count is 0, render the green "well done" pill verbatim per UI-SPEC.
- **Files modified:** `dashboard/components/home/TopFindings.tsx`
- **Commit:** `c5e9cc1`
- **Follow-up suggestion:** Either extend `/v1/scans` to return `top_findings: Finding[]
  | null` per row (~3 entries × 10 rows = small payload), or add a separate
  `/v1/findings/recent?limit=3&team_id=...` endpoint that the home RSC fetches in
  parallel with `/v1/scans?limit=10`.

**2. [Rule 1 — Bug] gradeInfo() thresholds reconciled to UI-SPEC**

- **Found during:** Task 1 (test run)
- **Issue:** Plan §behavior says `score>=85→'A'` and `score>=80→'B+'`. Plan §test cases
  assert `87→'B+'`. These contradict each other (87 ≥ 85 would map to A under the
  first statement). UI-SPEC §"Score-grade pills" + existing `ScansTable.ScoreGradePill`
  + `MetadataHeader` all use 90/80/70/60 brackets where 87 → B+.
- **Fix:** `gradeInfo()` follows UI-SPEC: A+ ≥95, A ≥90, B+ ≥85, B ≥80, C ≥70, D ≥60,
  F <60. (The 95/85 sub-thresholds for A+/B+ are the v1 plan's intent; UI-SPEC's
  "A / A+ ≥ 90" range is preserved by treating 90-94 = A and 95+ = A+.)
- **Files modified:** `dashboard/components/home/ScoreCard.tsx`,
  `dashboard/__tests__/home-dashboard.test.tsx`
- **Commit:** `c5e9cc1`

**3. [Rule 3 — Blocking] tsconfig.json `extends: next/tsconfig` is unresolvable**

- **Found during:** Task 1 RED phase (vitest startup)
- **Issue:** Vitest 4 + Vite tsconfck try to resolve the `extends` field; Next 15.5.15
  does not ship a `tsconfig.json` file inside the `next` package (the tsconfig
  preset is gone in v15). All test runs failed with "Cannot find module
  'next/tsconfig/tsconfig.json'".
- **Fix:** Inlined the equivalent compilerOptions (target ES2020, lib dom/esnext,
  module esnext, moduleResolution bundler, strict, esModuleInterop, isolatedModules,
  jsx preserve, plus the `plugins: [{ name: 'next' }]` entry the next-typescript-plugin
  uses). Removed the broken extends.
- **Files modified:** `dashboard/tsconfig.json`
- **Commit:** `c5e9cc1`
- **Follow-up suggestion:** Phase 7 close-out should review whether other dashboards
  (or future ones) need the same fix; this is a Next 15 upgrade artefact.
