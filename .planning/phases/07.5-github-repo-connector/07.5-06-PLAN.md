<rationale>
The scan_repo taskiq job is the single highest-risk piece of Phase 7.5: it shells out to git clone with a credential, runs a CLI subprocess, writes to /tmp, uploads to R2, and fires Stripe metering. Five separate threat classes (token leakage, path traversal, tmpdir leak, subprocess hang, double-meter) all converge here.

Isolating it as its own plan with dedicated test fixtures (local file:// git remote, real subprocess, fakeredis-backed tests) keeps the test surface focused and the security review contained. Wave 3 because:
- Depends on Plan 03 (mint_installation_token, get_installation_metadata)
- Depends on Plan 05 (finalize_scan helper + scan_repo dispatch contract)
- Depends on Plan 02 (scans columns + GithubInstallation table for retroactive UPDATE on failure)

Plan 11 (frontend polling) does not depend on this — it depends only on Plan 05's pending-row + extended GET response shape.

Note on subprocess style: this plan uses Python stdlib `asyncio.create_subprocess_exec` with an argv list (not shell=True, not exec(string)) — the safe form that prevents shell injection. All argv arguments are derived from validated Pydantic inputs (Plan 03/05 regex/length checks).
</rationale>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Worker process to GitHub | Bearer installation token in clone URL; visible in argv on host (Fly per-app VM tenancy mitigates) |
| Worker process to /tmp | Per-job UUIDv4 subdirectory; rm -rf in finally |
| Subprocess (git, infracanvas) to network | git only talks to api.github.com via clone; infracanvas scan is offline (HCL parsing) |
| Worker process to R2 | boto3 client with team-scoped key prefix |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07.5-06-01 | Information Disclosure | Token leakage in argv (visible to ps) | accept (documented) | Fly per-app VM tenancy isolates ps. RESEARCH landmine 2 + Security Notes call this out. If co-tenancy ever happens, switch to git credential helper. |
| T-07.5-06-02 | Information Disclosure | Token leakage in stderr/structlog | mitigate | _redact_token() applied to ALL stderr decode() outputs before logging. Test test_token_redacted_in_stderr_log + test_token_never_in_log_bind enforce. |
| T-07.5-06-03 | Tampering | Path traversal via user-supplied subpath | mitigate | (tmp_root / path).resolve().relative_to(tmp_root.resolve()) — raises ValueError on escape. Also rejects absolute paths via the same check. Test test_subpath_traversal_rejected + test_subpath_absolute_rejected. |
| T-07.5-06-04 | DoS | Hung git clone / infracanvas scan | mitigate | CLONE_TIMEOUT_S=60s, SCAN_TIMEOUT_S=480s; asyncio.wait_for + proc.kill() on TimeoutError. Test test_clone_timeout + test_scan_timeout. |
| T-07.5-06-05 | DoS | Resource exhaustion via huge repo | mitigate | --depth=1 shallow clone; clone timeout enforces wall-clock cap. RESEARCH Open Q3: blob:limit deferred. |
| T-07.5-06-06 | Tampering | Subprocess RCE via maliciously crafted Terraform | mitigate | infracanvas scan is read-only HCL parsing (Phase 5.1 well-tested); runs as non-root in Fly VM; argv-list subprocess invocation (no shell=True, no eval). |
| T-07.5-06-07 | Tampering | Race condition on status='ready' clobber | mitigate | UPDATE ... WHERE id=:id AND status='pending' guard prevents overwriting a row that already raced to ready. Test test_failed_update_uses_pending_guard. |
| T-07.5-06-08 | Spoofing | scan_repo invoked with team_id from arbitrary kwarg | mitigate | Kicker is the only code path setting team_id, and it's invoked only by the authenticated POST /v1/scans/from-github route after RLS-scoped installation lookup. No cross-team leak path. |
| T-07.5-06-09 | DoS | tmpdir leak on worker SIGKILL | accept | RESEARCH landmine 11 + A5 assumption. Fly worker restarts wipe /tmp; not adding housekeeping cron until observed leakage. |
| T-07.5-06-10 | Repudiation | Stripe meter fired without scan record | mitigate | finalize_scan helper guarantees UPDATE-then-meter in a single tx; if meter raises, the tx rolls back and status stays 'pending'. Test test_finalize_scan_propagates_stripe_error (Plan 05). |
</threat_model>
