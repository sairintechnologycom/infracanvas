## Overview

Phase 7.5 wires GitHub Apps into the InfraCanvas dashboard so authenticated users can install the App, browse the repos that install grants access to, pick a branch + optional subpath, and trigger an on-demand scan. The result lands in Phase 7's existing scan history/detail/share flow. The architecture is fully locked in CONTEXT.md (D-01..D-17); this research fills in library choices, GitHub-API edge cases, and known gotchas.

**Primary recommendation:** Use raw `httpx.AsyncClient` for the 4–5 GitHub API calls we make (no client library), `PyJWT[crypto]` for the App JWT, FastAPI direct for the install-callback redirect, and `tempfile.TemporaryDirectory()` inside an explicit `try/finally` for clone cleanup. This keeps the dependency surface minimal and matches the Phase 6 D-17 async stack already in place.

## GitHub Apps Auth Flow

### Install URL Construction (with state mitigation)

**Critical gotcha confirmed via web search:** GitHub Apps **DO support** the `state` parameter on `/apps/{slug}/installations/new?state=...`. The conflated lore that "GitHub Apps don't support state" applies to a *separate* App "Setup URL" callback flow — that one strips the state param. But the install URL itself preserves it. `[CITED: github.com/orgs/community/discussions/61291]`

**Mitigation strategy:** We do NOT use the App's "Setup URL" feature in any state-bearing way. Instead:

1. App is registered with **only a Callback URL** that points to `https://api.infracanvas.dev/v1/github/install-callback` (per D-10d).
2. App's "Request user authorization (OAuth) during installation" toggle is **OFF** — we don't need a user token, just install confirmation.
3. Frontend constructs the install URL with team_id as state:
   ```ts
   const installUrl = `https://github.com/apps/infracanvas-${env}/installations/new?state=${teamId}`;
   window.open(installUrl, '_blank');
   ```
4. After install, GitHub redirects to **the App's "Post Installation" URL** (which we set to our backend callback). Per the GitHub Community confirmation, the install URL `?state=` IS preserved through to the redirect on the `/installations/new` flow.
5. Backend validates `state == current Clerk session's team_id` (D-14).

**If `state` ever fails to round-trip in production**, fallback: encode team_id in a per-team install URL pattern, or require the user to confirm "you are installing into team {X}" on the post-redirect landing page. (UX fallback only.)

### App JWT signing

```python
# backend/app/integrations/github/auth.py
import time
import jwt  # pyjwt[crypto]
from cryptography.hazmat.primitives.serialization import load_pem_private_key

def mint_app_jwt(app_id: str, private_key_pem: bytes) -> str:
    """5-minute App JWT, RS256. iat -60s clock-skew buffer."""
    private_key = load_pem_private_key(private_key_pem, password=None)
    now = int(time.time())
    payload = {
        "iat": now - 60,         # -60s for clock skew per GitHub docs
        "exp": now + (10 * 60),  # max 10min per docs; we use 5min effective
        "iss": app_id,           # numeric App ID, not slug
    }
    return jwt.encode(payload, private_key, algorithm="RS256")
```
`[CITED: docs.github.com/apps/.../generating-a-json-web-token-jwt]`

### Clone URL embedding

```python
clone_url = f"https://x-access-token:{install_token}@github.com/{repo_full_name}.git"
# Pass GIT_TERMINAL_PROMPT=0 so a missing token never opens an interactive prompt

    clone_url, str(tmpdir),
    env=env,
    stdout=asyncio.subprocess.PIPE,
    stderr=asyncio.subprocess.PIPE,
)
```

**Why URL-embed (not GIT_ASKPASS, not credential helper):** D-07 locked it; URL embedding is the simplest pattern with no global git-config side effects, no temp ASKPASS race, no credential leakage to other processes.

**Caveat (landmine):** The clone URL appears in `git`'s argv (visible to `ps` on the same host). Fly's VM tenancy isolates this per-app. If this becomes a concern, switch to writing `.git/credentials` inside the tmpdir + `git config credential.helper store` after `git init` — but D-07 deferred this complexity.

## Endpoint Implementations

### Install callback — `GET /v1/github/install-callback`

**Triggered by:** GitHub redirects user's browser here after install completes. URL shape:
`/v1/github/install-callback?installation_id=12345&setup_action=install&state={team_id}`

**Note on HTTP method:** GitHub uses GET (browser navigation). The CONTEXT D-10d "POST" label is likely a typo — see Open Question 5. Use GET.

**FastAPI handler:**
```python
@router.get("/v1/github/install-callback")
async def install_callback(
    installation_id: int,
    setup_action: str,  # 'install' | 'update'
    state: str,         # team_id we encoded in the install URL
    request: Request,
    user: ClerkUser = Depends(get_clerk_user),  # 401 if unauthenticated
    db: AsyncSession = Depends(get_db),
):
    # 1. CSRF: state must match active team for the Clerk session
    if state != str(user.active_team_id):
        raise HTTPException(403, "state mismatch — install aborted")

    # 2. Fetch installation metadata using a fresh App JWT
    async with httpx.AsyncClient() as client:
        jwt_token = mint_app_jwt(settings.github_app_id, settings.github_app_private_key)
        r = await client.get(
            f"https://api.github.com/app/installations/{installation_id}",
            headers={"Authorization": f"Bearer {jwt_token}", "Accept": "application/vnd.github+json"},
            timeout=10.0,
        )
        r.raise_for_status()
        install = r.json()

    # 3. Upsert (RLS scopes by app.current_team_id from middleware)
    await db.execute(text("""
        INSERT INTO github_installations
            (team_id, github_installation_id, github_account_login, github_account_type, installed_by_user_id)
        VALUES (:team_id, :iid, :login, :type, :uid)
        ON CONFLICT (team_id, github_installation_id) DO UPDATE
        SET github_account_login = EXCLUDED.github_account_login,
            github_account_type = EXCLUDED.github_account_type
    """), {
        "team_id": user.active_team_id, "iid": installation_id,
        "login": install["account"]["login"], "type": install["account"]["type"],
        "uid": user.clerk_user_id,
    })
    await db.commit()

    # 4. Redirect back to dashboard
    return RedirectResponse(url=f"{settings.dashboard_url}/settings/integrations?install=success", status_code=302)
```

**Why FastAPI direct (not Next.js proxy):** GitHub's redirect target is just a URL — there's no benefit to a Next.js intermediary. The user's Clerk session cookie is on the apex domain (per Phase 6 D-03 + Phase 7 D-17), so the cross-subdomain redirect carries auth.

**Failure modes:**
- User not authenticated → `Depends(get_clerk_user)` raises 401 → render a "sign in to complete install" page.
- `state` mismatch → 403, log + Sentry-tag with `installation_id` for forensics.
- GitHub `GET /app/installations/{id}` returns 404 → install was revoked between redirect and our call. Show "install failed, try again."

### `GET /v1/github/repos?installation_id=X&q=foo`

**Cache key:** `gh:repos:{installation_id}:{q or '*'}` — TTL 60s.

**Two query modes:**
- **No `q`:** `GET https://api.github.com/installation/repositories?per_page=30&sort=pushed` (auth: installation token). Returns up to 30 most recently pushed repos from the install grant.
- **With `q`:** Search-style filter. NOTE: `/search/repositories` requires a USER token, not installation token. **Workaround:** call `/installation/repositories?per_page=100`, then substring-match on `full_name` server-side. With 100-repo cap, this is fine for MVP.

```python
INSTALL_REPO_LIST_TTL = 60  # seconds

async def list_repos(installation_id, q):
    cache_key = f"gh:repos:{installation_id}:{q or '*'}"
    cached = await redis.get(cache_key)
    if cached:
        return json.loads(cached)

    token = await get_installation_token(installation_id)
    async with httpx.AsyncClient() as client:
        r = await client.get(
            "https://api.github.com/installation/repositories",
            headers={"Authorization": f"Bearer {token}", "Accept": "application/vnd.github+json"},
            params={"per_page": 100, "sort": "pushed", "direction": "desc"},
            timeout=10.0,
        )
        r.raise_for_status()
        repos = r.json()["repositories"]

    if q:
        ql = q.lower()
        repos = [r for r in repos if ql in r["full_name"].lower()]

    out = [{"full_name": r["full_name"], "default_branch": r["default_branch"], "private": r["private"]} for r in repos[:30]]
    await redis.setex(cache_key, INSTALL_REPO_LIST_TTL, json.dumps(out))
    return out
```

**Rate-limit:** `/installation/repositories` is 5000/hr per installation. Even a 250ms-debounced typeahead with 60s cache rarely hits this. Surface `X-RateLimit-Remaining` in 429/403 with a friendly "GitHub rate-limited; try again in a minute" message.

### `POST /v1/scans/from-github`

```python
class ScanFromGitHubRequest(BaseModel):
    installation_id: int
    repo: str  # "org/name"
    branch: str
    path: str = "."

@router.post("/v1/scans/from-github")
async def scan_from_github(body, user, db, request):
    # 1. Verify installation belongs to this team (RLS does this implicitly, but be explicit)
    install = await db.execute(text(
        "SELECT 1 FROM github_installations WHERE github_installation_id = :iid"
    ), {"iid": body.installation_id})
    if install.scalar() is None:
        raise HTTPException(404, "installation not found for this team")

    # 2. Resolve HEAD sha (one GitHub API call — also catches "branch doesn't exist")
    token = await get_installation_token(body.installation_id)
    owner, name = body.repo.split("/", 1)
    async with httpx.AsyncClient() as client:
        r = await client.get(
            f"https://api.github.com/repos/{owner}/{name}/git/ref/heads/{body.branch}",
            headers={"Authorization": f"Bearer {token}", "Accept": "application/vnd.github+json"},
            timeout=10.0,
        )
        if r.status_code == 404:
            raise HTTPException(404, f"branch {body.branch} not found in {body.repo}")
        r.raise_for_status()
        sha = r.json()["object"]["sha"]

    # 3. Create scans row in 'pending' status
    scan_id = uuid7()
    await db.execute(text("""
        INSERT INTO scans (id, team_id, status, source, source_path,
                           github_installation_id, github_repo, github_branch, github_sha,
                           created_by_user_id)
        VALUES (:id, :team_id, 'pending', 'github', :path,
                :iid, :repo, :branch, :sha, :uid)
    """), {
        "id": scan_id, "team_id": user.active_team_id, "path": body.path,
        "iid": body.installation_id, "repo": body.repo, "branch": body.branch,
        "sha": sha, "uid": user.clerk_user_id,
    })
    await db.commit()

    # 4. Enqueue taskiq job
    await scan_repo.kiq(
        scan_id=str(scan_id),
        installation_id=body.installation_id,
        repo=body.repo, branch=body.branch, sha=sha, path=body.path,
        team_id=str(user.active_team_id),
        request_id=request.headers.get("X-Request-ID"),
    )

    return {"scan_id": str(scan_id)}
```

**Note on `scans` columns:** D-12 only adds `source_path`. The `github_repo`, `github_branch`, `github_sha`, `github_installation_id` columns are NOT in CONTEXT.md but are needed to render "scanning {repo}@{branch}" in the polling UI and to support D-08's "skip clone if HEAD sha matches existing scan within 60s." ## Taskiq Job

```python
# backend/app/jobs/scan_repo.py
import asyncio, json, os, shutil, uuid
from pathlib import Path
import httpx
import structlog

from app.core.taskiq import broker

log = structlog.get_logger()

CLONE_TIMEOUT_S = 60       # 1 min for shallow clone
SCAN_TIMEOUT_S = 8 * 60    # 8 min wall-clock for `infracanvas scan`

@broker.task(task_name="scan_repo")
async def scan_repo(
    scan_id: str, installation_id: int,
    repo: str, branch: str, sha: str, path: str,
    team_id: str, request_id: str | None = None,
) -> None:
    """Mint token → shallow clone → run scan subprocess → upload to R2 → commit handler.

    On any failure: flip scans.status='failed' + write scans.error_message.
    """
    log_ctx = log.bind(scan_id=scan_id, team_id=team_id, repo=repo, branch=branch,
                       sha=sha, installation_id=installation_id, request_id=request_id)
    log_ctx.info("scan_repo.start")

    tmp_root = Path("/tmp") / f"scan-{uuid.uuid4()}"

    try:
        # 1. Mint fresh installation token
        async with httpx.AsyncClient(timeout=10.0) as client:
            jwt_token = mint_app_jwt(settings.github_app_id, settings.github_app_private_key)
            token_resp = await mint_installation_token(client, jwt_token, installation_id)
        token = token_resp["token"]

        tmp_root.mkdir(parents=True, exist_ok=False)
        clone_url = f"https://x-access-token:{token}@github.com/{repo}.git"

        # 2. Shallow clone (timeboxed)
        clone_proc = await asyncio.create_subprocess_exec(
            "git", "clone", "--depth=1", "--single-branch", "--branch", branch,
            clone_url, str(tmp_root),
            env={**os.environ, "GIT_TERMINAL_PROMPT": "0"},
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.PIPE,
        )
        try:
            stdout, stderr = await asyncio.wait_for(clone_proc.communicate(), timeout=CLONE_TIMEOUT_S)
        except asyncio.TimeoutError:
            clone_proc.kill()
            await clone_proc.wait()
            raise RuntimeError(f"git clone timed out after {CLONE_TIMEOUT_S}s")

        if clone_proc.returncode != 0:
            # NEVER log stderr verbatim — it may contain the token in URL
            sanitized = stderr.decode().replace(token, "***")
            raise RuntimeError(f"git clone failed (rc={clone_proc.returncode}): {sanitized[:500]}")

        # 3. Subpath validation (path traversal guard)
        scan_dir = (tmp_root / path).resolve()
        if not str(scan_dir).startswith(str(tmp_root.resolve())):
            raise ValueError(f"path traversal detected: {path}")
        if not scan_dir.is_dir():
            raise ValueError(f"subpath {path} not found in {repo}@{branch}")

        # 4. Run `infracanvas scan` (timeboxed)
        scan_proc = await asyncio.create_subprocess_exec(
            "infracanvas", "scan", ".", "--ci", "--output", "scan.json",
            cwd=str(scan_dir),
            env={**os.environ, "PYTHONUNBUFFERED": "1"},
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.PIPE,
        )
        try:
            stdout, stderr = await asyncio.wait_for(scan_proc.communicate(), timeout=SCAN_TIMEOUT_S)
        except asyncio.TimeoutError:
            scan_proc.kill()
            await scan_proc.wait()
            raise RuntimeError(f"infracanvas scan timed out after {SCAN_TIMEOUT_S}s")

        # rc 0 = no findings, 1 = findings present (still success), 2 = error
        if scan_proc.returncode not in (0, 1):
            raise RuntimeError(f"scan failed (rc={scan_proc.returncode}): {stderr.decode()[:500]}")

        scan_json_path = scan_dir / "scan.json"
        scan_payload = json.loads(scan_json_path.read_text())

        # 5. Upload to R2 (Phase 6 D-06..D-09 path)
        await upload_scan_json(team_id=team_id, scan_id=scan_id, payload=scan_payload)

        # 6. Commit (Phase 6 D-09 sync handler — fires Stripe meter, flips status='ready')
        await commit_scan_internal(scan_id=scan_id, team_id=team_id)

        log_ctx.info("scan_repo.success")

    except Exception as exc:
        log_ctx.error("scan_repo.failed", error=str(exc))
        async with get_db() as db:
            await db.execute(text("""
                UPDATE scans SET status='failed', error_message = :msg
                WHERE id = :id AND status = 'pending'
            """), {"id": scan_id, "msg": str(exc)[:500]})
            await db.commit()
        raise

    finally:
        # Belt-and-suspenders cleanup
        if tmp_root.exists():
            shutil.rmtree(tmp_root, ignore_errors=True)
```

**Per-task timeout enforcement:** Taskiq does not natively kill long-running coroutines on timeout. We enforce timeouts at the subprocess level via `asyncio.wait_for`. Total job budget = `CLONE_TIMEOUT_S + SCAN_TIMEOUT_S` ≈ 9 min. As an outer guard, taskiq's `--ack-time` (per-broker) can be set so an unresponsive worker re-delivers, but since we kill subprocesses individually, this is rarely needed.

**Connection pools:** Per Phase 6 D-13 + D-17, the asyncpg pool and httpx client are constructed once at worker startup (taskiq lifespan hook) and reused across job invocations. Do NOT construct a new asyncpg pool per job — it leaks connections to Neon's connection cap. Use taskiq's `TaskiqEvents.WORKER_STARTUP` to wire pool creation.

**Sentry + structlog:** Bind context at job entry (already shown). Sentry tags: `team_id`, `installation_id`, `repo`, `branch`, `sha`, `request_id`. Use `sentry_sdk.set_tag(...)` early in the job body.

## Security Notes

- **Token blast radius:** D-06 mint-per-scan means a token is alive for ~1hr at most. Never logged (we sanitize stderr by replacing the token with `***` before any log emit). Never persisted.
- **Private key storage:** Fly secret `GITHUB_APP_PRIVATE_KEY` (PEM blob). Loaded once at worker boot via `cryptography.hazmat.primitives.serialization.load_pem_private_key`. Per-env (dev + prod) so a dev leak doesn't compromise prod.
- **Setting multi-line PEM in Fly:** the documented way is `flyctl secrets set GITHUB_APP_PRIVATE_KEY="$(cat private-key.pem)"` — quotes preserve newlines. Avoid manual escaping. `[CITED: community.fly.io discussion on PEM secrets]`
  ```
  fly secrets set GITHUB_APP_PRIVATE_KEY="$(cat infracanvas-dev.private-key.pem)" --app infracanvas-api-dev
  fly secrets set GITHUB_APP_ID="123456" --app infracanvas-api-dev
  fly secrets set GITHUB_APP_WEBHOOK_SECRET="$(openssl rand -hex 32)" --app infracanvas-api-dev
  ```
- **Install-CSRF:** D-14 `state == team_id` check. Belt-and-suspenders: also verify `installation_id` is reachable via fresh App JWT before persisting (catches forged callback URLs).
- **Sandbox:** Worker runs as non-root in Fly VM. `/tmp` is private to the VM. `try/finally` + `shutil.rmtree(ignore_errors=True)` for cleanup.
- **Path traversal guard:** D-09 subpath input is user-controlled. Resolve against the temp-root and verify the resolved path stays inside it (shown in the taskiq job code above).
- **Token-in-argv leakage:** acceptable on Fly's per-app VM tenancy. If we ever co-tenant, switch to git credential helper inside the tmpdir.
- **Webhook secret is provisioned now (D-15) so Phase 8 can use it without re-registering the App.** No webhook handler in this phase.

## Landmines & Gotchas

1. **`state` param IS preserved on install URL** but NOT on the App's "Setup URL" callback when manually invoked. Use only the install URL for state-bearing flows. Don't be confused by stale GitHub community threads. `[CITED: github.com/orgs/community/discussions/61291]`
2. **Token in `git clone` argv** is visible to `ps` on the same host. Acceptable on Fly's per-app VM tenancy. Mark this in security notes as a known posture; don't paper over it.
3. **GitHub App private key is multi-line PEM**; `flyctl secrets set` from a file via `"$(cat key.pem)"` is the only reliable pattern. Direct paste corrupts newlines.
4. **`pip install pyjwt`** does NOT install RS256 support — you must use `pyjwt[crypto]` to pull `cryptography`. Easy miss; PR-block on a unit test that signs with RS256.
5. **Installation token expires in ~1 hour, but worker scans should fit well under that.** Don't cache tokens across jobs unless scans/min crosses 10. Token TTL is also why we mint per-scan, not per-worker.
6. **`git clone --depth=1 --branch=X`** works on branch names, lightweight tags, and annotated tags — but if `branch` is actually a SHA, GitHub rejects with `Remote branch X not found`. Validate `branch` against `/branches` list before enqueue (already done implicitly by `GET /repos/{r}/git/ref/heads/{b}` in `POST /v1/scans/from-github`).
7. **Branch names with slashes** (e.g., `feature/foo`) are valid for git but require URL encoding when passed to `/repos/{r}/git/ref/heads/{branch}`. Use `httpx`'s `params=` so it encodes; or `urllib.parse.quote(branch, safe='/')`.
8. **GitHub `Link` header pagination on `/installation/repositories`** uses comma-separated rels. For MVP we don't paginate (per_page=100 is enough). Document that >100 repos is a known limitation; defer to follow-up.
9. **`/search/repositories` requires a USER token, not installation token** — that's why D-10b uses substring filtering on `/installation/repositories` results. Don't be tempted to reach for `/search`.
10. **taskiq does not natively kill long-running coroutines on timeout.** We must enforce timeouts at the subprocess level via `asyncio.wait_for`. A pure-Python infinite loop inside the job is NOT killable — but `infracanvas scan` is a subprocess, so this is moot here.
11. **Worker SIGKILL leaks tmpdir.** `try/finally` doesn't run. Add a periodic housekeeping task (or rely on Fly VM restarts wiping `/tmp`).
12. **Race on double Scan click:** D-08 explicitly accepts no dedup. Two jobs enqueue, two scan rows, two R2 uploads, two Stripe meter pings. Acceptable failure mode per the user's locked decision. **Do not add server-side dedup in this phase.**
13. **`infracanvas scan --ci` exit codes:** 0 = no findings, 1 = findings present, 2 = error. The worker MUST treat `rc in (0, 1)` as success, only `rc == 2` as failure. Easy bug source.
14. **Subpath edge case:** if `path` is empty string, the resolved path equals tmp_root — fine. If `path` is `..`, the path-traversal guard catches it. If `path` is an absolute path like `/etc`, the resolved path won't start with tmp_root — guard catches it. Test all three.
15. **Webhook secret is provisioned now (D-15) but unused.** Don't accidentally wire a webhook handler — that's Phase 8 scope. The secret sits in Fly only.
16. **Cleanup of `installations/repositories` when an install is uninstalled** (admin removes it from GitHub) — out of scope for Phase 7.5. Phase 8 webhook handler will receive `installation.deleted` and clean up. For now, calls to a deleted installation will just 404, which is acceptable.
17. **Clerk session must be active when GitHub redirects to install-callback.** If the user took >30min to install (Clerk session expired), the callback 401s. UX: render a friendly "sign in to complete installation" page that re-runs callback after auth. Defer this UX polish unless smoke testing shows it's a real issue.
18. **Don't log the clone URL anywhere** — it contains the token. The taskiq job sample explicitly sanitizes stderr; ensure structlog binds don't include `clone_url`.
19. **Repo full_name in URL paths** — must be split into owner/name. Don't pass `org/name` directly into a URL where path traversal is possible (none of ours are, but be careful).
20. **Worker must have `infracanvas` CLI installed at the same version as the API expects.** Pin in the worker Dockerfile. Schema drift between CLI scan output and Phase 6's Pydantic validator will surface as `commit` rejections. Wave 0 should add a smoke test that `infracanvas scan --version` is callable.

## Open Questions (RESOLVED)

1. **`scans` table extensions vs side table for GitHub metadata**
   - What we know: D-12 only locks `source_path`. The polling UI needs `repo + branch + sha`, and D-08's "skip clone if HEAD sha matches" needs a quick lookup on `(team_id, repo, sha)`.
   - What's unclear: whether to put `github_installation_id, github_repo, github_branch, github_sha` directly on `scans` (denormalized; simpler query path) or in a `scan_github_metadata(scan_id, ...)` side table (cleaner but a join on every fetch).
   - Recommendation: put them on `scans` directly. Phase 8 webhook scans will use the same shape. Cost is 4 nullable columns. Confirm with planner.
   - **RESOLVED:** denormalized columns on `scans` (Plan 02 — see migration 008_scan_github_columns adds `github_installation_id, github_repo, github_branch, github_sha, source_path, error_message` directly to `scans`).

2. **Polling endpoint shape — does `GET /v1/scans/{id}` already return `status` and `error_message`?**
   - What we know: Phase 6/7 created `GET /v1/scans/{id}` for the detail page; Phase 7 expects `ready` status.
   - What's unclear: whether the response shape currently includes `status`, `error_message`, and the GitHub metadata columns.
   - Recommendation: planner adds these to the response model in Phase 7.5 if missing. Backward compatible (new optional fields).
   - **RESOLVED:** response model extended in Plan 05 (ScanGetResp adds `status`, `error_message`, `source_path`, `github_installation_id`, `github_repo`, `github_branch`, `github_sha`; relaxes `presigned_get_url` to `str | None`). Backward compatible — all new fields optional/nullable.

3. **Should clone size be capped via `--filter=blob:limit=10m`?**
   - What we know: shallow clone limits depth, not blob size. A repo with a 500MB binary in HEAD will still pull it.
   - What's unclear: how often InfraCanvas users have huge non-Terraform binaries in their repos.
   - Recommendation: defer until smoke testing surfaces it. The 60s clone timeout is the natural cap.
   - **RESOLVED:** deferred — accept default; revisit if smoke surfaces large-repo issues. Plan 06 implements only `--depth=1 --single-branch` per CONTEXT D-08; the 60s clone timeout is the natural cap.

4. **Per-team Fly worker concurrency vs single-process model**
   - What we know: Phase 6 D-12 uses `[processes] worker` block; default is one worker process.
   - What's unclear: how many concurrent `scan_repo` jobs the worker can handle. With 3 concurrent users clicking Scan, do we serialize?
   - Recommendation: taskiq workers process jobs concurrently within the asyncio event loop. CPU-heavy step is the `infracanvas scan` subprocess, which is ~10-30s. Under low load this is fine. Add Fly machine scaling triggers later.
   - **RESOLVED:** deferred — Fly machine count tuned post-launch. Phase 7.5 ships with the existing single `[processes] worker` block from Phase 6 D-12; concurrent jobs handled within the asyncio event loop. Scaling triggers added if smoke or production reveals queueing.

5. **Should `POST /v1/github/install-callback` be `GET`?**
   - What we know: GitHub redirects via HTTP GET (browser navigation).
   - What's unclear: D-10d says POST. If GitHub uses GET, the route MUST be GET.
   - Recommendation: GitHub uses GET for the install redirect. The CONTEXT D-10d "POST" label is a typo. Planner should use GET (this RESEARCH.md uses GET in code samples).
   - **RESOLVED:** GET (browser redirect target); CONTEXT D-10d "POST" is a typo — Plan 04 implements `GET /v1/github/install-callback` as the redirect target.
