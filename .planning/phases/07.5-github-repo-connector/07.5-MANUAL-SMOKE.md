# Phase 7.5 — Manual Smoke Test Checklist

> Walk this end-to-end once before phase close. Pairs with
> `07.5-PHASE-VERIFICATION.md` Section 4 sign-off.
>
> The point of this checklist is to exercise everything that **cannot** be
> covered by automated tests because it requires a real GitHub App
> installation, a real `git clone` against `github.com`, and a real Stripe
> meter event that is observable in the Stripe test dashboard.
>
> The pre-flight (one-time) section is required only if you have not
> previously registered the InfraCanvas DEV App against this dev environment.
> Skip straight to the smoke section if it is already registered and the Fly
> secrets are already populated.

---

## Pre-Flight (one-time, per dev env)

> Goal: a registered GitHub App, a private key in Fly secrets, and a slug
> wired into the Vercel dashboard env. After this is done once, you never
> need to do it again for this dev env.

**Step P1 — Register the InfraCanvas DEV GitHub App**

Open <https://github.com/settings/apps/new>. Fill in:

| Field | Value |
|-------|-------|
| GitHub App name | `InfraCanvas Dev` (must be globally unique — append a suffix if taken) |
| Homepage URL | `https://dashboard-dev.infracanvas.dev` (or your dev dashboard URL) |
| Callback URL | `https://api-dev.infracanvas.dev/v1/github/install-callback` (your backend URL — **must match exactly**) |
| Setup URL | _leave blank_ (we use the Callback URL only — see RESEARCH §GitHub Apps Auth Flow) |
| Webhook → Active | ☑ checked |
| Webhook URL | `https://api-dev.infracanvas.dev/v1/github/webhooks` (Phase 8 will handle these; configure now so we don't need to re-register later) |
| Webhook Secret | run `openssl rand -hex 32` and paste — copy the value to your clipboard for step P3 |
| Repository permissions → Contents | Read-only |
| Repository permissions → Metadata | Read-only |
| Account permissions | _none_ (leave all "No access") |
| Subscribe to events → Push | ☑ checked (Phase 8 wires the handler) |
| Where can this GitHub App be installed? | Any account |

Click **Create GitHub App**.

- [ ] **Expected:** Redirected to the App settings page. The slug appears in the URL: `https://github.com/settings/apps/{slug}`. Copy the slug — you'll need it in step P4.
- [ ] **Expected:** The App ID (a 6–7 digit number) is visible at the top of the General tab. Copy it.

**Step P2 — Generate the App's RSA private key**

On the App settings General tab, scroll to "Private keys" and click **Generate a private key**. A `.pem` file downloads automatically.

- [ ] **Expected:** A file named `infracanvas-dev.YYYY-MM-DD.private-key.pem` is in your Downloads. **Move it to a path you'll reference in step P3** — e.g. `~/secrets/infracanvas-dev-app.pem`. Do **not** check this file into git.

**Step P3 — Push the secrets to Fly**

```bash
flyctl secrets set GITHUB_APP_ID="<numeric app id from P1>" --app infracanvas-api-dev
flyctl secrets set GITHUB_APP_PRIVATE_KEY="$(cat ~/secrets/infracanvas-dev-app.pem)" --app infracanvas-api-dev
flyctl secrets set GITHUB_APP_WEBHOOK_SECRET="<hex from P1>" --app infracanvas-api-dev
flyctl secrets set GITHUB_APP_SLUG="<slug from P1>" --app infracanvas-api-dev
```

- [ ] **Expected:** Each command prints `Secrets are staged for the next deployment` (or triggers a redeploy). The PRIVATE_KEY one **must** use the `"$(cat ...)"` form so newlines inside the PEM are preserved (T-07.5-01-PEM is the failure mode if you skip the quoting). If JWT signing later fails with "Could not deserialize key data", come back here.
- [ ] **Expected:** `flyctl secrets list --app infracanvas-api-dev` shows all four `GITHUB_APP_*` secrets present.

**Step P4 — Wire the slug into Vercel**

In Vercel → infracanvas-dashboard-dev project → Settings → Environment Variables:

```
NEXT_PUBLIC_GITHUB_APP_SLUG = <slug from P1>
```

Apply to: Preview + Production (or whichever env you're smoking). Trigger a redeploy or wait for the next push.

- [ ] **Expected:** New dashboard build picks up `NEXT_PUBLIC_GITHUB_APP_SLUG`. You can verify by viewing the page source on `/settings/integrations` — the install URL embedded in the InstallButton should read `https://github.com/apps/{slug}/installations/new?state=...`.

**Step P5 — Pick a sandbox fixture repo**

Choose a small Terraform repo you control (or fork one) — e.g. `infracanvas-fixtures/terraform-sample`, `your-handle/tf-aws-vpc-sample`, anything with at least one `*.tf` file at the path you intend to scan. The InfraCanvas GitHub App will be granted access to **only this repo** in step S5; do **not** grant whole-org access for the smoke.

- [ ] **Expected:** Repo is reachable in your browser at `https://github.com/<owner>/<name>` and you can see at least one `*.tf` file (default branch `main` recommended).

## Smoke Test (run end-to-end after pre-flight)

> Time-box: 15–30 minutes. If a step fails, **stop**, capture the error, and
> see the troubleshooting table below.

**Step S1 — Sign in to the dashboard**

Navigate to `https://dashboard-dev.infracanvas.dev/sign-in`. Sign in with a Clerk identity that belongs to the org you set up Stripe for in Phase 6.

- [ ] **Expected:** Redirected to `/scans` (or the dashboard root). No console errors. Clerk org switcher shows your org name top-right.

**Step S2 — Navigate to /settings/integrations**

Click Settings (or visit `/settings/integrations` directly).

- [ ] **Expected:** "GitHub" heading visible.
- [ ] **Expected:** **"Install InfraCanvas GitHub App"** button visible (no installations row underneath — the empty/preinstall state from Plan 09's 3-state machine).
- [ ] **Expected:** Slack alerts stub block also visible (preserved from Plan 07; not wired in Phase 7.5 — Phase 8 will).
- [ ] **Expected:** Browser DevTools Network tab shows `GET /api/github/installations` returned `200 []`.

**Step S3 — Click Install**

Click the install button.

- [ ] **Expected:** A new browser tab opens at `https://github.com/apps/{slug}/installations/new?state={your-clerk-org-id}` (verify the `state` query param matches the Clerk org id you're signed into — visible in your Clerk dashboard or in the page source).

**Step S4 — Pick "Only select repositories"**

In the GitHub install screen, select **Only select repositories** (NOT "All repositories"). In the repo picker, choose the fixture repo you set up in P5.

- [ ] **Expected:** Permission summary shows **Contents (read)** and **Metadata (read)** only — no surprises. If you see Pull requests / Issues / Workflows / etc, the App's permission set drifted from P1 — **stop and re-check**.

Click **Install**.

**Step S5 — Land back on dashboard with installation hydrated**

GitHub redirects you (via `/v1/github/install-callback`) to `/settings/integrations?install=success`. The page mounts. The installation row appears within 0–3 seconds (Plan 09's post-install hydration poll runs at 3 s intervals, capped at 5 attempts).

- [ ] **Expected:** Browser URL is `…/settings/integrations?install=success`.
- [ ] **Expected:** Within ~3 s, an installation row renders with: GitHub account login (your handle or org), an `installed_at` date, and a "Manage on GitHub" link to `https://github.com/settings/installations/{id}`.
- [ ] **Expected:** Below the installation row, a **ScanTriggerForm** renders inline with the RepoCombobox + BranchPicker + path input + Scan button.
- [ ] **Expected:** No console errors; Network tab shows `/api/github/installations` polling gracefully transitions from `[]` → `[<your installation>]`.

**Step S6 — Pick the fixture repo from the RepoCombobox**

Click the RepoCombobox. Type the first few letters of your fixture repo name.

- [ ] **Expected:** Typeahead is **debounced** — the network request fires ~250 ms **after** you stop typing, not per-keystroke. (Open DevTools Network tab and watch `/api/github/repos?installation_id=...&q=...` calls.)
- [ ] **Expected:** Your fixture repo appears in the dropdown. Private repos show a lock icon (lucide `Lock` with `aria-label="private"`).

Pick it.

**Step S7 — Pick a branch**

The BranchPicker auto-fetches branches for the selected repo and **defaults to the repo's default branch** (e.g. `main`).

- [ ] **Expected:** Branch picker shows the repo's branches; the default branch is pre-selected. (Verify against `https://github.com/<owner>/<repo>/branches`.)

**Step S8 — Set the path**

Leave the path as `.` (the repo root) **or** set it to a subdirectory you know contains `*.tf` files (e.g. `terraform/aws/vpc`). Do **not** set it to `..` or an absolute path — those will be rejected by the Plan 06 traversal guard (you'll exercise that case in S17).

**Step S9 — Click Scan**

- [ ] **Expected:** Browser navigates immediately to `/scans/{scan_id}` (URL contains a UUID).
- [ ] **Expected:** Page shows a skeleton + a copy line like "Scanning {repo}@{branch}…" rendered by `ScanPendingClient`.
- [ ] **Expected:** Network tab shows `/api/scan-status?id={scan_id}` polling every **2 seconds** (CC-14 setInterval(2000) per PATTERNS A1).

**Step S10 — Wait for the scan to finish**

Wait 10–60 seconds (clone + scan time depends on repo size; Plan 06 caps clone at 60 s and scan at 480 s).

- [ ] **Expected:** When the worker finishes, the polling fetch returns `status: 'ready'` with a `presigned_get_url`. The page transitions automatically — `router.refresh()` re-renders the parent RSC and the `ScanViewerClient` (Phase 7 D-08 path) mounts in place.
- [ ] **Expected:** The viewer renders the scan output (canvas / scorecard / findings — whichever default tab is active).
- [ ] **Expected:** Network tab confirms polling **stops** once status flipped (no more `/api/scan-status` calls after the transition — CC-14 cancelled-flag teardown).

**Step S11 — Verify the Stripe meter event**

Open <https://dashboard.stripe.com/test/customers> (Stripe **test** dashboard — make sure the toggle top-right says "Viewing test data"). Find the customer matching your team's `stripe_customer_id`. Click into Usage → confirm one new metered usage event with timestamp matching the scan you just ran.

- [ ] **Expected:** A meter event with the configured meter event name (per Phase 6 D-08; e.g. `infracanvas_scan_uploaded`) timestamped within the last few minutes.

**Step S12 — Verify the R2 object**

Open Cloudflare → R2 → infracanvas-scans bucket → browse to `teams/{team_id}/scans/{scan_id}.json`.

- [ ] **Expected:** Object exists. Size is non-zero (typically a few KB to a few hundred KB depending on resource count). Click → preview → JSON parses.

## Negative Path (failure handling)

**Step S13 — Trigger a scan with a nonexistent path**

Go back to `/settings/integrations`. In the same ScanTriggerForm, pick the same repo + branch but set **path** to `nonexistent-subdir-that-does-not-exist`.

Click Scan.

- [ ] **Expected:** You land on `/scans/{new_scan_id}` showing the pending skeleton. Within ~5–15 s the scan flips to `status: 'failed'`. The page renders the error_message (something like "subpath not found" or similar — exact wording per Plan 06's traversal/not-found guard).
- [ ] **Expected:** A **Retry** button is visible.

**Step S14 — Click Retry**

- [ ] **Expected:** A new scan is enqueued; browser navigates to a **new** `/scans/{another_new_scan_id}`. The retry payload is sourced **server-side** from the original scan row (Plan 10 T-07.5-10-03 mitigation — the user can't tamper with installation_id / repo / branch / path client-side).
- [ ] **Expected:** This second scan also fails (same wrong path) — but the second scan row exists with its own `error_message`, distinct `scan_id`, and the user can Retry again or navigate away.

## Token Leakage Check (defense-in-depth)

**Step S15 — Grep Fly logs for raw GitHub install tokens**

```bash
fly logs -a infracanvas-api-dev | grep -E "ghs_|x-access-token"
```

- [ ] **Expected: zero matches** across the entire smoke test window. GitHub installation tokens have the prefix `ghs_`, and the embedded clone-URL credential is `x-access-token` — neither must appear in any structlog emission, error frame, or stderr capture. (Plan 06 applies token redaction in three layers: stderr decode chokepoint `_redact_token`, outer `except` re-redaction of `str(exc)`, and never binding `token` as a structlog kwarg. The gating test is `tests/jobs/test_scan_repo.py::test_token_redacted_in_stderr_log`.)

**Step S16 — Spot-check failure-path error messages**

```bash
fly logs --app infracanvas-api-dev | grep "scan_repo.failed" | tail -5
```

- [ ] **Expected:** Each error message contains `***` (the redaction sentinel) wherever a token would otherwise appear — never the raw `ghs_*` string.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| Step S3: install URL has wrong slug or no slug | `NEXT_PUBLIC_GITHUB_APP_SLUG` not set in Vercel or build is stale | Re-set on Vercel (P4), trigger redeploy, hard-refresh the page |
| Step S5: 403 on install-callback ("state mismatch") | The `state` the dashboard sent doesn't match `team.clerk_org_id` (your active org changed between InstallButton click and callback, or you changed Clerk orgs in another tab) | Sign out, sign back in, ensure you're on the **same** org for the whole flow |
| Step S5: 401 on install-callback | Clerk session expired during the GitHub install screen | Sign back in on the dashboard, then re-do steps S2–S5 |
| Step S5: 500 with "Could not deserialize key data" or JWT signing error | PEM newlines were corrupted in `flyctl secrets set` (you didn't use `"$(cat ...)"`) | Re-run step P3 with the exact `"$(cat ~/secrets/infracanvas-dev-app.pem)"` form |
| Step S6: typeahead fires per-keystroke (no debounce) | RepoCombobox's 250 ms `setTimeout` debounce regressed | Open DevTools Network tab to confirm; if regression, file as a Plan 08 regression and revert |
| Step S6: combobox empty even though repo exists | Installation lacks access to the fixture repo (you picked "Only select repositories" but excluded it, or the App was installed against a different org) | Open `https://github.com/settings/installations/{id}` → Configure → grant access to the fixture repo → reload `/settings/integrations` |
| Step S10: scan stays pending forever | Worker not running, or clone is hanging on auth (token mint failed silently) | `flyctl status -a infracanvas-api-dev` to check worker health; `flyctl logs -a infracanvas-api-dev` to see scan_repo emissions; check `GITHUB_APP_PRIVATE_KEY` is intact |
| Step S10: scan flips to `failed` immediately with "clone failed (Repository not found)" | Installation lacks access to that specific repo | See "combobox empty" above |
| Step S11: Stripe meter event missing | `team.stripe_customer_id` is null (team never went through Phase 6's billing setup), or Stripe meter event name mismatch | Verify `SELECT stripe_customer_id FROM teams WHERE id=...;` in Neon dev; check Phase 6 D-09 expected meter event name matches `app.settings.stripe_meter_event_name` |
| Step S15: grep finds matches | Token redaction regressed somewhere on the failure path | Capture the matching log line, identify the source (file:line), file as a Plan 06 regression — **do not approve the smoke** |

## Notes

- This file is the canonical end-to-end manual test for Phase 7.5. Phase 8
  (push webhooks + auto-scan) will reuse the same `scan_repo` taskiq job, so
  most of this checklist will carry over with one change: the trigger becomes
  a webhook delivery instead of an explicit "Scan" button click.
- Stripe should be **test mode** for this smoke. Production meter events would
  bill you for the smoke run.
- Do **not** install the App against an org you do not own or do not have
  written permission to install Apps into.
- The fixture repo data (parsed Terraform, findings, etc) is uploaded to **dev**
  R2 — fine for smoke. Don't smoke against repos containing customer data.
