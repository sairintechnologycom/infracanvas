# Phase 7.5: GitHub Repo Connector — Context

**Gathered:** 2026-05-03

<domain>
## Phase Boundary

Let authenticated dashboard users install the InfraCanvas GitHub App, browse the repos that install grants access to, pick a branch and (optional) subdirectory, and trigger an on-demand scan. Result lands in Phase 7's existing scan history/detail/share flow.

**In scope:**
- GitHub App registration + install callback handler
- `github_installations` Postgres table (team-scoped via existing RLS)
- Three new backend endpoints: `GET /v1/github/installations`, `GET /v1/github/repos`, `GET /v1/github/branches`
- One new taskiq job: `scan_repo(install_id, repo, branch, sha, path)` — clones, scans, uploads to R2, commits via the existing Phase 6 D-09 sync path
- Live UI on `/settings/integrations` replacing Phase 7 D-03's disabled placeholder: install button, repo combobox, branch dropdown, path input, "Scan" button
- New scan status value `pending` (Phase 6 currently has `ready`); dashboard polls `/v1/scans/{id}` every 2s until ready
- `scans.source = 'github'` and `scans.source_path` columns

**Out of scope (belongs in later phases):**
- Push webhooks + auto-scan worker → Phase 8 (uses the same `scan_repo` taskiq job built here)
- Slack alerts on Critical findings → Phase 8
- GitLab / Bitbucket / Azure DevOps connectors → v1.2 Enterprise (per ROADMAP success criterion 5)
- GitHub PR Bot (diff comment + status check) → v1.2 (PRB-01..02)
- CLI changes (e.g., `--root` flag) — Phase 7.5 must not touch CLI parser
- First-run "no repos yet" onboarding tour → not in this scope
- Token rotation/audit trail surfaces beyond what GitHub already exposes

</domain>

<decisions>
## Implementation Decisions

### Auth model
- **D-01:** GitHub App (not OAuth App, not both). Single InfraCanvas App per env (dev + prod). Org admin installs the App and selects which repos to grant. We persist only the long-lived `github_installation_id` (opaque integer, not a secret). Webhook delivery in Phase 8 reuses the same install. Rationale: clean fit for SaaS multi-tenant model; Phase 8 webhooks need installation-level identity; aligns with Phase 6 D-04 (Clerk org webhook pattern); avoids OAuth's per-user-token brittleness when the installer leaves the team.
- **D-06:** Tokens minted per scan, never persisted. App's RSA private key lives in Fly secrets (`GITHUB_APP_PRIVATE_KEY` per env). At scan time, worker reads key → signs a 5-min App JWT → POSTs to GitHub `/app/installations/{id}/access_tokens` → uses returned ~1hr installation token in clone URL → lets it expire. DB stores no token material. Rationale: smallest secret blast radius; matches GitHub's recommended pattern; no DB-encryption complexity; key rotation = rotate one Fly secret.
- **D-07:** Token enters clone subprocess via env var + URL embedding. Worker sets `GIT_TOKEN` on subprocess env, builds clone URL `https://x-access-token:${GIT_TOKEN}@github.com/org/repo.git`, runs `git clone --depth=1 --branch={branch} URL /tmp/scan-{uuid}`. After scan + upload, `rm -rf /tmp/scan-{uuid}`. Rationale: standard pattern, no global git-config side effects, no temp ASKPASS race.

### Execution model
- **D-02:** Scan execution = taskiq worker. API enqueues `scan_repo(install_id, repo, branch, sha, path)` on the existing Phase 6 D-13 Upstash Redis queue. Same Fly worker process picks it up, shallow-clones, runs `infracanvas scan` subprocess, uploads JSON to R2, calls the existing Phase 6 D-09 commit handler so Stripe meter fires through the same path the CLI uses. Phase 8 webhook scans will reuse this exact job. Rationale: clean async by default; bounded blast radius for hung scans; reuses the queue we already pay for; one job = both phases.
- **D-08:** No dedup on repeat Scan clicks. Every click enqueues. Stripe meter fires per scan upload (Phase 6 D-08), so cost is bounded by user intent. Worker MAY skip the clone if HEAD sha matches an existing scan within the last 60s (cheap GitHub `GET /repos/{r}/git/ref/heads/{b}` call), but still creates a row pointing at the existing R2 object so the user gets immediate feedback. Rationale: simplest UX; user trust beats hidden silent rejection; if abuse emerges, add per-team rate limit later.
- **D-09:** Subpath handling = `cd /tmp/scan-{uuid}/{subpath} && infracanvas scan .` — no CLI changes. The CLI's existing Phase 5.1 module resolution handles `../` references back into the repo. Path stored on `scans` row as `source_path`. Rationale: zero CLI surface change in this phase; matches the experience CLI users already have.

### Backend API surface
- **D-10a:** `GET /v1/github/installations` — returns `[{ installation_id, github_account_login, github_account_type, installed_at, installed_by_user_id }]` for the team. Pulled from local `github_installations` table.
- **D-10b:** `GET /v1/github/repos?installation_id=X&q=foo` — proxies to GitHub `GET /installation/repositories` (or `/search/repositories?q=...+user:{owner}` when `q` is set). Returns `[{ full_name, default_branch, private }]`. Cached 60s in Upstash Redis keyed by `(installation_id, q)`. Rationale: combobox typeahead can chew rate limit fast; 60s TTL is a lossy-cache sweet spot.
- **D-10c:** `GET /v1/github/branches?installation_id=X&repo=org/name` — proxies `GET /repos/{owner}/{repo}/branches`. Lazy-loaded on repo select in the UI. No cache (branches change frequently and are usually <30 per repo).
- **D-10d:** `POST /v1/github/install-callback` — endpoint that GitHub redirects to after install completes (`?installation_id=X&setup_action=install`). Verifies the redirect carries a Clerk session cookie (user is authenticated), upserts a `github_installations` row keyed by `(team_id, installation_id)`. Returns 302 redirect back to `/settings/integrations`.
- **D-10e:** `POST /v1/scans/from-github` — takes `{installation_id, repo, branch, path}`. Server resolves HEAD sha (one GitHub API call), creates `scans` row in `pending` status with `source='github'`, enqueues taskiq `scan_repo` job, returns `{scan_id}`. Frontend redirects to `/scans/{scan_id}`.

### Database schema additions
- **D-11:** New `github_installations` table:
  ```sql
  CREATE TABLE github_installations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    team_id UUID NOT NULL REFERENCES teams(id) ON DELETE CASCADE,
    github_installation_id BIGINT NOT NULL,  -- GitHub's opaque integer ID
    github_account_login TEXT NOT NULL,       -- denormalized for display
    github_account_type TEXT NOT NULL,        -- 'User' or 'Organization'
    installed_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    installed_by_user_id TEXT NOT NULL,       -- clerk user id
    UNIQUE (team_id, github_installation_id)
  );
  ```
  RLS policy: same `current_setting('app.current_team_id', true)::uuid = team_id` as other team-scoped tables (Phase 6 D-02).

- **D-12:** `scans` table extensions: add `source_path TEXT NULL` (the `/{subpath}` field, default `.`). The existing `source` column (Phase 6) gains `'github'` as an allowed value alongside `'cli'` and `'manual'`.

- **D-13:** New `scans.status = 'pending'` value alongside the existing `'ready'`. Phase 6's `commit` handler flips `pending → ready` on success. On scan failure (clone error, parse error, R2 upload error), worker flips `pending → failed` and writes `scans.error_message`. Dashboard polling treats `failed` as a terminal state with a retry CTA.

### Frontend (dashboard) surface
- **D-04:** `/settings/integrations` is the connect+browse+scan surface. Replaces Phase 7 D-03's disabled placeholder. Page state machine:
  - **Pre-install** (no rows in `/v1/github/installations` for this team): show "Install InfraCanvas GitHub App" button. Click opens `https://github.com/apps/infracanvas/installations/new?state={team_id}` in a new tab. After install, GitHub redirects to our callback; the user is back on `/settings/integrations` with installations populated.
  - **Post-install:** list of connected installations (account login + installed-at + "Manage on GitHub" link); for each, an inline scan-trigger row: repo combobox → branch dropdown → path input (default `.`) → "Scan" button.
- **D-05:** Repo+branch picker = searchable shadcn Combobox. Backend returns first ~30 repos sorted by `pushed_at` desc; user can type to filter (debounced 250ms, server-side search). Branch dropdown lazy-loads on repo select, defaults to repo's `default_branch`. Path field is a free-text input.
- **D-03:** Scan progress = optimistic redirect + poll. On `POST /v1/scans/from-github` success, dashboard redirects to `/scans/{scan_id}`. The detail page polls `/v1/scans/{scan_id}` every 2s. While `status='pending'`, render a skeleton with branch + commit + "Scanning…" spinner. On `'ready'`, render the viewer (Phase 7 D-08 path). On `'failed'`, show error message + "Retry" button (re-POSTs to `/v1/scans/from-github` with same params).

### Security
- **D-14:** Install callback verifies the `state` parameter equals the user's current `team_id` (mitigates install-CSRF where a malicious link installs the App into team B while user A is signed in).
- **D-15:** GitHub App webhook secret (`GITHUB_APP_WEBHOOK_SECRET`) lives in Fly secrets per env. Phase 7.5 doesn't process webhooks yet (that's Phase 8), but the App is configured with the secret on registration so Phase 8 doesn't need to re-register.
- **D-16:** Clone subprocess runs in `/tmp/scan-{uuid}` (UUIDv4, not the scan_id) and is `rm -rf`'d after upload regardless of success/failure (Python `try/finally`). Worker has no privileged filesystem access beyond `/tmp`.

### Observability
- **D-17:** Carry forward Phase 6 D-18..D-21: Sentry tags every `scan_repo` job with `team_id`, `installation_id`, `repo`, `branch`, `sha`. Structlog logs include the same context. Request-ID from the originating `POST /v1/scans/from-github` propagates into the taskiq job (Phase 6 D-21 pattern).

### Claude's Discretion (planner picks)
- Exact GitHub API client library — `PyGithub` vs `githubkit` vs raw `httpx`. `httpx` may be cleanest given we already use it for backend calls.
- JWT signing library for App auth — `pyjwt[crypto]` is the GitHub-recommended choice but planner verifies.
- Whether the install-callback redirect is handled by FastAPI directly or proxied through Next.js (likely FastAPI for simplicity since the redirect target is a backend URL).
- Exact polling interval ramp (2s flat vs 1s/2s/5s backoff) — UI-side tuning; default to 2s flat.
- shadcn Combobox vs custom Popover+Command — shadcn already landed in Phase 7.1; planner picks the closest existing primitive.
- Repo-list cache key strategy — `(installation_id, q)` vs invalidate-on-search; planner verifies under load assumptions.

</decisions>
