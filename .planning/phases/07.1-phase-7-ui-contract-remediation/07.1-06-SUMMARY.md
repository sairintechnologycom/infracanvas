## Decisions Made

1. **AlertDialogAction variant="destructive" pattern** — Used the simpler `<AlertDialogAction variant="destructive">Revoke</AlertDialogAction>` form rather than wrapping a `<Button asChild>` (RESEARCH §Pattern 3 sample). shadcn's `AlertDialogAction` already does `<Button variant={variant} asChild><AlertDialogPrimitive.Action /></Button>` internally, so wrapping another `<Button>` via `asChild` would conflict on the Slot. Pitfall 4 is satisfied (the destructive variant preserves focus-visible ring), and the plan's `grep -q 'variant="destructive"'` acceptance check matches.

2. **`refreshKey={generatedUrl ? 1 : 0}` instead of a counter** — UX is one-generation-per-modal-session (the modal collapses the form after success). Toggling 0→1 once is sufficient to trigger a single refetch. Matches the plan's "simplest implementation" guidance.

3. **URL+method-aware fetch mock** — The existing `mockGenerateOk` returned a single canned response for any URL. Mounting `ShareLinksList` inside ShareModal added a sibling `GET /api/scan-share` call on mount, which the single-shot mock could not satisfy (the test rendered with `links: undefined`, crashing on `.length`). The fix routes by method on the same URL prefix. This is a scope-bounded test-only change directly caused by Task 3's mount, classified as deviation Rule 1.

4. **`vi.hoisted()` for sonner spies** — `vi.mock(...)` factory bodies are hoisted above top-level `const` declarations, so referring to `toastSuccess` directly throws "Cannot access … before initialization". `vi.hoisted()` returns the spies in the same hoist phase as the mock factory, eliminating the race.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] mockGenerateOk broke after mounting ShareLinksList in ShareModal**

- **Found during:** Task 3, after the new mount-test ran the existing copy-toast tests through ShareModal.
- **Issue:** The single-response `mockGenerateOk()` returned `{ id, token, share_url, … }` for every fetch — but `ShareLinksList` (now mounted by ShareModal) expected `{ links: ShareLink[] }` on its mount-time GET. `body.links` was `undefined`; `.length` threw.
- **Fix:** Replaced the `vi.fn().mockResolvedValue(...)` with `vi.fn().mockImplementation(...)` that returns `{links: []}` for GET and the share-create response for POST.
- **Files modified:** `dashboard/__tests__/share-modal.test.tsx`
- **Commit:** `af2527e` (rolled into Task 3)

**2. [Rule 1 — Bug] vi.mock factory hoisting referenced uninitialized top-level vars**

- **Found during:** Task 2 GREEN — first run of revoke-flow tests.
- **Issue:** `const toastSuccess = vi.fn(); vi.mock('sonner', () => ({ toast: { success: toastSuccess … } }))` — Vitest hoists `vi.mock` above the `const`, so when the mocked `RevokeShareLinkButton` is imported at test load it accesses `toastSuccess` before it is initialized.
- **Fix:** Switched to `const { toastSuccess, toastError } = vi.hoisted(() => ({ toastSuccess: vi.fn(), toastError: vi.fn() }))` so spies and mock are in the same hoist phase.
- **Files modified:** `dashboard/__tests__/share-links-list.test.tsx`
- **Commit:** `a7c5dcd`
