## Priority Fixes

### P0 — Viewer embed layout break: `App.tsx` `h-screen w-screen`

**Surface:** `/scans/[id]` — `viewer/src/App.tsx:95`

**Evidence:** `<div className="flex flex-col h-screen w-screen">` — `w-screen` is 100vw and breaks out of the flex column that contains the sidebar. The viewer tries to be full-viewport-width regardless of the sidebar container. The `h-screen` ignores the parent container's constrained height. User report confirmed: nodes pushed off-screen, sidebar appears to disappear (viewport overflow, not DOM removal).

**Root cause:** `App.tsx` was designed for standalone HTML mode (single-file export) where it IS the root element. When embedded inside the dashboard shell, `w-screen` breaks the flex layout.

**Fix:** In `App.tsx`, change `h-screen w-screen` to `h-full w-full`. The `ScanViewerClient` already provides `h-full w-full` on its container div, and the scan detail page's `flex-1 min-h-0` provides the correct height constraint. Two-character class change but unblocks the entire canvas surface.

### P0 — Singleton store leak: FilterPanel, DetailPanel, SummaryBar, SearchBar ignore per-scan state

**Surface:** `/scans/[id]` — `viewer/src/components/FilterPanel.tsx:10-18`, `DetailPanel.tsx:12-13`, `SummaryBar.tsx:10-14`, `SearchBar.tsx:5-6`

**Evidence:** All four components call `useStore(...)` (the module-level singleton) rather than `useViewerStoreOrSingleton(...)`. When the dashboard mounts `<ViewerProvider store={perScanStore}>`, these components read from the global singleton which has `graph: null` forever. Result: SummaryBar is completely invisible (returns null at line 16 because `!graph`); FilterPanel is invisible (same); DetailPanel never shows selected nodes; SearchBar doesn't search anything.

**Root cause:** The dual-store fix (debug session `scan-detail-empty-canvas`, 2026-05-01) migrated `DiagramCanvas` and `ResourceNode` to `useViewerStoreOrSingleton` but explicitly deferred `FilterPanel`, `DetailPanel`, `SummaryBar`, `SearchBar`, `TabBar`, and FlowMap variants as not-yet-needed. Phase 7.1's unified tab shell now mounts these components; the deferred work was never picked up.

**Fix:** In each of the 4 components (plus FlowMap variants), replace `import { useStore }` with `import { useViewerStoreOrSingleton }` and replace each `useStore(...)` call with `useViewerStoreOrSingleton(...)`. The hook already exists and is correctly implemented.

### P0 — `/settings` route 404: no `page.tsx` at settings root

**Surface:** `/settings` — `dashboard/app/(dashboard)/settings/` directory

**Evidence:** The settings directory contains only `layout.tsx` + sub-routes (`billing/page.tsx`, `members/page.tsx`, `integrations/page.tsx`). Next.js requires a `page.tsx` at every route that should render. Clicking "Settings" in the sidebar navigates to `/settings` which returns Next.js's default bare "404 | This page could not be found." — no branding, no sidebar, no spec-compliant error copy.

**Fix:** Add `dashboard/app/(dashboard)/settings/page.tsx` that does `redirect('/settings/members')` (using Next.js `import { redirect } from 'next/navigation'`). 3 lines of code.

### P0 — Sparkline circles rendered as large ovals

**Surface:** `/` — `dashboard/components/scans/Sparkline.tsx:61`, `ScoreSparkline.tsx:35`

**Evidence:** `<svg viewBox="0 0 80 24" preserveAspectRatio="none">` inside `<div className="relative w-full h-full">` — the SVG stretches to fill the container without preserving aspect ratio. At 1440px desktop, the container is ~1200px wide × 64px tall, producing a 15:1 stretch ratio. The `<circle r={2.5}>` elements render as ~37px wide × 5px tall ovals (confirmed in screenshot: red and green blobs at min/max points). The polyline also skews horizontally.

**Fix:** In `Sparkline.tsx`, change `preserveAspectRatio="none"` to `preserveAspectRatio="xMidYMid meet"` OR change the SVG to use a wider viewBox that scales near-uniformly. Simpler: remove `preserveAspectRatio="none"`, keep `className="w-full"` (not `h-full`), let the SVG scale proportionally. In `ScoreSparkline.tsx`, increase the container from `h-[64px]` to `h-[80px]` (or spec's 120px). The viewBox height of 24 inside 64px produces 2.67× vertical scale, and 1200px width produces 15× horizontal — `preserveAspectRatio="none"` is the immediate culprit.

### P1 — Grade threshold inconsistency across 4 components

**Surface:** `/`, `/scans`, `/scans/[id]` — `ScoreCard.tsx:21-22`, `ScansTable.tsx:17-18`, `MetadataHeader.tsx:13-19`, `ShareViewer.tsx`

**Evidence:**
- `ScoreCard.gradeInfo()`: B+ at `≥85`, B at `≥80`
- `ScansTable.ScoreGradePill()`: B+ at `≥80`, no B grade (skips from B+ to C)
- `MetadataHeader.ScoreGradePill()`: same as ScansTable (B+ at `≥80`, no B)

A scan scoring 82 shows "B" on the home score card hero but "B+" in the table and detail header. Users see contradictory letter grades for identical scans across the same page load.

**Fix:** Extract a single `gradeInfo()` function (currently partial in `ScoreCard.tsx`) and import it into `ScansTable`, `MetadataHeader`, `ShareViewer`. Delete the 3 inline `ScoreGradePill` functions. Canonical thresholds: A+ ≥95, A ≥90, B+ ≥85, B ≥80, C ≥70, D ≥60, F <60.

### P1 — Sidebar `w-12` icon-only dead zone at md–xl breakpoints (768–1279px)

**Surface:** All authenticated routes — `dashboard/components/layout/Sidebar.tsx:41`

**Evidence:** `className="${visibility} md:flex xl:w-[220px] w-12 ..."` — at md (768px), sidebar shows but collapses to `w-12` (48px) icon-only with labels hidden. The hamburger button in TopBar is `md:hidden` (only visible below 768px). The spec explicitly targets 1440p and 1080p as desktop-only and says "no mobile breakpoints." Current behavior creates an unintended icon-only state at 768–1279px where the nav labels are hidden and there's no toggle to expand.

The user-reported "sidebar disappears on /scans" was an SSR snapshot artifact (layout.tsx is `'use client'`); after hydration the sidebar renders correctly at 1440px. The icon-only dead zone at 768–1279px is the persistent design defect.

**Fix:** Change sidebar breakpoint from `xl:w-[220px] w-12` to `w-[220px]` (always full width on desktop; spec doesn't define a collapsed state). Remove the `xl:inline hidden` and `xl:block hidden` qualifiers from label spans. Keep the `hidden md:flex` toggle for genuine mobile (below 768px).

### P1 — `TopFindings` does not render individual finding cards

**Surface:** `/` — `dashboard/components/home/TopFindings.tsx`

**Evidence:** Spec §/ item 3 requires: "3 finding cards (`border-l-4 border-sev-critical`, `bg-red-50/40`, `p-4`) each with: rule-id badge, finding title, resource id, `Open scan →` link." The implemented component shows a single count (`"1 critical finding in the latest scan"`) with a placeholder note `"Per-finding details (rule, title, resource) load with the full scan graph."` Full section regression — the spec's intended value (top 3 rule_id + title + resource combinations) is missing entirely.

The component documents the reason (list endpoint returns counts not full Finding objects) but the spec explicitly expected individual finding rows. This is a spec-versus-API data mismatch that must be resolved.

**Fix sketch:** Three options:
1. Add a route handler `/api/top-findings?scan_id={id}` that fetches the scan JSON from R2 (using the presigned URL) and returns only the top 3 critical findings. The home page RSC calls this route handler.
2. Add a `/v1/findings/top?limit=3` backend endpoint.
3. Accept the data constraint and formally amend the spec to remove the individual finding cards.

**Evidence:** `<select id="share-expiry" ... className="w-full border border-slate-300 rounded-md px-2 py-1.5 text-sm">` — native HTML select element in a modal that uses shadcn `<Dialog/>` and `<Input/>`. Also: label reads `"Expires in"` (spec: `"Link expires in"`).

### P1 — Homepage `<header>` element adds visual dead space

**Evidence:** `<header className="flex items-baseline justify-end"><p className="text-xs text-slate-500">Latest scan from <span className="font-mono">{latestScan.branch}</span></p></header>` — renders a right-aligned muted label above the score card that says "Latest scan from main". Not in the spec. With `gap-12` above and below, this sparse single-line header adds 48px of whitespace before the first real card. Combined with gap-12 between all four cards, total vertical rhythm is far too loose.

### P2 — Home dashboard `gap-12` used between all sections (spec: only hero→sparkline break)

**Surface:** `/` — `dashboard/app/(dashboard)/page.tsx:39`

**Evidence:** `<div className="max-w-7xl mx-auto px-8 py-12 flex flex-col gap-12">` — gap-12 (48px) is declared in spec as "2xl — Major section breaks (rare — only on the home dashboard between hero card and sparkline strip)." Currently applies between ALL four sections. Spec intends gap-6 (24px) between most cards and gap-12 only for the hero→sparkline pair.

**Fix:** Change container to `gap-6`. No `mt-6` overrides needed.

### P2 — MiniMap solid black: ReactFlow CSS may not load correctly in dashboard

**Surface:** `/scans/[id]` canvas — `viewer/src/components/DiagramCanvas.tsx:14`, `viewer/dist/lib/styles.css`

**Evidence:** User confirmed: "MiniMap renders as SOLID BLACK rectangle." The viewer's `dist/lib/styles.css` does NOT include `@xyflow/react/dist/style.css`. The `import '@xyflow/react/dist/style.css'` in `DiagramCanvas.tsx` is a side-effect import inside a library build that externalizes `@xyflow/react`. When Next.js processes the viewer via `transpilePackages`, this CSS import should be resolved from the workspace root `node_modules/@xyflow`. However, `@xyflow/react` is not listed in `dashboard/package.json` and is not present in `dashboard/node_modules/`. It exists only at the repo root via workspace hoisting — not guaranteed in all environments.

Without ReactFlow CSS: `.react-flow__minimap` has no background-color rule, and panels use CSS-variable defaults that may resolve to dark/transparent fills.

**Fix:** Add `@xyflow/react` as an explicit dependency in `dashboard/package.json`. Alternatively, add `import '@xyflow/react/dist/style.css'` to `dashboard/app/globals.css` or `ScanViewerClient.tsx`.

### P2 — `viewer/src/components/FilterPanel.tsx` uses sub-spec font sizes (9px, 10px, 11px)

**Surface:** Viewer FilterPanel — `viewer/src/components/FilterPanel.tsx:40,59,71,91`

**Evidence:** FilterPanel uses inline pixel font sizes `text-[10px]`, `text-[11px]`, `text-[9px]` throughout. Spec defines exactly 4 sizes: 12/14/16/28px. The viewer's dark-theme FilterPanel predates the spec but is now embedded in the dashboard via `ViewerApp`, making it a visible surface that violates the typography contract.

**Fix:** Replace `text-[10px]` and `text-[11px]` with `text-xs` (12px); replace `text-[9px]` with `text-xs`. Update section header dividers to use `text-xs uppercase tracking-wide` consistently.

### P2 — "View all →" and "Open scan →" links use `text-slate-900` (spec: `text-amber-600`)

**Surface:** Home dashboard — `dashboard/components/home/RecentScansTable.tsx:59`, `TopFindings.tsx:52`

**Evidence:** Both links use `text-sm text-slate-900 hover:text-slate-700 hover:underline` instead of spec-required `text-sm text-amber-600`. These are exactly the navigational CTAs the spec reserves amber for. Plan-08's global amber sweep over-converted these.

**Fix:** Change both to `text-sm text-amber-600 hover:underline`.

### P3 — `ScanViewerClient` error state uses `window.location.reload()` not `router.refresh()`

**Surface:** `/scans/[id]` error — `dashboard/components/scans/ScanViewerClient.tsx:88`

**Evidence:** Error button calls `window.location.reload()` — full page reload, losing sidebar state. Spec says: "Action: `[Try again]` (re-fetches the same RSC)." RSC re-fetch should use `router.refresh()` not full reload.

**Fix:** Import `useRouter` from `'next/navigation'`, call `router.refresh()` instead.

### P3 — `/scans/compare` 404 error copy missing "team" context

**Surface:** `/scans/compare` — `dashboard/app/(dashboard)/scans/compare/page.tsx:64`

**Evidence:** Error body text: `"you may not have access to it."` — spec says: `"you may not have access to its team."` — "team" is significant in a multi-tenant product.

**Fix:** Change `"it."` to `"its team."`.
