## Top 3 Priority Fixes

1. **ScansTable column sorting absent** — Users cannot sort scan history by Date, Score, Crit, High, or Drift as the spec requires; the table header renders static labels with no click handlers, no sort state, and no `aria-sort` attribute. Fix: add `sortCol` / `sortDir` state to ScansTable, wire `onClick` on each non-Source `<th>`, pass sort params to the URL (already plumbed in `ScansPage`), and set `aria-sort="ascending"|"descending"|"none"` per header.

2. **Pagination is Prev/Next cursor-only — numbered pages missing** — UI-SPEC §/scans specifies `shadcn <Pagination/> — Prev / numbered / Next`. The installed `Pagination.tsx` is a hand-rolled Prev/Next cursor pair (no page numbers, no "Showing N–N of N" count). Fix: replace `Pagination.tsx` with the shadcn `<Pagination/>` primitive; backend already provides `next_cursor`; add a `total_count` field to `ScanListResp` and render the `Showing N–N of total` footer row per spec.

3. **ShareModal uses native `<select>` for expiry — shadcn `<Select/>` required** — Plan 03 migrated `ScanFilters` to shadcn `<Select/>` but `ShareModal` still uses a native `<select id="share-expiry">` (line 139). This creates an inconsistent primitive surface within the same modal that also uses shadcn `<Dialog/>` and `<Input/>`. The expiry label text also deviates (`"Expires in"` vs spec `"Link expires in"`). Fix: replace the `<select>` with `<Select value={expiryChoice} onValueChange={...}><SelectTrigger/><SelectContent>...` and update the label to `"Link expires in"`.

## Detailed Findings

### Pillar 1: Copywriting (3/4)

**PASS — most spec strings correctly implemented:**
- `"No scans yet"`, `"No scans match your filters"`, `"Try widening the date range…"`, `"Clear all filters"` — all correct.
- Share modal: `"Share this scan"`, `"Generate share link"`, `"1 day"`, `"7 days"`, `"30 days"`, `"Never (not recommended)"`, `"Password (optional)"`, `"Active share links"` — all correct.
- `⚠ Anyone with this link…` — correct.
- Toast strings: `'Link copied to clipboard'`, `'Share link revoked'`, `"Couldn't generate share link. Try again."` — all spec-compliant.
- `"Revoke this share link?"` AlertDialog heading — correct.
- Share landing: `"This share link has expired"` / `"Ask the person who…"` and `"This share link is no longer active"` / `"The link has been revoked."` — correct.
- `"No active share links."` — correct (minor: spec says `"No share links yet for this scan."` but the plan-06 SUMMARY confirms the deviation to `"No active share links."` was intentional for the refactored component, and the test asserts this form).
- `"Compare against…"` modal title — correct.
- Settings copy: `"Save webhook URL"`, `"Connect GitHub (coming in 7.5)"`, `"Open billing portal"` — all correct.

**WARNING — deviations found:**

1. **Expiry label: "Expires in" vs spec "Link expires in"** (`ShareModal.tsx:137`). The spec §Copywriting defines the label as `Link expires in`.

2. **Copy button flash: "Copied" without checkmark** (`ShareModal.tsx:217`). Spec says button label flips to `Copied ✓` for 2s. Implementation renders `<Check size={12}/> Copied` (icon + text, no ✓ glyph in string). The `✓` is not explicitly in the rendered text — the check icon substitutes visually, but the spec's exact string form is not met.

3. **Dead-end card renders "Made with InfraCanvas" wordmark** (`share/[token]/page.tsx:41-49`). UI-SPEC §Share landing — State C explicitly says: `No "Made with InfraCanvas" wordmark on this state (it's a dead-end view)`. The `DeadEndCard` component used for expired/revoked/404 states renders the wordmark with the `InfraCanvas` link. This applies to all three dead-end states (expired, revoked, 404/not-found).

4. **"View all →" link color** (`RecentScansTable.tsx:57`). Spec §/ says `text-sm text-amber-600`. Implementation: `text-sm text-slate-900 hover:text-slate-700 hover:underline`. Plan 08 swept all `text-amber-600` text-links to slate — this conversion was correct for most links but the spec explicitly reserved amber for the `"View all →"` link on the home dashboard (it falls under "reserved for" CTA-adjacent navigation). Score impact absorbed; this straddles color and copy.

5. **"Open scan →" link color** (`TopFindings.tsx:53`). Spec §/ says `text-xs text-amber-600 hover:underline`. Implementation: `text-xs text-slate-900 hover:text-slate-700 hover:underline`. Same plan-08 sweep issue.

6. **"Scan not found" body text deviation** (`scans/compare/page.tsx:64`). Spec: `"you may not have access to its team."` Actual: `"you may not have access to it."` — drops the word "team", which is meaningful in a multi-tenant context.

7. **"Sign-in expired" (401) error state not present**. UI-SPEC defines a specific error card for 401: heading `Sign-in expired`, body `Your session ended. Please sign in again.`, action `[Sign in]`. No component exists for this state across any authenticated route.

### Pillar 2: Visuals (3/4)

**PASS:**
- App shell layout (sidebar 220px with `xl:w-[220px]`, top-bar `h-12`/48px, main content scrolls) — correct structure.
- MetadataHeader 52px strip — `h-[52px]` confirmed, correctly stripped of action buttons (moved to top-bar slot per RMD-05).
- CompareLayout 52px sticky strip — `h-[52px] sticky top-0` confirmed.
- Top-bar action slot pattern (TopBarActions provider + slot) — correctly ships Compare/Share into top-bar on `/scans/[id]`.
- shadcn `<Dialog/>` for ShareModal and ScanPickerModal — both fully migrated.
- 4-section diff card layout (Added/Removed/Changed/Findings) with `<Sheet/>` drill-down — present.
- Score-grade pills on ScoreCard, ScansTable, MetadataHeader, ShareViewer — all implemented.
- No spinner anywhere; loading states use text or Skeleton.

**WARNING:**

1. **Sparkline strip container height: 64px vs spec 120px** (`ScoreSparkline.tsx:35`). The spec §/ declares the sparkline card body height as `120px`. The implemented `div` that wraps the SVG is `h-[64px]`. The SVG viewBox itself is `0 0 80 24` (W=80, H=24 — a very compressed SVG). This makes the sparkline significantly smaller than the design contract intended and reduces readability of the score trend at the home dashboard.

2. **ScoreCard layout: flex-wrap / gap-x-10 instead of fixed 3-column gap-8** (`ScoreCard.tsx:58`). Spec §/ says `3-column flex (gap-8, items-center)`. Actual: `flex flex-wrap items-center gap-x-10 gap-y-4`. The `flex-wrap` allows the three columns to wrap to a second row at narrow viewports; since spec targets 1440p as primary and explicitly states no mobile breakpoints, this is a visual contract deviation. At the primary 1440p target it may look similar, but the column gap is 40px (gap-x-10) vs the spec's 32px (gap-8).

3. **"Identical scans" empty state not implemented in CompareLayout** (`components/compare/CompareLayout.tsx`). When all four diff sections (Added/Removed/Changed/Findings) contain zero items, the spec requires a special `"Identical scans"` card with body `"These two scans have no resource changes and no findings delta."` plus the four section cards showing `"No changes"` muted text. The DiffSection and ChangedDiffSection components each show per-section empty states (`"No added/removed resources."`, etc.) but there is no top-level detection that all four are empty and no `"Identical scans"` heading card.

4. **RecentScansTable in ScansPage uses `{data.items.length} loaded` not a proper total count** (`scans/page.tsx:40`). Minor visual gap — users cannot tell if there are more scans beyond the first page.

### Pillar 3: Color (3/4)

**PASS — core contract met:**
- 60/30/10 split correctly implemented: `bg-white` dominant, `bg-slate-50` for sidebar/top-bar/header strips, `bg-amber-400` accent reserved for CTA buttons and active sidebar indicator only.
- Accent (`bg-amber-400`) usage limited to: ScanPickerModal Compare CTA, ShareModal Generate CTA, BillingPage CTA, PasswordGate Unlock CTA, `/compare` landing CTA, sidebar active `border-l-2 border-amber-400` — all are spec-declared uses.
- No `ring-amber-*` focus rings found — all focus states use `ring-slate-400`.
- No `bg-amber-500/600` outside compare/ drift palette.
- Grade pills: `bg-green-100/sky-100/amber-100/orange-100/red-100` with matching text colors — correct.
- Drift palette (Added=green, Removed=red, Changed=amber) in `components/compare/*` — correctly exempted.
- Severity tokens (`text-sev-critical`, `text-sev-high`, `text-sev-medium`) used correctly in table cells.
- Destructive color: `text-red-600` on Revoke button — correct.
- Hardcoded hex values in `SeverityBadge.tsx` and `MetadataHeader.tsx` use CSS variable fallbacks (`text-[color:var(--color-sev-critical,#ef4444)]`) — acceptable pattern for token fallbacks.

**WARNING:**

1. **`bg-amber-500` and `text-amber-600` in compare components conflict with global gate** (`ChangedDiffSection.tsx:23`, `DiffNodeList.tsx:17`). The `ChangedDiffSection` renders `<span className="... bg-amber-500" />` for the amber dot, and `DiffNodeList` uses `bg-amber-50 border-l-2 border-amber-500` for the changed row style. Plan 08 exempted `components/compare/*` from the drift-gate, but UI-SPEC §Color "Accent reserved for" says amber-500 is never used (only amber-400 is the accent). The exemption was intended to permit `bg-amber-50` and `text-amber-600` for semantic drift uses (the `changed` state), not to permit `bg-amber-500` (the banned higher saturation). This is a minor precision issue — visually close, but it breaks the token contract.

2. **"View all →" and "Open scan →" links converted from spec amber to slate** (`RecentScansTable.tsx:57`, `TopFindings.tsx:53`). Spec explicitly declares these as `text-amber-600`. Plan 08's global `text-amber-600 → text-slate-900` sweep over-applied, converting spec-required amber navigation links. At a visual level these now look identical to body text, losing the affordance that marks them as navigational CTAs.

### Pillar 4: Typography (3/4)

**PASS — 4-size / 2-weight contract largely met:**
- Font size distribution: `text-sm` (128 uses), `text-xs` (60), `text-base` (26), `text-[28px]` (1). No `text-lg`, `text-xl`, `text-2xl` found anywhere outside `components/ui/` (shadcn primitives exempt). This passes the 4-size limit.
- Font weights: `font-semibold` (54), `font-medium` (20), `font-normal` (2). The 2 `font-normal` uses are within the spec's acceptable range (body text is `weight: 400 regular`). No `font-bold`, `font-extrabold`, `font-light`, `font-thin` found.
- `text-[28px] font-semibold tabular-nums` on the score display — exact spec match.
- `font-mono text-xs` for commit SHAs (MetadataHeader, ScansTable, ScanPickerModal) — correct.
- `font-mono text-sm` for branch names in the detail strip — correct.
- `text-xs font-semibold uppercase tracking-wide` for table headers — correct.
- `tabular-nums` on all numeric table cells — confirmed.

**WARNING:**

1. **Grade threshold inconsistency across components** — This is a data-display accuracy issue visible in the typography of score-grade pills:
   - `ScoreCard.tsx`: B+ assigned at `score >= 85`, B assigned at `score >= 80` (5-point sub-range for B+).
   - `ScansTable.tsx`: B+ assigned at `score >= 80`, B grade **does not exist** (jumps A → B+ → C).
   - `MetadataHeader.tsx` and `ShareViewer.tsx`: Same table as ScansTable (A/B+/C/D/F, no B).
   - UI-SPEC §Color score-grade table defines: A/A+ ≥ 90, **B/B+ 80-89**, C 70-79, D 60-69, F < 60. The spec implies B and B+ should both exist as grades in the 80-89 range.
   - Impact: a scan scoring 82 shows "B+" in the ScansTable/MetadataHeader but would show "B" in the ScoreCard hero widget. Users see contradictory grade labels for the same score across the dashboard.

- Arbitrary spacing used in ScanPickerModal (`w-[min(560px,90vw)]`, `max-h-[80vh]`, `min-h-[200px]`) — not on the declared scale but is modal-internal chrome; acceptable.

The spec §/scans defines the pagination footer as `bg-slate-50 border-t px-6 py-3 flex items-center justify-between` with left-side `"Showing 1-25 of 127"` count. The actual page heading shows `"{data.items.length} loaded"` — not the same and not in the pagination footer position. The `Pagination` component itself only renders Prev/Next without total count.

### Pillar 6: Experience Design (2/4)

**PASS — foundational states present:**
- Loading states: `ScanViewerClient` and `ShareViewer` show `"Loading scan diagram…"` text while JSON fetches; `ScansPage` wraps `ScansTable` in `<Suspense fallback={<ScansTableSkeleton/>}>` with `animate-pulse` skeleton rows; `ScanPickerModal` shows loading text; `ShareLinksList` has `loading` boolean state — all covered.
- Error states: `ScanViewerClient` catches fetch errors and shows "Could not load scan diagram" with retry; `ShareModal` surfaces generation failure as persistent toast; `RevokeShareLinkButton` catches DELETE failure with `toast.error`; share landing handles 410/404/5xx with `DeadEndCard`. No silent `catch {}` blocks (RMD-03 resolved).
- Empty states: `/` no-scans state, `/scans` filtered-empty state, `ScanPickerModal` no-scans state, `ShareLinksList` no-links state — all present.
- Disabled states: `ScanPickerModal` Compare CTA disabled until selection; `PasswordGate` submit disabled while submitting; `Pagination` Prev/Next disabled at boundaries; GitHub integration button disabled per spec.
- No spinners found — compliant.
- Toaster mounted at app root `layout.tsx` (not in `(dashboard)/layout` — Pitfall 3 compliant).
- Focus rings: `ring-slate-400` everywhere, no amber focus rings.

**BLOCKER — missing interactions:**

1. **ScansTable column sorting not implemented** (`ScansTable.tsx:110-126`). The spec §/scans defines sortable columns (Date, Score, Crit, High, Drift) with `aria-sort` attributes and lucide sort icons in headers. The actual `ScansTable` renders all 8 column headers as static non-interactive `<th>` elements with no click handlers, no sort state, and no `aria-sort` attribute. The scans page URL already supports `?sort=` and `?order=` params (wired in `ScansPage`). The table-header interaction layer is entirely absent.

2. **Pagination is cursor-only Prev/Next — spec requires numbered pages with total count** (`Pagination.tsx`). UI-SPEC §/scans: `shadcn <Pagination/> — Prev / numbered / Next; cursor-based per Phase 6 paginated endpoint`. The implementation is a hand-rolled `Prev / Next` cursor pair with no page numbers, no `shadcn <Pagination/>` primitive, and no `"Showing 1-25 of 127"` count display. Backend API already returns `next_cursor`; this is a frontend gap only.

**WARNING — missing interactions:**

3. **ShareModal uses native `<select>` instead of shadcn `<Select/>`** (`ShareModal.tsx:139`). All other filter/form selects in the dashboard were migrated to shadcn `<Select/>` in Plan 03, but the expiry `<select>` in ShareModal was not. It is a native HTML element that will not match shadcn's visual styling (height, padding, border radius, dropdown appearance).

4. **Keyboard j/k navigation on /scans table absent**. UI-SPEC §Interaction: `j / k (vim) on /scans table to move row selection up/down; Enter to open scan. Documented in a kbd hint on hover of the table title.` No `keydown` listeners, no `aria-activedescendant`, no `kbd` hint anywhere in `ScansTable` or `ScansPage`.

5. **"Identical scans" empty state for /compare not implemented**. UI-SPEC §/compare defines: when all four diff sections are empty, show a full-width centered card with heading `"Identical scans"` / body `"These two scans have no resource changes and no findings delta."` The current `CompareLayout` has no top-level empty-state detection — each section independently shows `"No {type} resources."` which is functionally equivalent but visually and contractually incorrect.

6. **"Sign-in expired" 401 error state not wired**. UI-SPEC §Copywriting defines a specific error card for backend 401: heading `Sign-in expired`, action `[Sign in]` (Clerk `<SignInButton/>`). No component or route-level handler exists for this state. Clerk middleware handles unauthenticated access to protected routes, but mid-session 401s from backend API calls (e.g. expired Clerk tokens) surface as unhandled errors.

7. **DrillDownSheet body is a placeholder stub** (`DrillDownSheet.tsx`). Plan 05 accepted this as a deferred disposition (T-07.1-12), but the spec §/compare defines the drill-down as a `<Sheet/>` with `<ViewerProvider scan={scopedScan}><DiagramCanvas/></ViewerProvider>`. Users clicking a resource row in the compare view see an empty/placeholder sheet. This is a functional gap in the compare flow.
