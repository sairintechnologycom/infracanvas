# Phase 7.1: Phase 7 UI Contract Remediation — Research

**Researched:** 2026-04-29
**Domain:** Frontend design-system installation + UI remediation (shadcn/ui v4, Tailwind v4, Next.js 15 App Router, React 18.3, Radix-direct → shadcn migration, Sonner toasts, AlertDialog destructive flow, Sheet drill-down, slot composition)
**Confidence:** HIGH (verified against npm registry + shadcn official docs + full codebase grep on 2026-04-29)

## Summary

Phase 7 shipped functional but diverged from the approved `07-UI-SPEC.md` design contract on three blockers and ten polish-drift warnings. `07-UI-REVIEW.md` (2026-04-29, score 16/24) names the gaps verbatim. The corrective work is bounded: this is a remediation pass, not new feature work. Backend endpoints (`POST /v1/scans/{id}/share-links`, `DELETE /v1/scans/{id}/share-links/{share_id}`, `GET /v1/scans/{a}/compare/{b}` with full `ResourceDiff` payload including per-node `changed_fields[]`, `before`, `after`) are already shipped and tested. The diff data needed for attribute-level expanders is already in the wire format — the dashboard simply ignores most of it today.

The dominant cross-cutting fix is **initializing shadcn/ui**. Once the design system is installed, four hand-rolled or Radix-direct components (`ScanPickerModal`, `ShareModal`, `SettingsLayout`, `ScanFilters`) collapse to thin shadcn wrappers, the `<Toaster/>` slot becomes available at app root, the `<AlertDialog/>` destructive flow is one block away, and the `<Sheet/>` drill-down for compare is mechanical. The compare page rebuild is the second-largest piece: replace `CompareViewerPair` (the explicitly-rejected dual canvas) with the spec'd 4-section card layout (Added / Removed / Changed / Findings), where Changed rows expand inline to attribute-level `before → after` tables driven by `NodeDiff.changed_fields`/`NodeDiff.before`/`NodeDiff.after`. The polish-drift items are mechanical search-and-replace once shadcn standardizes the primitives.

**Primary recommendation:** Run shadcn init in Wave 0, install all 17 declared blocks plus `sonner` and `alert-dialog`, then sequence remediation as: (Wave 1) primitive migrations + Toaster + focus-ring/typography sweep (parallelizable across files) → (Wave 2) compare page rebuild + share-link list/revoke flow → (Wave 3) top-bar action slot + custom-range filter + sparkline tooltip + copy/breadcrumb cleanup. Total surface ≈ 8–10 plans.

### Pattern 3: AlertDialog destructive confirm (revoke flow)

**Source:** shadcn `alert-dialog` block (built on `@radix-ui/react-alert-dialog@1.1.15`)

```tsx
// dashboard/components/share/RevokeShareLinkButton.tsx (new, Wave 2)
'use client'
import {
  AlertDialog, AlertDialogAction, AlertDialogCancel, AlertDialogContent,
  AlertDialogDescription, AlertDialogFooter, AlertDialogHeader, AlertDialogTitle,
  AlertDialogTrigger,
} from '@/components/ui/alert-dialog'
import { toast } from 'sonner'

export function RevokeShareLinkButton({ scanId, shareId, shortToken, onRevoked }: Props) {
  async function handleRevoke() {
    const res = await fetch(`/api/scan-share?scan_id=${scanId}&share_id=${shareId}`, { method: 'DELETE' })
    if (!res.ok) { toast.error('Could not revoke share link.'); return }
    toast.success('Share link revoked')
    onRevoked()
  }
  return (
    <AlertDialog>
      <AlertDialogTrigger asChild>
        <button
          aria-label={`Revoke share link ${shortToken}`}
          className="text-sm text-red-600 hover:underline"
        >
          Revoke
        </button>
      </AlertDialogTrigger>
      <AlertDialogContent>
        <AlertDialogHeader>
          <AlertDialogTitle>Revoke this share link?</AlertDialogTitle>
          <AlertDialogDescription>
            Anyone with this link will get a 410 Gone response immediately. This cannot be undone — they&apos;ll need a new link to view the scan.
          </AlertDialogDescription>
        </AlertDialogHeader>
        <AlertDialogFooter>
          <AlertDialogCancel>Cancel</AlertDialogCancel>
          <AlertDialogAction onClick={handleRevoke} className="bg-red-600 hover:bg-red-700">
            Revoke
          </AlertDialogAction>
        </AlertDialogFooter>
      </AlertDialogContent>
    </AlertDialog>
  )
}
```

### Pattern 4: Top-bar action slot via React Context

**What:** Per-route action injection without parallel routes or prop drilling. **When:** Wave 3 — affects MetadataHeader and TopBar.

```tsx
// dashboard/components/layout/TopBarActions.tsx (new)
'use client'
import { createContext, useContext, useState, useCallback, type ReactNode } from 'react'

type ActionsContextShape = {
  actions: ReactNode
  set: (jsx: ReactNode) => void
  clear: () => void
}

const ActionsContext = createContext<ActionsContextShape | null>(null)

export function TopBarActionsProvider({ children }: { children: ReactNode }) {
  const [actions, setActions] = useState<ReactNode>(null)
  const set = useCallback((jsx: ReactNode) => setActions(jsx), [])
  const clear = useCallback(() => setActions(null), [])
  return <ActionsContext.Provider value={{ actions, set, clear }}>{children}</ActionsContext.Provider>
}

export function useTopBarActions() {
  const ctx = useContext(ActionsContext)
  if (!ctx) throw new Error('useTopBarActions must be inside TopBarActionsProvider')
  return ctx
}

export function TopBarActionsSlot() {
  const ctx = useContext(ActionsContext)
  return <>{ctx?.actions ?? null}</>
}
```

### Anti-Patterns to Avoid

- **Keeping `CompareViewerPair.tsx` behind a toggle.** The dual-canvas pattern is explicitly rejected in D-10. A toggle adds product ambiguity in a remediation phase. **Delete the file outright** (git history preserves it for any future "side-by-side" experiment).
- **Mounting `<Toaster/>` inside `(dashboard)/layout.tsx`.** Public `/share/{token}` toasts (revoke-link landing copy errors etc.) need a Toaster too. Mount at the *root* `app/layout.tsx` so both `(dashboard)` and `(public)` route groups inherit it.
- **Hand-rolling the "Active share links" list before backend confirms `GET /v1/scans/{id}/share-links`.** UI-REVIEW notes a TODO comment in `ShareModal.tsx:245-248` saying the backend list endpoint is "not yet implemented". The existing `DELETE` endpoint per success-criterion #4 IS shipped, but planner MUST verify the GET endpoint exists or the share-links-list task expands to include it. See "Open Questions" §1.
- **Initializing shadcn into a route worktree.** Run init in the canonical `dashboard/` directory on a single dedicated branch (Wave 0). `components.json` + `components/ui/*` files are import targets for every subsequent task — making them in parallel worktrees creates merge conflicts.
- **Migrating `<select>` → `<Select/>` and the `Custom range…` filter in the same task.** They land in the same file (`ScanFilters.tsx`) but the calendar work has separate test surface (popover open/close, two-month range). Split for test clarity and reviewer cognitive load.
- **Treating "polish drift closed" as one task.** It's mechanical but spans ~10 files. Group by violation type (typography, color, spacing, copy), not by file, so the commit + acceptance criteria are grep-able (see §"Common Pitfalls" pitfall 5).

## Common Pitfalls

### Pitfall 1: Tailwind v4 + shadcn theme variable collision
**What goes wrong:** shadcn init writes a fresh `globals.css` with `@theme inline` and OKLCH variables. The current `globals.css` already imports `@infracanvas/viewer/styles.css` BEFORE `@import "tailwindcss"` — this is load-bearing (gives viewer's `--color-sev-*` tokens precedence) per Phase 7 PATTERNS.md note 5. If init blindly overwrites, every severity color in the dashboard breaks.
**Why it happens:** The shadcn CLI is conservative — it usually prompts before overwriting — but the prompt is easy to dismiss as "yes overwrite, I'll restore later".
**How to avoid:** Wave 0 task explicitly checks-out `globals.css` to a backup before init; after init, re-add the viewer import as line 1 and any custom `.sidebar-collapsed` rules at the bottom. Verify the order: viewer styles → tailwindcss → shadcn theme block → custom utilities.
**Warning signs:** Severity badges go grey or default-color in `MetadataHeader`/`ScansTable`; `.sidebar-collapsed` no longer hides labels.

### Pitfall 2: shadcn assumes React 19, project is React 18
**What goes wrong:** shadcn v4 components are stripped of `forwardRef` because React 19 doesn't need it. They use `data-slot` for variant styling. On React 18 they still work, but `forwardRef`-style refs from third-party code (e.g. `react-day-picker` v9 internals) may print warnings.
**Why it happens:** Mismatch between shadcn's React-19 default assumption and the project's pinned React 18.3.
**How to avoid:** Accept the warnings as informational; they don't break runtime. Do NOT bump React to 19 in this phase — it's outside scope and risks viewer-package fallout (`@infracanvas/viewer` peer-deps target React 18 per Phase 5 D-10).
**Warning signs:** Console warnings like "componentWillReceiveProps is deprecated" from third-party packages; tests still pass.

### Pitfall 3: `<Toaster/>` not picked up by `/share/{token}` route
**What goes wrong:** Mounting Toaster inside `(dashboard)/layout.tsx` instead of the root `app/layout.tsx` means public share-link routes (which use `(public)/share/[token]/layout.tsx`) get no toasts. PasswordGate's "Incorrect password" path silently fails to surface a toast (today's behavior; UI-SPEC says this stays inline-only, but other share-landing toasts may be added later).
**Why it happens:** Easy mental model error — "toasts are for authed UI".
**How to avoid:** Mount `<Toaster/>` exactly once, in `app/layout.tsx` at the `<body>` level. Both route groups inherit it.
**Warning signs:** Share-modal toasts work, but any toast called from `/share/*` pages goes nowhere.

### Pitfall 4: AlertDialog destructive style overrides shadcn focus ring
**What goes wrong:** Adding `bg-red-600` to `AlertDialogAction` removes shadcn's default focus ring (`ring-slate-400`). Without an explicit `focus-visible:ring-…` reapplied, keyboard users see no focus state on the destructive Confirm button.
**Why it happens:** shadcn's `Button` variant tokens are merged with `cn()`; appending custom classes without `focus-visible:ring-2 focus-visible:ring-red-500 focus-visible:ring-offset-2` drops the ring.
**How to avoid:** Use the shadcn `Button` `variant="destructive"` instead of raw `bg-red-600`; or re-add `focus-visible:ring-…` classes when overriding.
**Warning signs:** Tab to Revoke button in keyboard testing → no visible focus.

### Pitfall 5: Polish-drift acceptance criteria too soft
**What goes wrong:** A task description like "normalize headings to spec" is unverifiable without subjective UI review. The grep audit in 07-UI-REVIEW.md gives concrete counts; planner MUST translate these to grep-able assertions.
**Why it happens:** Polish work feels visual; planners default to "human checks at end".
**How to avoid:** Use exact grep commands as acceptance criteria (see §"Code Examples" → "Polish-drift grep gates"). Re-run them in the verification phase.
**Warning signs:** "Polish closed" task closes but `grep -rn 'text-xl\|text-lg' dashboard/app dashboard/components` still returns hits.

### Pitfall 6: Scan picker keyboard nav without focus management
**What goes wrong:** UI-SPEC §Interaction mandates ↑/↓/Enter on the compare picker. Naive `onKeyDown` on the list `<div>` works but leaves the actual button focus elsewhere (search input). Screen readers announce the wrong item.
**Why it happens:** Keyboard handlers are written for the list element but focus is anchored to the first input.
**How to avoid:** Use `useRef` per-row + manage `tabindex={selected ? 0 : -1}` + call `el.focus()` on key navigation; OR adopt shadcn `<Command/>` which does this for free. **Recommendation: defer keyboard nav to a follow-on phase if shadcn `<Command/>` is not in scope.** Document the gap in 07.1 verification rather than partially implement.
**Warning signs:** Screen reader announces the search input continuously while user navigates list with arrow keys.

### Pitfall 7: Compare RSC fetches `ResourceDiff` but ignores `before`/`after`
**What goes wrong:** Backend already returns `NodeDiff.before` and `.after` (full attribute dicts) per `dashboard/lib/types.ts:60-66`. The current `DiffNodeList` ignores them. Planner could over-engineer "we need to expand the API" — wrong: the data is already on the wire.
**Why it happens:** Plan author doesn't grep the existing types file.
**How to avoid:** Cite `dashboard/lib/types.ts:60-66` as ground truth in the plan's must_haves. New code in `ChangedAttributesTable` reads `node.changed_fields[]` to drive the table rows, then looks up `node.before[field]` and `node.after[field]` for each.
**Warning signs:** Plan adds backend tasks ("extend compare endpoint with attribute deltas") — STOP, the data is already there.

### Pitfall 8: Top-bar slot Context fights RSC
**What goes wrong:** `useTopBarActions` is a client hook. If the RSC `/scans/[id]/page.tsx` tries to call it directly, build error. The slot must be set by a `'use client'` child component (`<ScanDetailActions … />`).
**Why it happens:** Mixing server and client contexts.
**How to avoid:** Render a thin client wrapper (`<ScanDetailActions scanId={…} branch={…} />`) inside the RSC; the wrapper effects on mount/unmount.
**Warning signs:** Build error: "useTopBarActions cannot be used in a Server Component".
