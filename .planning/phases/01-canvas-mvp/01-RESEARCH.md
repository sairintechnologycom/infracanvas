## Summary

Phase 1 is not a greenfield build — it's a completion and hardening sprint on a codebase that is roughly 60-70% implemented. The Python CLI pipeline (parse → graph → security → export) is structurally complete. The React viewer (DiagramCanvas, FilterPanel, DetailPanel, SummaryBar, ResourceNode, GroupNode, layout) is also substantially complete. The single-file HTML export and `build.sh` pipeline work end-to-end.

The critical gaps that block the Phase 1 success criteria are: (1) the `scan` command defaults to JSON output rather than auto-opening HTML — the "magical browser open" UX is not the default path today; (2) the free-tier gate (VWR-06) has no implementation — every finding detail is currently shown to all users; (3) the `serve` command (D-12) does not exist — `--watch` exists on `scan` but does not serve HTTP; (4) module recursion (PRS-04) is unimplemented; (5) the score dimensions need realignment to the required 5 (Security, Encryption, IAM Hygiene, Cost Efficiency, Tagging); (6) GitHub Actions publish workflow is absent; (7) the score card layout (D-08) needs redesign away from the SVG circle to a letter-grade-prominent layout with horizontal progress bars.

The final 15 AWS resource types (Claude's discretion within D-05 guidance) should be: `aws_vpc`, `aws_subnet`, `aws_security_group`, `aws_instance`, `aws_s3_bucket`, `aws_iam_role`, `aws_db_instance`, `aws_lambda_function`, `aws_lb` (ALB), `aws_eks_cluster`, `aws_iam_policy`, `aws_kms_key`, `aws_cloudwatch_log_group`, `aws_elasticache_cluster`, `aws_nat_gateway`. This selection maximises finding coverage (IAM policies are wildcard-prone, KMS keys have rotation rules, ElastiCache is often unencrypted) while covering the network tier for complete diagrams.

**Primary recommendation:** Sequence work as: (1) fix `scan` default to HTML + auto-open, (2) add free-gate to DetailPanel, (3) add `serve` command, (4) module recursion, (5) score dimension realignment + score card redesign, (6) GHA publish workflow. This ordering unblocks the "magical first run" demo earliest.

### Pattern 5: Module Resolution (PRS-04)

**What:** Detect `module` blocks in HCL, resolve `source = "./path"`, recurse up to 3 levels.
**When to use:** `parse_directory()` extension for module support.

```python
# Source: ASSUMED — extends existing hcl.py pattern
def _resolve_modules(directory: Path, parsed: ParsedTerraform, depth: int = 0) -> None:
    if depth >= 3:
        return
    for module_block in parsed._raw_modules:
        source = module_block.get("source", "")
        if source.startswith("./") or source.startswith("../"):
            module_dir = (directory / source).resolve()
            if module_dir.is_dir():
                sub_parsed = parse_directory(module_dir)
                # Tag resources with module path
                for res in sub_parsed.resources:
                    res.module = f"module.{module_block['name']}"
                parsed.resources.extend(sub_parsed.resources)
                _resolve_modules(module_dir, sub_parsed, depth + 1)
```

### Anti-Patterns to Avoid

- **Making `scan` default to JSON in Phase 1:** The success criteria explicitly requires browser opening; the current default format is `json` and must change to `html` with auto-open.
- **Storing finding details in free mode:** The gate must be enforced in the HTML itself (gateMode flag in exported HTML), not just in UI state that can be manipulated.
- **Using dagre for layout:** dagre is in package.json but the codebase uses a superior custom tier layout. Do not switch to dagre.
- **Running `build.sh` logic in Python at scan time:** The `viewer_template.html` must be pre-built. The `build.sh` pipeline copies it into the CLI package. Do not rebuild at scan time.
- **Adding pip dependencies for the `serve` command:** Python stdlib `http.server` is sufficient. No Flask/FastAPI.
- **Allowing `serve` to scan on every HTTP request:** Scan should only re-run on `.tf` file change events, not on every browser refresh.

## Common Pitfalls

### Pitfall 1: scan command defaults to JSON, not HTML
**What goes wrong:** Running `infracanvas scan ./terraform` today produces a `.json` file and does NOT open the browser. The "magical UX" success criterion is broken by default.
**Why it happens:** The `scan` command's `format` parameter defaults to `"json"`. HTML export and browser-open only activate with `--format html`.
**How to avoid:** Change the default format to `"html"` and always call `webbrowser.open()` unless CI is detected (D-11).
**Warning signs:** Integration test `test_integration.py` may need updating if it asserts JSON as default output.

### Pitfall 2: viewer_template.html not built before scan
**What goes wrong:** `export_html()` raises `FileNotFoundError` if `viewer_template.html` is absent. This happens on a fresh `pip install` before `build.sh` is run.
**Why it happens:** The template is built by `npm run build` in `viewer/` — this is not part of the pip install step.
**How to avoid:** Include the pre-built `viewer_template.html` in the Python package via `pyproject.toml` `[tool.hatch.build.targets.wheel] include` directive. The build pipeline (CI) must run `build.sh` before `pip install -e .` or wheel build.
**Warning signs:** `FileNotFoundError` in export during first-run tests in CI.

### Pitfall 3: Score dimensions mismatch
**What goes wrong:** The current `CATEGORY_RULES` in `scorer.py` defines Security, Encryption, Networking, IAM, Tagging — the requirements specify Security, Encryption, IAM Hygiene, Cost Efficiency, Tagging.
**Why it happens:** Networking and IAM Hygiene/Cost Efficiency are different from the spec's 5 dimensions.
**How to avoid:** Remap CATEGORY_RULES to: Security (SEC-001,003,004,005), Encryption (SEC-002,006,009), IAM Hygiene (SEC-007,008), Cost Efficiency (static heuristics on instance type/reserved), Tagging (SEC-010).
**Warning signs:** Score card shows "Networking" category or missing "Cost Efficiency".

### Pitfall 4: Free-gate data exposure
**What goes wrong:** If gateMode is implemented only as a UI boolean in Zustand state, a user can open DevTools and toggle `gateMode` to false to see all findings.
**Why it happens:** Client-side-only enforcement is bypassable.
**How to avoid:** Gate enforcement is acceptable at the UI layer for Phase 1 (D-03 says no license key mechanism). However, the finding details (title, description, remediation) should still be present in the JSON payload — they are blurred in the UI. This is the intended Phase 1 design. Do NOT strip finding details from the JSON (that would break Phase 4 SaaS unlock). The blur is intentional UI gating, not security enforcement.
**Warning signs:** Temptation to strip data from the JSON payload.

### Pitfall 5: Module source paths outside the project
**What goes wrong:** Module `source` attributes can point to Terraform Registry paths (`hashicorp/vpc/aws`) or Git URLs — not just relative local paths.
**Why it happens:** PRS-04 specifies "local source resolution only" but the parser may encounter registry/Git sources.
**How to avoid:** Only follow `source` values that start with `./` or `../`. Skip all others silently (log at `--verbose`). Max depth 3.
**Warning signs:** Infinite recursion if circular local module references exist.

### Pitfall 6: vite-plugin-singlefile asset size limit
**What goes wrong:** Very large Terraform projects (500 resources) can produce JSON payloads that approach or exceed Vite's inline limit for single-file builds.
**Why it happens:** `assetsInlineLimit: 100000000` is already set in `vite.config.ts` (100MB), so this is unlikely. JSON for 500 resources is typically 1-5MB.
**How to avoid:** No action needed; current config handles it. Monitor HTML output size in integration tests.
**Warning signs:** Vite build warns about asset size during CI.

### Pitfall 7: PyPI package missing viewer template
**What goes wrong:** `pip install infracanvas` installs the wheel but `viewer_template.html` is a non-Python file that won't be included by default.
**Why it happens:** `hatchling` only includes Python files unless explicitly told to include other file types.
**How to avoid:** Add to `pyproject.toml`:
```toml
[tool.hatch.build.targets.wheel]
packages = ["infracanvas"]
include = ["infracanvas/export/viewer_template.html"]
```
**Warning signs:** `FileNotFoundError` on first `infracanvas scan` after `pip install`.

### Pitfall 8: GitHub Actions PyPI publish — Trusted Publisher setup required
**What goes wrong:** `pypa/gh-action-pypi-publish` requires Trusted Publisher to be configured on PyPI project settings first — cannot be done purely via code.
**Why it happens:** PyPI's Trusted Publisher is a server-side configuration.
**How to avoid:** Document in the GHA workflow README that PyPI Trusted Publisher must be configured at https://pypi.org/manage/project/infracanvas/settings/publishing/ before the workflow will succeed. Alternatively, use a PyPI API token as a secret fallback.
**Warning signs:** Workflow fails with 403 on first publish attempt.

## Open Questions

1. **Does `serve` require true live-reload (SSE / WebSocket) or is a page meta-refresh acceptable?**
   - What we know: D-12 says "browser auto-refreshes"; the existing `--watch` flag only re-scans to stdout
   - What's unclear: The mechanism — SSE injection into the HTML vs. a polling meta-refresh interval
   - Recommendation: Implement via SSE endpoint (`/events`) injected as a `<script>` into the served HTML; simpler than WebSocket and requires only stdlib

2. **Should the Homebrew formula live in a separate tap repo (`infracanvas/homebrew-infracanvas`) or in a `Formula/` subdirectory of the main repo?**
   - What we know: A `Formula/` directory already exists in the project root
   - What's unclear: Whether the `Formula/` dir approach works with `brew tap` or requires a dedicated repo
   - Recommendation: Use a separate GitHub repo `infracanvas/homebrew-infracanvas` for the tap — this is the standard brew pattern and supports `brew install infracanvas/infracanvas/infracanvas`

3. **Shadow infra flagging (PRS-05): should state parsing diff against the HCL graph in the Phase 1 pipeline?**
   - What we know: `parser/state.py` exists; shadow flagging is in requirements as PRS-05
   - What's unclear: Whether "shadow infra" in Phase 1 just means "resources in state not in HCL" (simple diff) or requires live API comparison (Phase 2)
   - Recommendation: Phase 1 shadow = state-vs-HCL diff only; flag as `DriftStatus.shadow` (new enum value) on nodes present in state but absent in HCL graph
